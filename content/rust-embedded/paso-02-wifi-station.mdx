## La memoria Flash del ESP32 вҖ” Fundamento fГӯsico

Antes de hablar de NVS, credenciales, o particiones, necesitamos entender **dГіnde** se guarda todo.

### ВҝQuГ© es la Flash?

<Callout>
 **Flash** es el chip de memoria **no volГЎtil** del ESP32-C3. Tiene **4MB** (4,194,304 bytes). Es el equivalente al "disco duro" de una computadora вҖ” los datos **sobreviven** a reinicios y cortes de energГӯa.

 Todo vive en esta flash: el bootloader, tu firmware compilado, la calibraciГіn WiFi, y los datos que tu programa guarde (como credenciales WiFi).
 
</Callout>

### Particiones вҖ” Dividir la flash en zonas

## NVS вҖ” Software sobre la Flash



**NVS (Non-Volatile Storage)** no es un hardware diferente a la flash вҖ” es una **librerГӯa de software** de ESP-IDF que usa una **particiГіn** de la flash para implementar un sistema **clave-valor**.

 Pensalo asГӯ:
 
 - **Flash** = el medio fГӯsico (como un SSD)
 - **ParticiГіn NVS** = un pedazo reservado de esa flash (como una particiГіn de disco)
 - **NVS (la librerГӯa)** = el software que lee/escribe clave-valor en esa particiГіn (como un mini filesystem)
 
 NVS es "Non-Volatile" precisamente **porque** estГЎ en flash вҖ” flash retiene datos sin electricidad. El nombre describe la propiedad del hardware, no un hardware diferente.



### CГіmo funciona NVS internamente

<Callout>
 **Estructura interna de NVS:**

 - La particiГіn NVS se divide en **pГЎginas** de 4KB (tamaГұo de un sector de flash)
 - Cada pГЎgina contiene **entries** de 32 bytes
 - Cada entry guarda: namespace index + tipo + key + value
 - Strings y blobs largos usan mГәltiples entries consecutivas
 
 **Wear leveling:** flash tiene ciclos de escritura limitados (~100,000). NVS distribuye escrituras entre pГЎginas para no gastar siempre la misma. Cuando una pГЎgina se llena, copia las entries vГЎlidas a otra y borra la vieja.
 
 **Namespaces:** agrupan keys relacionadas. Nuestro namespace `"credentials"` tiene keys como `wifi_ssid`, `wifi_pass`, etc. Otro mГіdulo podrГӯa usar namespace `"settings"` sin conflicto.
 
 **ProtecciГіn contra corrupciГіn:** si se corta la energГӯa durante una escritura, NVS puede recuperar el estado anterior gracias a su diseГұo transaccional.
 
</Callout>

Con esto claro вҖ” flash es el hardware, NVS es el software encima вҖ” podemos ver cГіmo lo usamos en cГіdigo.

### Antes de flashear вҖ” Limpieza recomendada

Si tu ESP32 ya fue usado con otro firmware (u otro partition table), es recomendable hacer una **limpieza completa** antes de flashear este proyecto. Esto evita problemas con datos viejos en NVS, particiones incompatibles, o configuraciones cacheadas.

**Paso 1 вҖ” Borrar toda la flash del ESP32:**

```bash
espflash erase-flash
```

<Callout type="warning">
 Este comando sobreescribe los **4MB completos** de la flash con ceros. Elimina:

 - Bootloader viejo
 - Partition table anterior (que puede tener offsets diferentes)
 - Datos de NVS (credenciales, configuraciГіn anterior)
 - Firmware anterior
 - CalibraciГіn PHY del radio WiFi (se regenera automГЎticamente)
 
 DespuГ©s de esto, el ESP32 queda como si fuera nuevo de fГЎbrica. No puede bootear hasta que le flashees un firmware.
 
</Callout>

**Paso 2 вҖ” Limpiar el cache del build:**

```bash
cargo clean
```

<Callout type="warning">
 Esto elimina el directorio `target/` donde se guardan los artefactos compilados, incluyendo el `sdkconfig` generado por ESP-IDF.

 **ВҝPor quГ© importa?** ESP-IDF genera un archivo `sdkconfig` durante el primer build, basado en `sdkconfig.defaults`. Si cambiaste algo en `sdkconfig.defaults` (como la partition table), el cambio **no se aplica** hasta que borres el `sdkconfig` cacheado. `cargo clean` fuerza que se regenere todo desde cero.
 
 **Regla:** cada vez que modifiques `sdkconfig.defaults`, haz `cargo clean` antes de compilar.
 
</Callout>

**Paso 3 вҖ” Compilar y flashear:**

```bash
cargo espflash flash --release --monitor
```

<Callout>

 - `--release` вҖ” compila con optimizaciones (el firmware es mГЎs chico y rГЎpido)
 - `--monitor` вҖ” despuГ©s de flashear, abre el monitor serial para ver los logs del ESP32
 
 El primer build despuГ©s de `cargo clean` es largo porque recompila todo el ESP-IDF (el SDK en C). Los builds siguientes son incrementales y mucho mГЎs rГЎpidos.
 
</Callout>



**ВҝSiempre hay que hacer erase-flash?** No. Solo es necesario cuando:

 - Cambiaste la partition table (`sdkconfig.defaults`)
 - El ESP32 tenГӯa un firmware con particiones diferentes
 - Ves comportamiento raro (boot loops, NVS corrupta, crashes al iniciar)
 - Quieres empezar con flash 100% limpia
 
 Para builds normales del dГӯa a dГӯa, solo necesitas `cargo espflash flash --release --monitor`.



## Archivo: `secure_storage.rs` вҖ” Credenciales seguras en flash

### ВҝQuГ© problema resuelve este archivo?

<Callout>
 Necesitamos guardar credenciales (WiFi, API key, device ID) que **sobrevivan reinicios**. Pero tambiГ©n necesitamos que esas credenciales **no queden expuestas en memoria** despuГ©s de usarlas.

 Este archivo construye una **capa de abstracciГіn** sobre NVS que resuelve ambos problemas. El plan es:
 
 1. **Definir constantes** para namespace y keys de NVS (evitar typos, centralizar nombres)
 2. **Crear un struct** `Credentials` que agrupe los 4 datos sensibles, con borrado automГЎtico de memoria via `Zeroize`
 3. **Crear un struct** `SecureStorage` que encapsule el acceso a NVS y exponga mГ©todos claros: `is_provisioned()`, `store_credentials()`, `load_credentials()`, `clear_credentials()`
 
 **PatrГіn Rust que usamos:** encapsulaciГіn via structs + traits derivados. En vez de llamar a NVS directamente desde `main.rs` con strings sueltos, creamos una API tipada donde el compilador nos ayuda a no cometer errores. Si manejamos mal las credenciales, Rust nos avisa en compilaciГіn (no en producciГіn).
 
</Callout>

### NVS en cГіdigo вҖ” Namespace y Keys

Ya sabemos que NVS es una librerГӯa que guarda clave-valor en una particiГіn de la flash (ver secciГіn anterior). Ahora veamos cГіmo lo usamos.

Lo primero que definimos en `secure_storage.rs` son las **constantes** que identifican dГіnde y quГ© guardamos:

```rust

const KEY_WIFI_SSID: &str = "wifi_ssid";
const KEY_WIFI_PASS: &str = "wifi_pass";
const KEY_API_KEY: &str = "api_key";
const KEY_DEVICE_ID: &str = "device_id";
const KEY_PROVISIONED: &str = "provisioned";
```

<Callout>
 `NVS_NAMESPACE` es como una "carpeta" dentro de NVS que agrupa todas nuestras keys. Otro mГіdulo podrГӯa usar un namespace diferente (ej: `"settings"`) sin conflicto.

 Las `KEY_*` son los nombres con los que guardamos y leemos cada valor. Son `&str` constantes para evitar typos вҖ” si escribimos mal una key, el dato simplemente no se encuentra.
 
</Callout>

### Struct `SecureStorage`


<Callout>
 `EspNvs<NvsDefault>` es el handle a un namespace de NVS. `NvsDefault` indica que usamos la particiГіn NVS default del ESP32 (hay otras para OTA, PHY, etc.).

 El struct encapsula el handle вҖ” todas las operaciones de lectura/escritura pasan por aquГӯ. Esto centraliza el acceso y facilita agregar encryption en producciГіn.
 
 ### `impl SecureStorage` вҖ” ImplementaciГіn de mГ©todos
 
 En Rust, un `struct` solo define **datos** (campos). La **lГіgica** (mГ©todos y funciones asociadas) se define en un bloque `impl` separado:
 
 ```rust
  // 1. El struct define QUГү datos tiene
  pub struct SecureStorage {
      nvs: EspNvs<NvsDefault>,
  }
  
  // 2. El impl define QUГү puede HACER con esos datos
  impl SecureStorage {
      pub fn new(...) -> Result<Self> { ... }
      pub fn is_provisioned(&self) -> Result<bool> { ... }
      pub fn store_credentials(&mut self, ...) -> Result<()> { ... }
      pub fn load_credentials(&self) -> Result<Credentials> { ... }
      pub fn clear_credentials(&mut self) -> Result<()> { ... }
  }
  ```
 
 <Callout>
 **ВҝPor quГ© separar struct de impl?** Porque en Rust puedes tener **mГәltiples bloques** `impl` para el mismo struct (incluso en archivos diferentes). Esto permite implementar traits externos, extender funcionalidad, y organizar cГіdigo. La separaciГіn datos/comportamiento es fundamental en Rust.
 </Callout>
 
 

**FunciГіn asociada vs MГ©todo вҖ” la diferencia es el primer parГЎmetro:**
 
 - **MГ©todo** вҶ’ recibe `self` (en alguna forma: `&self`, `&mut self`, `self`). Se llama con **punto**: `storage.is``_provisioned()`
 - **FunciГіn asociada** вҶ’ NO recibe `self`. Se llama con **doble dos puntos**: `SecureStorage::new(nvs)`
 
 En nuestro caso, `new()` es la Гәnica **funciГіn asociada** (equivalente a un "constructor" o "static method" en otros lenguajes). Las otras 4 (`is_provisioned`, `store_credentials`, `load_credentials`, `clear_credentials`) son **mГ©todos** porque operan sobre una instancia existente del struct.**


 
 Antes de ver cada mГ©todo en detalle, veamos el **mapa completo** вҖ” quГ© hace cada uno, por quГ© sus parГЎmetros son asГӯ, y quГ© retorna:
 
 | **MГ©todo** | **Firma** | **QuГ© hace y por quГ©** |
 | --- | --- | --- |
 | `new()` | `pub fn new(nvs_partition: EspNvsPartition<NvsDefault>) -> Result<Self>` | **Constructor.** Recibe el handle de la particiГіn NVS y abre el namespace `"credentials"`. Retorna `Result<Self>` porque abrir NVS puede fallar (particiГіn corrupta, etc.). `Self` = `SecureStorage`. |
 | `is_provisioned()` | `pub fn is_provisioned(&self) -> Result<bool>` | **Consulta.** `&self` = solo lectura (no modifica nada). Lee un flag `u8` de NVS. Retorna `Result<bool>` вҖ” puede fallar la lectura, y el valor es true/false. |
 | `store_credentials()` | `pub fn store_credentials(&mut self, mut creds: Credentials) -> Result<()>` | **Escritura.** `&mut self` = necesita mutar NVS (escribir datos). `mut creds` = toma ownership Y permiso de mutar (para llamar `.zeroize()` al final). Retorna `Result<()>` вҖ” puede fallar, pero no retorna datos. |
 | `load_credentials()` | `pub fn load_credentials(&self) -> Result<Credentials>` | **Lectura.** `&self` = solo lectura de NVS. Retorna `Result<Credentials>` вҖ” el struct completo con los 4 campos, protegido por `ZeroizeOnDrop`. |
 | `clear_credentials()` | `pub fn clear_credentials(&mut self) -> Result<()>` | **Factory reset.** `&mut self` = necesita mutar NVS (sobreescribir con vacГӯos). Borra todas las keys y marca `provisioned = 0`. Retorna `Result<()>`. |

## MГ©todo `is_provisioned()` вҖ” ВҝYa tiene credenciales?

```rust
pub fn is_provisioned(&self) -> Result<bool> {
    match self.nvs.get_u8(KEY_PROVISIONED) {
        Ok(Some(val)) => Ok(val == 1),
        Ok(None) => Ok(false),
        Err(e) => {
            warn!("Error checking provisioned status: {:?}", e);
            Ok(false)
        }
    }
}
```

<Callout type="warning">
 `get_u8()` retorna `Result<Option<u8>>`:

 - `Ok(Some(1))` вҶ’ la key existe y vale 1 вҶ’ provisionado
 - `Ok(None)` вҶ’ la key no existe вҶ’ no provisionado (primer boot)
 - `Err(e)` вҶ’ error de NVS вҶ’ asumimos no provisionado (safe default)
 
 Este patrГіn de "triple estado" (`Result<Option<T>>`) es comГәn en Rust cuando una operaciГіn puede fallar Y el valor puede no existir.
 
</Callout>


## MГ©todo `store_credentials()` вҖ” Guardar en flash

```rust
pub fn store_credentials(&mut self, mut creds: Credentials) -> Result<()> {
    self.nvs.set_str(KEY_WIFI_SSID, &creds.wifi_ssid)?;
    self.nvs.set_str(KEY_WIFI_PASS, &creds.wifi_password)?;
    self.nvs.set_str(KEY_API_KEY, &creds.api_key)?;
    self.nvs.set_str(KEY_DEVICE_ID, &creds.device_id)?;
    self.nvs.set_u8(KEY_PROVISIONED, 1)?;

    creds.zeroize(); // Borrar credenciales de memoria explГӯcitamente
    Ok(())
}
```

<Callout type="warning">
 **ВҝPor quГ©** `mut creds`**?** Porque `.zeroize()` necesita mutar los campos para sobreescribirlos con ceros. El `mut` en el parГЎmetro permite esto.

 **ВҝPor quГ© zeroizar explГӯcitamente si ya tiene** `ZeroizeOnDrop`**?** Porque `ZeroizeOnDrop` se ejecuta cuando el struct se dropea al final de la funciГіn. Con `.zeroize()` explГӯcito, limpiamos la memoria **inmediatamente** despuГ©s de guardar вҖ” minimizamos la ventana de exposiciГіn.
 
</Callout>


## MГ©todo `load_credentials()` вҖ” Leer de flash

```rust
pub fn load_credentials(&self) -> Result<Credentials> {
    if !self.is_provisioned()? {
        bail!("Device not provisioned. Run provisioning first.");
    }

    let mut creds = Credentials::default();
    let mut buf = [0u8; 256];

    if let Some(val) = self.nvs.get_str(KEY_WIFI_SSID, &mut buf)? {
        creds.wifi_ssid = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    if let Some(val) = self.nvs.get_str(KEY_WIFI_PASS, &mut buf)? {
        creds.wifi_password = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    if let Some(val) = self.nvs.get_str(KEY_API_KEY, &mut buf)? {
        creds.api_key = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    if let Some(val) = self.nvs.get_str(KEY_DEVICE_ID, &mut buf)? {
        creds.device_id = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    info!("Credentials loaded from NVS");
    Ok(creds)
}
```

<Callout type="warning">
 **Guard clause con** `bail!` вҖ” Lo primero que hace es verificar `is_provisioned()`. Si no hay credenciales, retorna error inmediatamente con `bail!` (macro de `anyhow` que hace `return Err(...)`).

 Esto evita leer keys que no existen y da un mensaje de error claro al caller.
 
</Callout>



**`Credentials::default()` вҖ” crea un struct con todos los campos vacГӯos (`""`) gracias al trait `Default` que derivamos. Luego llenamos campo por campo desde NVS.**



<Callout>
 **El buffer** `buf`** se zeroiza despuГ©s de cada lectura.** NVS necesita un buffer donde copiar el string leГӯdo. Si no lo limpiamos, quedarГӯa una copia del password en el stack.

 `.trim_end_matches('\0')` вҖ” NVS agrega null terminators al final del string (herencia de C). Los eliminamos antes de convertir a `String` de Rust.
 
 `if let Some(val)` вҖ” `get_str()` retorna `Option<&str>`. Si la key no existe, retorna `None` y el campo queda vacГӯo. Es un patrГіn defensivo: aunque `is_provisioned()` diga `true`, alguna key individual podrГӯa faltar.
 
 **ВҝPor quГ© un buffer de 256 bytes?** Es el tamaГұo mГЎximo que soportamos para API keys. SSID max = 32, password max = 64. El buffer se reutiliza para cada campo.
 
</Callout>


## MГ©todo `clear_credentials()` вҖ” Factory reset

```rust
#[allow(dead_code)]
pub fn clear_credentials(&mut self) -> Result<()> {
    warn!("Clearing all credentials from NVS...");

    self.nvs.set_str(KEY_WIFI_SSID, "")?;
    self.nvs.set_str(KEY_WIFI_PASS, "")?;
    self.nvs.set_str(KEY_API_KEY, "")?;
    self.nvs.set_str(KEY_DEVICE_ID, "")?;
    self.nvs.set_u8(KEY_PROVISIONED, 0)?;

    info!("Credentials cleared");
    Ok(())
}
```

<Callout type="warning">
 **ВҝPor quГ© sobreescribir con strings vacГӯos en vez de borrar las keys?** Porque NVS no garantiza que `remove()` sobreescriba los bytes fГӯsicos en flash вҖ” podrГӯa solo marcar el entry como invГЎlido. Escribir strings vacГӯos fuerza una escritura real que pisa los datos anteriores.
</Callout>



`#[allow(dead_code)]` вҖ” Este mГ©todo no se usa todavГӯa en el cГіdigo actual (no hay UI para hacer factory reset). Sin este atributo, el compilador darГӯa un warning: "function is never used". Lo dejamos preparado para pasos futuros donde agreguemos un botГіn fГӯsico o comando remoto de reset.

 `&mut self` вҖ” Necesita referencia mutable porque escribe en NVS (sobreescribe cada key con vacГӯo y pone `provisioned = 0`).
 
 `warn!`** en vez de** `info!` вҖ” Usamos `warn!` porque borrar credenciales es una acciГіn destructiva. En los logs del monitor serial, los warnings aparecen en amarillo para llamar la atenciГіn.

## Firma de `start_provisioning()`

```rust
pub fn start_provisioning(
    modem: impl peripheral::Peripheral<P = esp_idf_svc::hal::modem::Modem> + 'static,
    sysloop: EspSystemEventLoop,
    storage: Arc<Mutex<SecureStorage>>,
) -> Result<()> {
```

Recibe tres cosas:

- `modem` вҖ” el perifГ©rico WiFi (ownership, se consume aquГӯ)
- `sysloop` вҖ” el bus de eventos del sistema (clonable, barato)
- `storage` вҖ” acceso compartido al NVS, envuelto en `Arc<Mutex<>>` porque el handler HTTP corre en otro thread

<Callout>
 **ВҝPor quГ©** `Arc<Mutex<SecureStorage>>`**?**

 El handler HTTP del POST corre en un thread separado (creado por el servidor HTTP). Necesita acceso al storage para guardar credenciales. `Arc` permite tener mГәltiples dueГұos, y `Mutex` garantiza acceso exclusivo.
 
 - `Arc` вҶ’ multiple ownership (reference counting)
 - `Mutex` вҶ’ exclusive access (lock/unlock)
 - Sin `Arc`: no podemos compartir entre threads
 - Sin `Mutex`: data race (undefined behavior)
 
</Callout>

Vamos a seguir la funciГіn paso a paso, en el mismo orden que el cГіdigo.


## Paso 1 вҖ” Crear driver WiFi e interfaces de red

```rust

let sta_netif = EspNetif::new(NetifStack::Sta)?;
```

`WifiDriver` es el driver de bajo nivel del WiFi. Toma ownership del modem y se conecta al event loop.

`EspNetif::new(NetifStack::Sta)` crea una interfaz de red tipo Station. Aunque no la usamos directamente, `EspWifi::wrap_all()` la requiere internamente.

<Callout>
 **ВҝPor quГ© crear una interfaz STA si estamos en modo AP?**

 El ESP-IDF internamente necesita ambas interfaces (STA + AP) para funcionar correctamente. `wrap_all()` combina el driver con ambas interfaces. Si solo pasamos la AP, falla.
 
</Callout>

## Paso 4 вҖ” Preparar variables compartidas

```rust
let provisioned = Arc::new(Mutex::new(false));
let provisioned_clone = provisioned.clone();
let storage_clone = storage.clone();
```

Antes de crear el servidor HTTP, preparamos las variables que el handler POST necesita:

- `provisioned` вҖ” flag `bool` envuelto en `Arc<Mutex<>>`. El handler POST lo pone en `true` cuando el usuario completa el formulario. El loop principal lo lee para saber cuГЎndo reiniciar.
- **Clones** вҖ” `provisioned_clone` y `storage_clone` son los que se mueven al closure del POST con `move`. Los originales quedan para el loop principal.

<Callout type="warning">
 **ВҝPor quГ© clonar antes del closure?**

 Un closure `move` toma ownership de todo lo que captura. Si pasГЎramos `provisioned` directamente al closure, el loop principal ya no podrГӯa leerlo. Al clonar el `Arc`, ambos (closure y loop) tienen su propia referencia al mismo dato.
 
</Callout>

```rust
    Method::Get,
    |req| -> core::result::Result<(), esp_idf_svc::io::EspIOError> {
        let mut response = req.into_ok_response()?;
        response.write_all(HTML_FORM.as_bytes())?;
        Ok(())
    },
)?;
```

El servidor HTTP escucha en el puerto 80 (default). El handler GET en `/` simplemente sirve el formulario HTML que definimos en `HTML_FORM`.

- `req.into_ok_response()` вҖ” consume el request y crea una response con status 200
- `write_all(HTML_FORM.as``_bytes())` вҖ” escribe el HTML como bytes en el body de la response

Este closure NO necesita `move` porque no captura ninguna variable del entorno.


## Paso 6 вҖ” Handler POST: recibir, validar y guardar

```rust
server.fn_handler(
    "/provision",
    Method::Post,
    move |mut req| -> core::result::Result<(), esp_idf_svc::io::EspIOError> {
        // Leer body del POST (formulario URL-encoded)
        let mut body = [0u8; 512];
        let len = req.read(&mut body)?;
        let body_str = std::str::from_utf8(&body[..len]).unwrap_or("");

        // Parsear campos del formulario
        let mut ssid = String::new();
        let mut password = String::new();
        let mut device_id = String::new();
        let mut api_key = String::new();

        for pair in body_str.split('&') {
            if let Some((key, value)) = pair.split_once('=') {
                let decoded = urlencoding_decode(value);
                match key {
                    "ssid" => ssid = decoded,
                    "password" => password = decoded,
                    "device_id" => device_id = decoded,
                    "api_key" => api_key = decoded,
                    _ => {}
                }
            }
        }

        // Validar campos requeridos
        if ssid.is_empty() || password.is_empty() || device_id.is_empty() {
            error!("Missing required fields in provisioning form");
            let mut response = req.into_status_response(400)?;
            response.write_all(b"Missing required fields")?;
            return Ok(());
        }

        // Crear credenciales y guardar en NVS
        let creds = Credentials {
            wifi_ssid: ssid,
            wifi_password: password,
            api_key,
            device_id,
        };

        if let Ok(mut storage) = storage_clone.lock() {
            if let Err(e) = storage.store_credentials(creds) {
                error!("Failed to store credentials: {:?}", e);
                let mut response = req.into_status_response(500)?;
                response.write_all(b"Failed to store credentials")?;
                return Ok(());
            }
        }

        // Marcar provisioning como completado
        if let Ok(mut p) = provisioned_clone.lock() {
            *p = true;
        }

        // Enviar pГЎgina de Г©xito
        let mut response = req.into_ok_response()?;
        response.write_all(HTML_SUCCESS.as_bytes())?;
        Ok(())
    },
)?;
```

Este es el handler mГЎs importante. El keyword `move` transfiere ownership de `provisioned_clone` y `storage_clone` al closure.

El flujo completo:

1. **Leer body** вҖ” `req.read``(&mut body)` lee los bytes del formulario. Buffer de 512 bytes es suficiente para las credenciales
2. **Parsear** вҖ” el body viene como `ssid=Mi+Red&password=abc%21`. Se split por `&`, luego por `=`, y se decodifica URL encoding
3. **Validar** вҖ” si falta SSID, password o device_id, retorna error 400
4. **Guardar** вҖ” crea un `Credentials` y lo guarda en NVS via `storage_clone`
5. **SeГұalizar** вҖ” pone `provisioned_clone` en `true` para que el loop principal detecte que debe reiniciar
6. **Responder** вҖ” envГӯa `HTML_SUCCESS` al browser

<Callout type="warning">
 **ВҝQuГ© hace** `move`** en el closure?**

 **Sin** `move`**:** el closure intenta tomar **referencias** a `provisioned_clone` y `storage_clone`. Pero estas variables viven en `start_provisioning()`, y el closure vive mГЎs allГЎ (en el thread del servidor HTTP). Las referencias serГӯan invГЎlidas вҶ’ **no compila**.
 
 **Con** `move`**:** el closure toma **ownership** de los clones. Los datos se mueven al closure y viven tanto como Г©l. El compilador estГЎ satisfecho.
 
</Callout>

<Callout type="warning">
 **ВҝPor quГ©** `if let Ok(mut storage)`** y no** `.unwrap()`**?**

 En embedded, si el thread que tiene el lock hace panic, el Mutex queda "envenenado". `.lock()` retorna `Err` en ese caso. Con `if let Ok(...)` el handler simplemente no guarda вҖ” es mГЎs robusto que crashear todo el dispositivo con `.unwrap()`.
 
</Callout>

## FunciГіn urlencoding_decode() вҖ” Decodificar datos del formulario

<Callout>
 **ВҝQuГ© es URL encoding y por quГ© necesitamos decodificarlo?

**Cuando el usuario llena el formulario en `192.168.4.1` y presiona "Save & Connect", el browser envГӯa los datos como `application/x-www-form-urlencoded`. Este formato tiene dos reglas de escape:

вҖў Espacios вҶ’ `+` (el browser reemplaza cada espacio por un signo +)
вҖў Caracteres especiales вҶ’ `%XX` (donde XX es el cГіdigo hexadecimal del byte)

Por ejemplo, si el SSID es "Mi Red WiFi" y el password es "p@ssw0rd!", el browser envГӯa:
`ssid=Mi+Red+WiFi&password=p%40ssw0rd%21`

Nuestra funciГіn revierte esto: `+` вҶ’ espacio, `%40` вҶ’ `@`, `%21` вҶ’ `!`.
</Callout>

### El cГіdigo

```rust
fn urlencoding_decode(input: &str) -> String {
    let mut result = String::with_capacity(input.len());
    let mut chars = input.chars().peekable();

    while let Some(c) = chars.next() {
        match c {
            '+' => result.push(' '),
            '%' => {
                let hex: String = chars.by_ref().take(2).collect();
                if let Ok(byte) = u8::from_str_radix(&hex, 16) {
                    result.push(byte as char);
                }
            }
            _ => result.push(c),
        }
    }

    result
}
```

## AnГЎlisis del cГіdigo

<Callout>
 **Pre-alocaciГіn y memoria en embedded

**`String::with_capacity(input.len())` вҖ” Pre-aloca memoria en el heap del tamaГұo del input. El string decodificado siempre es вүӨ al encoded (los `%XX` de 3 chars se convierten en 1 char), asГӯ que esta capacidad es suficiente.

**ВҝPor quГ© importa en ESP32?** El ESP32-C3 tiene solo ~400KB de RAM. Sin `with_capacity`, Rust empezarГӯa con un buffer pequeГұo y lo re-alocarГӯa cada vez que se llena (duplicando su tamaГұo). Cada re-alocaciГіn implica: pedir memoria nueva вҶ’ copiar todo вҶ’ liberar la vieja. En un microcontrolador con heap limitado, esto puede causar fragmentaciГіn de memoria. Pre-alocar elimina ese riesgo.
</Callout>



**PatrГіn iterador: while let + by_ref().take(2).collect()

**`input.chars().peekable()` вҖ” Crea un iterador sobre caracteres Unicode. `peekable()` permite mirar el siguiente carГЎcter sin consumirlo (no se usa directamente aquГӯ pero es buena prГЎctica para parsers).

`while let Some(c) = chars.next()` вҖ” PatrГіn idiomГЎtico de Rust: "mientras haya un siguiente carГЎcter, extrГЎelo en `c`". Combina iteraciГіn + destructuring.

`match c` вҖ” Pattern matching con tres brazos:
вҖў `'+'` вҶ’ push espacio (convenciГіn HTML form encoding)
вҖў `'%'` вҶ’ tomar los 2 siguientes chars como hex:
 `chars.by_ref().take(2).collect()` вҖ” Tres operaciones:
 1. `by_ref()` toma prestado el iterador (sin consumir ownership)
 2. `take(2)` limita a los prГіximos 2 caracteres
 3. `collect()` los junta en un `String`
вҖў `_` вҶ’ cualquier otro carГЎcter se copia tal cual.



<Callout>
 **ConversiГіn segura y manejo de errores sin panic

**`u8::from_str_radix(&hex, 16)` вҖ” Convierte el string hexadecimal a un byte (`u8`). Retorna `Result<u8>`, NO un u8 directo.

`if let Ok(byte)` вҖ” Si el hex es invГЎlido (ej. `%ZZ`), simplemente lo ignora en vez de hacer panic. **En embedded NUNCA usamos.unwrap() con datos externos** вҖ” un panic reinicia el chip.

`byte as char` вҖ” Cast de `u8` a `char`. Funciona porque los valores URL-encoded en formularios WiFi estГЎn en rango ASCII (0-127).

**ВҝPor quГ© no usar un crate?** Existen `urlencoding` y `percent-encoding`, pero agregar un crate para 15 lГӯneas no vale la pena: mГЎs binario en flash limitada, mГЎs superficie de bugs, mГЎs tiempo de compilaciГіn. Regla prГЎctica: si puedes escribirlo en menos de 20 lГӯneas y cubre tu caso, no agregues un crate.
</Callout>

## Firma de la funciГіn `connect()`

```rust
pub fn connect(
    ssid: &str,
    password: &str,
    modem: impl peripheral::Peripheral<P = esp_idf_svc::hal::modem::Modem> + 'static,
    sysloop: EspSystemEventLoop,
) -> Result<Box<EspWifi<'static>>>
```

## `impl Trait` en parГЎmetros

<Callout>
 `impl peripheral::Peripheral<P = ...Modem> + 'static` вҖ” En vez de recibir un tipo concreto, usamos `impl Trait`. Esto le dice a Rust: "acepto cualquier tipo que implemente este trait". Es azГәcar sintГЎctico para genГ©ricos вҖ” el compilador genera una versiГіn concreta en compilaciГіn (monomorphization).

**ВҝPor quГ© no un tipo concreto?** Porque ESP-IDF puede cambiar el tipo interno del modem entre versiones. Con `impl Trait` nuestro cГіdigo es flexible: funciona con cualquier tipo que cumpla el contrato `Peripheral`.

`+ 'static` вҖ” Significa que el valor no contiene referencias temporales. Es necesario porque el driver WiFi vive en un `Box` que puede vivir indefinidamente вҖ” no puede depender de datos que se destruyan antes.
</Callout>

### `Box<EspWifi<'static>>` вҖ” ВҝPor quГ© Box?



`Result<Box<EspWifi<'static>>>` вҖ” La funciГіn retorna el driver WiFi envuelto en un `Box` (heap allocation). ВҝPor quГ© no retornarlo directamente en el stack?

**1.** `EspWifi` es un struct grande (~200+ bytes) con buffers internos. Moverlo por el stack implica copiar todos esos bytes cada vez que se retorna o asigna.
**2.** Con `Box`, solo se mueve un puntero (4 bytes en ESP32). El struct real vive en el heap y no se copia.
**3.** `'static` indica que el driver puede vivir todo el programa вҖ” no estГЎ atado al lifetime de ninguna funciГіn.

**RAII en acciГіn:** mientras el `Box` exista, la conexiГіn WiFi se mantiene. Si se dropea (sale de scope o se hace `drop()`), el destructor de `EspWifi` apaga el radio WiFi automГЎticamente.



### `EspSystemEventLoop` вҖ” El bus de eventos

<Callout>
 `EspSystemEventLoop` es el bus de eventos del sistema ESP-IDF. WiFi, IP, Bluetooth вҖ” todo pasa por este bus. Es un singleton que se obtiene con `EspSystemEventLoop::take()` en `main.rs` y se pasa como parГЎmetro.

**ВҝPor quГ© se puede pasar a varias funciones?** Porque internamente usa `Arc` (puntero atГіmico con cuenta de referencias). Cada `.clone()` solo incrementa un contador вҖ” no duplica el bus. Es como tener mГәltiples "suscriptores" al mismo bus de eventos.
</Callout>

## BlockingWifi: Wrapper sГӯncrono

```rust

```

<Callout type="warning">
 `EspWifi` es **async** por naturaleza вҖ” dispara eventos y no bloquea. `BlockingWifi` es un wrapper que:

 1. Se suscribe a los eventos del event loop
 2. **Bloquea** el thread actual hasta que el evento llega
 3. Te da una API sГӯncrona: `wifi.connect()` bloquea hasta conectar
 
 **ВҝPor quГ© bloquear?** En nuestro firmware, WiFi es secuencial: no podemos hacer nada hasta tener conexiГіn. Async serГӯa over-engineering para este caso.
 
 **Borrow vs Move:** `&mut esp_wifi` es un **borrow mutable**. `BlockingWifi` NO toma ownership вҖ” solo lo "pide prestado". `esp_wifi` sigue siendo dueГұo del driver. Esto es importante porque al final retornamos `esp_wifi`, no el `BlockingWifi`.
 
</Callout>

## Scan de redes

```rust
let ap_infos = wifi.scan()?;
let target_ap = ap_infos.into_iter().find(|ap| ap.ssid == ssid);
let channel = target_ap.as_ref().map(|ap| ap.channel);
```

<Callout>
 **ВҝPor quГ© escanear?** Para obtener el **canal exacto** del AP. Si le damos el canal correcto al driver, la conexiГіn es mГЎs rГЎpida (no tiene que probar los 13 canales).

 **Desglose del iterador:**
 
 - `.into_iter()` вҖ” **consume** el Vec (move, no borrow). Cada AP se mueve fuera del Vec.
 - `.find(|ap| ap.ssid == ssid)` вҖ” busca el primer elemento que cumple la condiciГіn. Retorna `Option<AccessPointInfo>`.
 - `.as_ref()` вҖ” convierte `&Option<T>` en `Option<&T>` (para no mover el valor)
 - `.map(|ap| ap.channel``)` вҖ” transforma el valor interno si es `Some`. Resultado: `Option<u8>` con el canal.
 
 **Closures:** `|ap| ap.ssid == ssid` es un closure (funciГіn anГіnima). `|ap|` es el parГЎmetro, `ap.ssid == ssid` es el cuerpo. El closure **captura** `ssid` por referencia del scope externo.
 
</Callout>

## `try_into()` вҖ” ConversiГіn falible

<Callout>
 `ssid.try_into().expect("SSID too long")` вҖ” `ClientConfiguration` espera un `heapless::String<32>` (string de mГЎximo 32 bytes, sin heap). Nuestro `ssid` es un `&str` normal, asГӯ que necesitamos convertirlo.

`try_into()` retorna `Result` porque puede fallar: si el SSID tiene mГЎs de 32 bytes, no cabe y retorna `Err`.

**ВҝPor quГ©.expect() aquГӯ y no if let?** Porque un SSID de mГЎs de 32 bytes es un error de programaciГіn nuestro (los SSIDs WiFi tienen mГЎximo 32 bytes por estГЎndar IEEE 802.11). No es input externo malicioso вҖ” es un bug. En ese caso, `expect()` con mensaje descriptivo es aceptable.
</Callout>

### `..Default::default()` вҖ” Struct update syntax

<Callout>
 `..Default::default()` вҖ” Struct update syntax. Significa: "para todos los campos que no mencionГ© explГӯcitamente, usa los valores default". `ClientConfiguration` tiene muchos campos (bssid, scan_method, pmf_cfg, etc.), pero solo nos importan 4: `ssid`, `password`, `channel` y `auth_method`.

Sin esta sintaxis, tendrГӯamos que listar cada campo manualmente вҖ” tedioso y frГЎgil ante actualizaciones de ESP-IDF. Con `..` el cГіdigo sobrevive si ESP-IDF agrega campos nuevos en futuras versiones.
</Callout>

```rust
info!("IP: {}", ip_info.ip);
info!("Gateway: {}", ip_info.subnet.gateway);
info!("Mask: {}", ip_info.subnet.mask);

Ok(Box::new(esp_wifi))
```

<Callout>
 **Cadena de acceso y el retorno final

**`wifi.wifi().sta_netif().get_ip_info()` вҖ” Cadena de mГ©todos para llegar al IP:
вҖў `wifi.wifi()` вҶ’ accede al `EspWifi` interno del `BlockingWifi`
вҖў `.sta_netif()` вҶ’ obtiene la interfaz de red Station
вҖў `.get_ip_info()` вҶ’ lee IP, gateway y mГЎscara asignados por DHCP

`Ok(Box::new(esp_wifi))` вҖ” AquГӯ sucede algo sutil: `wifi` (el `BlockingWifi`) se dropea al salir de la funciГіn porque no lo retornamos. Pero la conexiГіn WiFi **no se pierde** вҖ” `BlockingWifi` solo era un wrapper sГӯncrono sobre `esp_wifi` (tomГі `&mut`, no ownership). El driver real (`esp_wifi`) sobrevive dentro del `Box` y mantiene el radio encendido y conectado.
</Callout>

### DeclaraciГіn de mГіdulos

```rust
mod provisioning;
mod secure_storage;
mod wifi;
```

<Callout>
 Cada `mod X;` le dice a Rust: "incluГӯ el archivo `src/X.rs` como un mГіdulo llamado `X`". El mГіdulo tiene su propio **namespace** вҖ” para usar sus funciones pГәblicas, escribes `wifi::connect()` o `provisioning::start_provisioning()`.

 En Rust, **todo es privado por defecto**. Solo las funciones marcadas con `pub` son visibles desde fuera del mГіdulo.
 
</Callout>


## PatrГіn `main()` вҶ’ `run()`

```rust
fn main() {
    esp_idf_svc::sys::link_patches();
    esp_idf_svc::log::EspLogger::initialize_default();

    if let Err(e) = run() {
        error!("Error fatal: {:?}", e);
        std::thread::sleep(Duration::from_secs(10));
        unsafe { esp_idf_svc::sys::esp_restart(); }
    }
}

fn run() -> anyhow::Result<()> {
    // toda la lГіgica aquГӯ con ? libre
}
```

<Callout type="warning">
 **ВҝPor quГ© separar** `main()`** de** `run()`**?**

 `main()` sin tipo de retorno no puede usar el operador `?`. Al poner la lГіgica en `run() -> Result<()>`, podemos usar `?` libremente. Si algo falla, el error "burbujea" hasta `main()` que:
 
 1. Loguea el error completo (`{:?}` usa el trait `Debug`)
 2. Espera 10 segundos (para que leas el error en serial)
 3. Reinicia el chip via `esp_restart()`
 
 **El firmware padre usa exactamente este patrГіn.**
 
</Callout>


## InicializaciГіn del sistema вҖ” Peripherals, EventLoop y LED

```rust
let peripherals = Peripherals::take()?;
let sysloop = EspSystemEventLoop::take()?;

let nvs_partition = EspDefaultNvsPartition::take()?;

let mut led = PinDriver::output(peripherals.pins.gpio8)?;
```

<Callout>
 **Tres singletons del sistema ESP-IDF

**Cada `::take()` solo puede llamarse una vez en todo el programa. Si se llama dos veces, retorna `Err`. Este patrГіn garantiza ownership exclusivo:

вҖў `Peripherals::take()` вҖ” Toma ownership de TODOS los pines GPIO, modem WiFi, buses I2C/SPI, etc. El campo `peripherals.modem` se pasarГЎ a `wifi::connect()` o `start_provisioning()` (move).
вҖў `EspSystemEventLoop::take()` вҖ” El bus de eventos. Internamente usa `Arc`, asГӯ que `.clone()` es barato.
вҖў `EspDefaultNvsPartition::take()` вҖ” La particiГіn NVS default de la flash.

`PinDriver::output(peripherals.pins.gpio8)` вҖ” Herencia del Paso 1. Configura GPIO8 como salida digital para el LED. AquГӯ funciona como heartbeat: si el LED parpadea, el firmware estГЎ corriendo.
</Callout>


## InicializaciГіn de NVS

```rust
let nvs_partition = EspDefaultNvsPartition::take()?;
```

<Callout>
 Otro **singleton** вҖ” como `Peripherals::take()` y `EspSystemEventLoop::take()`. Solo puede llamarse una vez. Retorna un handle a la particiГіn NVS default del ESP32.

 Este handle se pasa a `SecureStorage::new()` para inicializar el acceso a NVS. `.clone()` es barato porque internamente usa `Arc`.
 
</Callout>


## SecureStorage con `Arc<Mutex<>>`

```rust
let storage = SecureStorage::new(nvs_partition.clone())?;
let storage = Arc::new(Mutex::new(storage));
```

<Callout>
 **Shadowing + concurrencia segura

**LГӯnea 1: `SecureStorage::new()` abre el namespace NVS y crea el handle. `nvs_partition.clone()` es necesario porque `SecureStorage::new()` toma ownership de la particiГіn, y la necesitamos tambiГ©n para el WiFi.

LГӯnea 2: `let storage = Arc::new(Mutex::new(storage))` вҖ” **Shadowing:** reusar el nombre `storage` es vГЎlido en Rust. La primera variable se consume (moved) y el nombre ahora apunta al `Arc<Mutex<SecureStorage>>`.

**ВҝPor quГ© `Arc<Mutex<>>`?** Porque `storage` se usa en dos contextos: aquГӯ en `run()` (thread principal) y dentro del handler HTTP del provisioning (otro thread del servidor). `Arc` permite shared ownership entre threads, `Mutex` garantiza que solo uno acceda a la vez.
</Callout>


## Check de provisioning

```rust
let is_provisioned = {
    let storage = storage.lock().unwrap();
    storage.is_provisioned()?
};
```

<Callout type="warning">
 **ВҝPor quГ© el bloque** `{ ... }`**?** Para que el `MutexGuard` (retornado por `.lock()`) se dropee al final del bloque. Si no lo hacemos, el lock quedarГӯa tomado y ningГәn otro cГіdigo podrГӯa acceder a `storage` вҖ” deadlock.

 Este patrГіn de "lock, operar, soltar" es fundamental cuando usas `Mutex`.
 
</Callout>


## Branch: Modo Provisioning

```rust
if !is_provisioned {
    warn!("Device not provisioned!");
    info!("Starting provisioning mode...");
    info!("Connect to WiFi: 'Leonobitech-Setup' / Password: 'setup1234'");
    info!("Then open http://192.168.4.1 in your browser");

    // start_provisioning() NUNCA retorna вҖ” reinicia el chip al completar
    provisioning::start_provisioning(peripherals.modem, sysloop, storage)?;

    return Ok(());
}
```

<Callout>
 **El gran branch: provisioning vs normal

**Todo `main.rs` se divide en dos caminos basado en `is_provisioned`:

**Si NO estГЎ provisionado:**
1. Loguea instrucciones para el usuario (SSID, password, URL)
2. Llama a `start_provisioning()` que crea el Access Point y servidor HTTP
3. `start_provisioning()` internamente hace `esp_restart()` al guardar credenciales вҖ” el chip reinicia y la prГіxima vez `is_provisioned` serГЎ `true`
4. `return Ok(())` es unreachable (nunca se ejecuta porque el chip ya reiniciГі), pero Rust lo necesita para que el flujo de control compile.

**Ownership:** Nota que `peripherals.modem` se pasa por move. Si este branch se ejecuta, el modem se consume aquГӯ y el cГіdigo despuГ©s del `if` nunca corre. Si NO se ejecuta (ya estГЎ provisionado), el modem sigue disponible para `wifi::connect()`. Rust verifica esto en compilaciГіn вҖ” los dos branches son mutuamente exclusivos.
</Callout>


## Branch: Modo Normal вҖ” Cargar credenciales

```rust
info!("Device is provisioned, loading credentials...");

let credentials = {
    let storage = storage.lock().unwrap();
    storage.load_credentials()?
};

info!("Device ID: {}", credentials.device_id);
info!("Connecting to WiFi: {}", credentials.wifi_ssid);
```

<Callout>
 **Mismo patrГіn de bloque para el Mutex

**De nuevo el patrГіn `{ let guard = mutex.lock(); ... }` вҖ” el bloque limita el scope del `MutexGuard` para que se libere inmediatamente despuГ©s de leer. Si no usГЎramos el bloque, el lock se mantendrГӯa hasta el final de `run()` вҖ” bloqueando cualquier otro acceso al storage.

DespuГ©s del bloque, `credentials` es un struct `Credentials` owned (no es referencia). Contiene copias de los strings WiFi SSID, password, API key y device ID leГӯdos de la flash. Loguear el device ID y SSID es seguro вҖ” el password nunca se imprime.
</Callout>

## Variable `_wifi` вҖ” Mantener vivo sin usar

```rust
let _wifi = wifi::connect(
    &credentials.wifi_ssid,
    &credentials.wifi_password,
    peripherals.modem,
    sysloop,
)?;
```

<Callout type="warning">
 **CONCEPTO CRГҚTICO.** El prefijo `_` indica que no usamos la variable, pero **NECESITAMOS mantenerla viva**. Si se dropea (sale de scope), la conexiГіn WiFi se pierde.

 **Si escribieras:**
 
 ```rust
  wifi::connect(...)?;  // sin asignar
  ```
 
 El `Box<EspWifi>` se dropearГӯa **inmediatamente** y perderГӯas WiFi. RAII (Resource Acquisition Is Initialization) en acciГіn: cuando el dueГұo muere, el recurso se libera.
 
 **PatrГіn comГәn en embedded:** "No necesito interactuar con este valor, pero necesito que exista para que el recurso subyacente siga activo."
 
</Callout>


## Drop explГӯcito de credenciales

```rust
drop(credentials);
info!("Credentials zeroized from memory");
```

<Callout>
 `drop()` destruye el valor inmediatamente (no espera a que salga de scope). Como `Credentials` implementa `ZeroizeOnDrop`, el drop sobreescribe todos los campos con ceros antes de liberar la memoria.

 **ВҝPor quГ© no dejar que se dropee solo?** Porque `credentials` vivirГӯa hasta el final de `run()` вҖ” que es un loop infinito. Las credenciales quedarГӯan en memoria indefinidamente. Con `drop()` explГӯcito, las limpiamos apenas las usamos.
 
</Callout>


## `unsafe` block вҖ” FFI a C

```rust
unsafe {
    esp_idf_svc::sys::esp_restart();
}
```

<Callout type="warning">
 `unsafe` no significa "peligroso". Significa: "el compilador no puede verificar la seguridad de este cГіdigo, yo (el programador) garantizo que es correcto".

 `esp_restart()` es una funciГіn C llamada via FFI (Foreign Function Interface). Rust no puede analizar cГіdigo C, entonces requiere `unsafe` para cualquier llamada FFI directa.
 
 En la prГЎctica, `esp_restart()` es perfectamente seguro вҖ” solo reinicia el chip limpiamente. El `unsafe` es una formalidad del sistema de tipos.
 
</Callout>


## Loop principal вҖ” LED heartbeat

```rust
info!("Entering main loop...");
loop {
    led.set_high()?;
    info!("LED ON");
    FreeRtos::delay_ms(500);

    led.set_low()?;
    info!("LED OFF");
    FreeRtos::delay_ms(500);
}
```

<Callout>
 **El loop infinito y por quГ© run() nunca retorna Ok

**Este `loop {}` no tiene `break` вҖ” es un loop infinito intencional. En embedded, el firmware nunca "termina": siempre hay un loop principal corriendo. Si `run()` sale, es porque un `?` propagГі un error (ej: fallo en GPIO), y `main()` lo captura, espera 10s y reinicia el chip.

`FreeRtos::delay_ms(500)` вҖ” No es un busy-wait como `std::thread::sleep()`. FreeRTOS es el sistema operativo en tiempo real del ESP32. `delay_ms()` cede el CPU al scheduler para que otros tasks (como el stack WiFi) puedan ejecutarse durante esos 500ms. Si usГЎramos busy-wait, bloquearГӯamos el WiFi.

**Herencia del Paso 1:** Por ahora el LED solo parpadea como heartbeat ("estoy vivo"). En pasos futuros, este loop manejarГЎ WebSocket messages, telemetrГӯa y scheduling.
</Callout>

**`Arc<Mutex<T>>`** вҖ” Compartir datos entre threads de forma segura. El handler HTTP y el loop principal acceden al mismo `SecureStorage` sin data races. En C esto serГӯa un global con un semГЎforo que alguien olvida tomar. En Rust, el compilador no te deja compilar sin el Mutex.





**La filosofГӯa embedded que estГЎs absorbiendo

**Sin darte cuenta, ya estГЎs pensando como un embedded developer:

вҖў Nunca imprimes credenciales en logs
вҖў Usas `if let` en vez de `.unwrap()` cuando el input es externo
вҖў Zeroizas datos sensibles en vez de confiar en que el garbage collector los limpie (spoiler: no hay garbage collector)
вҖў Entiendes que `FreeRtos::delay_ms()` cede el CPU y `thread::sleep()` no

Estas decisiones son la diferencia entre firmware que funciona en el escritorio y firmware que sobrevive 24/7 en un dispositivo real con 400KB de RAM.

### Pregunta 1: ВҝQuГ© hace `mod wifi;` en `main.rs`?

<details>
 <summary>Ver respuesta</summary>

 Le dice a Rust que incluya el archivo `src/wifi.rs` como un mГіdulo llamado `wifi`. El mГіdulo tiene su propio namespace вҖ” para usar sus funciones pГәblicas, escribes `wifi::connect()`. En Rust, todo es **privado por defecto**: solo las funciones marcadas con `pub` son visibles desde fuera del mГіdulo.
 
</details>

### Pregunta 2: ВҝQuГ© es NVS y para quГ© lo usamos?

<details>
 <summary>Ver respuesta</summary>

 **NVS (Non-Volatile Storage)** es un key-value store que vive en una particiГіn de la flash del ESP32. Sobrevive reinicios y power cycles. Lo usamos para guardar credenciales WiFi, device ID, API key, y un flag de provisioning. Es como una mini base de datos persistente.
 
</details>

### Pregunta 3: ВҝQuГ© hacen `Zeroize` y `ZeroizeOnDrop`?

<details>
 <summary>Ver respuesta</summary>

 `Zeroize` agrega el mГ©todo `.zeroize()` que sobreescribe los bytes de un valor con ceros. `ZeroizeOnDrop` llama `.zeroize()` automГЎticamente cuando el struct sale de scope (via el trait `Drop`). Esto previene que credenciales queden en memoria despuГ©s de usarlas вҖ” un atacante con acceso a memoria no podrГӯa leerlas.
 
</details>

### Pregunta 4: ВҝPor quГ© usamos `Arc<Mutex<SecureStorage>>` y no solo `SecureStorage`?

<details>
 <summary>Ver respuesta</summary>

 Porque el servidor HTTP corre handlers en threads separados del main thread. Ambos necesitan acceso a `SecureStorage`. `Arc` permite shared ownership entre threads (cada `.clone()` solo incrementa un contador atГіmico). `Mutex` garantiza que solo un thread accede a la vez (exclusiГіn mutua). Sin `Arc` no podrГӯamos compartir, sin `Mutex` no podrГӯamos mutar de forma segura.
 
</details>

### Pregunta 5: ВҝQuГ© es SoftAP y por quГ© lo necesitamos para provisioning?

<details>
 <summary>Ver respuesta</summary>

 SoftAP (Software Access Point) es cuando el ESP32 crea su propia red WiFi. Lo necesitamos porque no sabemos a quГ© red conectarnos вҖ” eso es exactamente lo que el usuario va a configurar. Es un chicken-and-egg problem: necesitamos red para recibir la config de red. SoftAP lo resuelve creando una red temporal para la configuraciГіn.
 
</details>

### Pregunta 6: ВҝQuГ© hace `move` en un closure?

<details>
 <summary>Ver respuesta</summary>

 Sin `move`, el closure captura variables por referencia. Con `move`, el closure toma **ownership** de las variables capturadas вҖ” se mueven dentro del closure. Es necesario cuando el closure vive mГЎs que la funciГіn donde se define (como los handlers HTTP que siguen corriendo despuГ©s de que `start_provisioning()` entra en el loop). Sin `move`, las referencias serГӯan invГЎlidas y Rust no compila.
 
</details>

### Pregunta 7: ВҝQuГ© significa `impl Peripheral<P = Modem> + 'static`?

<details>
 <summary>Ver respuesta</summary>

 Es un genГ©rico inline que dice: "acepto cualquier tipo que implemente el trait `Peripheral` con parГЎmetro asociado `Modem`, y que tenga lifetime `'static`". Es equivalente a escribir `fn connect<M: Peripheral<P=Modem> + 'static>(modem: M)`. El `'static` garantiza que el valor no contiene referencias temporales вҖ” vive toda la ejecuciГіn del programa.
 
</details>

### Pregunta 8: ВҝPor quГ© retornamos `Box<EspWifi<'static>>` y no solo `EspWifi<'static>`?

<details>
 <summary>Ver respuesta</summary>

 Porque `EspWifi` es un struct grande (buffers, estado del driver, referencias al hardware). `Box` lo mueve al **heap** para: 1) No copiar un struct grande en el stack al retornar, 2) Tener un tamaГұo fijo conocido (un puntero = 4 bytes en 32-bit), 3) Permitir que el caller almacene el driver con lifetime `'static`. Cuando el `Box` sale de scope, Rust libera la memoria automГЎticamente (RAII + Drop).
 
</details>

### Pregunta 9: ВҝCuГЎl es la diferencia entre `WifiDriver` y `EspWifi`?

<details>
 <summary>Ver respuesta</summary>

 `WifiDriver` es el driver de bajo nivel вҖ” da mГЎs control sobre la configuraciГіn. `EspWifi` es un wrapper de alto nivel que combina el driver + interfaces de red (STA y AP). En `wifi.rs` usamos `EspWifi::new()` directamente (defaults). En `provisioning.rs` necesitamos `WifiDriver` + `EspWifi::wrap_all()` para configurar la interfaz AP con IP estГЎtica y DHCP server antes de ensamblarlo todo.
 
</details>

### Pregunta 10: ВҝCuГЎl es la diferencia entre `connect()` y `wait_netif_up()`?

<details>
 <summary>Ver respuesta</summary>

 Son dos pasos distintos: `connect()` вҶ’ autenticaciГіn con el router (espera `WIFI_EVENT_STA_CONNECTED`). `wait_netif_up()` вҶ’ obtener IP por DHCP (espera `IP_EVENT_STA_GOT_IP`). Estar "conectado" al AP **no significa tener IP**. Sin IP no puedes hacer nada en la red (ni HTTP, ni WebSocket, ni DNS).
 
</details>

### Pregunta 11: ВҝQuГ© pasa si escribimos `wifi::connect(...)?;` sin asignar a `_wifi`?

<details>
 <summary>Ver respuesta</summary>

 El `Box<EspWifi>` se **dropea inmediatamente** porque no tiene dueГұo. RAII en acciГіn: cuando el dueГұo muere, el recurso se libera. PerderГӯamos la conexiГіn WiFi instantГЎneamente. Por eso usamos `let _wifi = ...` вҖ” el prefijo `_` indica que no usamos la variable directamente, pero la mantenemos viva en scope.
 
</details>

### Pregunta 12: ВҝPor quГ© hacemos `drop(credentials)` explГӯcitamente?

<details>
 <summary>Ver respuesta</summary>

 Porque `credentials` vivirГӯa hasta el final de `run()` вҖ” que es un loop infinito. Las credenciales quedarГӯan en memoria indefinidamente. Con `drop()` explГӯcito, las destruimos inmediatamente despuГ©s de usarlas. Como `Credentials` implementa `ZeroizeOnDrop`, el drop sobreescribe todos los campos con ceros antes de liberar la memoria.
 
</details>

### Pregunta 13: ВҝPor quГ© el bloque `{ let storage = storage.lock().unwrap(); ... }` tiene llaves?

<details>
 <summary>Ver respuesta</summary>

 Para que el `MutexGuard` (retornado por `.lock()`) se dropee al final del bloque. Si no usamos llaves, el lock quedarГӯa tomado y ningГәn otro cГіdigo podrГӯa acceder a `storage` вҖ” causando un deadlock. El patrГіn "lock, operar, soltar" es fundamental con `Mutex`.
 
</details>

### Pregunta 14: ВҝPor quГ© `esp_restart()` necesita `unsafe`?

<details>
 <summary>Ver respuesta</summary>

 Porque es una funciГіn C llamada via **FFI** (Foreign Function Interface). Rust no puede analizar cГіdigo C, entonces requiere `unsafe` para cualquier llamada FFI directa. `unsafe` no significa "peligroso" вҖ” significa "el compilador no puede verificar la seguridad, yo (el programador) garantizo que es correcto". En la prГЎctica, `esp_restart()` es perfectamente seguro.
 
</details>

### Pregunta 15: ВҝPor quГ© `sysloop.clone()` es "barato"?

<details>
 <summary>Ver respuesta</summary>

 `EspSystemEventLoop` usa `Arc` (Atomic Reference Counted) internamente. Clonar un `Arc` solo incrementa un **contador atГіmico** вҖ” no copia datos. Es O(1) y cuesta unos pocos nanosegundos. MГәltiples partes del cГіdigo pueden tener su propia referencia al mismo event loop sin overhead.
 
</details>

### Pregunta 16: ВҝQuГ© diferencia hay entre `BlockingWifi` y `EspWifi`?

<details>
 <summary>Ver respuesta</summary>

 `EspWifi` es el driver real вҖ” async por naturaleza, dispara eventos y no bloquea. `BlockingWifi` es un **wrapper sГӯncrono** que se suscribe a los eventos del event loop y bloquea el thread hasta que el evento llega. Te da una API simple: `wifi.connect()` bloquea hasta conectar. `BlockingWifi` **no toma ownership** de `EspWifi` вҖ” solo tiene un `&mut` (borrow mutable).
 
</details>

### Pregunta 17: ВҝPor quГ© clonamos `provisioned` y `storage` antes del closure `move`?

<details>
 <summary>Ver respuesta</summary>

 Porque `move` transfiere ownership de las variables capturadas al closure. Si moviГ©ramos los originales, no podrГӯamos usarlos despuГ©s (en el loop de espera). Clonamos los `Arc` (que es barato вҖ” solo incrementa contador atГіmico) y movemos los clones al closure. Los originales siguen vivos para el loop principal.
 
</details>

### Pregunta 18: Si el HTTP server corre en otro thread, ВҝcГіmo sabe el loop principal que el provisioning terminГі?

<details>
 <summary>Ver respuesta</summary>

 Via `Arc<Mutex<bool>>`. El handler HTTP setea `*provisioned_clone.lock().unwrap() = true` al completar. El loop principal chequea `provisioned.lock()` cada segundo. `Arc` permite compartir el bool entre threads, `Mutex` sincroniza el acceso. Es el patrГіn mГЎs simple de comunicaciГіn entre threads.
 
</details>
