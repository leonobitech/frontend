In this step the ESP32 stops being a blinking LED and becomes a real IoT device: it connects to a WiFi network. But not with hardcoded credentials — we implement a **full provisioning system** where the user configures the network through a browser, and the credentials are stored securely in flash.

## ESP32 Flash memory — Physical foundation

Before we talk about NVS, credentials, or partitions, we need to understand **where** everything is stored.

### What is Flash?

<Callout>
 **Flash** is the ESP32-C3's **non-volatile** memory chip. It holds **4MB** (4,194,304 bytes). It's the equivalent of a computer's "hard drive" — data **survives** reboots and power outages.

 Everything lives on this flash: the bootloader, your compiled firmware, WiFi calibration data, and anything your program saves (like WiFi credentials).

</Callout>

### Partitions — Splitting flash into zones

Flash is not an amorphous block. ESP-IDF divides it into **partitions**, each with a purpose. It's like partitioning a hard drive into C: and D:.

```plain text
ESP32-C3 4MB flash (with SINGLE_APP_LARGE)
┌─────────────────────────────────────────────────┐
│ 0x0000  Bootloader (Second stage)     ~28KB     │
├─────────────────────────────────────────────────┤
│ 0x8000  Partition Table               4KB       │
├─────────────────────────────────────────────────┤
│ 0x9000  NVS (Non-Volatile Storage)    ~24KB     │
│         → Where we store WiFi credentials       │
├─────────────────────────────────────────────────┤
│ 0xF000  PHY Init Data                 4KB       │
│         → WiFi radio calibration                │
├─────────────────────────────────────────────────┤
│ 0x10000 App (your firmware)           ~3.5MB    │
│         → All your compiled Rust code           │
└─────────────────────────────────────────────────┘
```

<Callout type="warning">
 **What are** `SINGLE_APP` **and** `SINGLE_APP_LARGE`? ESP-IDF ships **predefined partition tables** — CSV files inside the SDK that define how to split flash. We don't invent them, we pick from them:

 - `SINGLE_APP` → default table, reserves ~1MB for the app
 - `SINGLE_APP_LARGE` → extended table, reserves ~3.5MB for the app
 - `TWO_OTA` (for over-the-air updates) and `CUSTOM` (you define your own CSV) also exist

 The line `CONFIG_PARTITION_TABLE_SINGLE_APP_LARGE=y` in `sdkconfig.defaults` tells ESP-IDF: "use the predefined `singleapp_large` table." Our firmware with WiFi + HTTP server + mbedTLS certificate bundle weighs ~1.1MB — it doesn't fit in `SINGLE_APP`'s ~1MB. That's why we use the LARGE version.

</Callout>

## NVS — Software on top of Flash



**NVS (Non-Volatile Storage)** is not a different hardware from flash — it's an ESP-IDF **software library** that uses a flash **partition** to implement a **key-value** system.

 Think of it like this:

 - **Flash** = the physical medium (like an SSD)
 - **NVS partition** = a reserved chunk of that flash (like a disk partition)
 - **NVS (the library)** = the software that reads/writes key-value to that partition (like a mini filesystem)

 NVS is "Non-Volatile" precisely **because** it lives on flash — flash retains data without power. The name describes the hardware property, not a different piece of hardware.



### How NVS works internally

<Callout>
 **NVS internal structure:**

 - The NVS partition is split into **pages** of 4KB (the size of a flash sector)
 - Each page contains 32-byte **entries**
 - Each entry stores: namespace index + type + key + value
 - Long strings and blobs use multiple consecutive entries

 **Wear leveling:** flash has a limited number of write cycles (~100,000). NVS spreads writes across pages so it never wears out the same one. When a page fills up, it copies the valid entries to another and erases the old one.

 **Namespaces:** group related keys. Our `"credentials"` namespace contains keys like `wifi_ssid`, `wifi_pass`, etc. Another module could use a `"settings"` namespace without conflict.

 **Corruption protection:** if power is lost during a write, NVS can recover the previous state thanks to its transactional design.

</Callout>

With this clear — flash is the hardware, NVS is the software on top — we can see how we use it in code.

### Before flashing — Recommended cleanup

If your ESP32 was previously used with another firmware (or another partition table), it's a good idea to do a **full wipe** before flashing this project. That avoids problems with stale NVS data, incompatible partitions, or cached configuration.**

Step 1 — Erase the entire ESP32 flash:**

```bash
espflash erase-flash
```

<Callout type="warning">
 This command overwrites the **full 4MB** of flash with zeros. It removes:

 - The old bootloader
 - The previous partition table (which may have different offsets)
 - NVS data (credentials, prior configuration)
 - The previous firmware
 - The WiFi PHY calibration (regenerates automatically)

 After this, the ESP32 is essentially factory-fresh. It can't boot until you flash firmware again.

</Callout>

**Step 2 — Clear the build cache:**

```bash
cargo clean
```

<Callout type="warning">
 This deletes the `target/` directory where compiled artifacts live, including the `sdkconfig` generated by ESP-IDF.

 **Why does it matter?** ESP-IDF generates an `sdkconfig` file during the first build, based on `sdkconfig.defaults`. If you change something in `sdkconfig.defaults` (like the partition table), the change **doesn't apply** until you delete the cached `sdkconfig`. `cargo clean` forces everything to be regenerated from scratch.

 **Rule:** every time you modify `sdkconfig.defaults`, run `cargo clean` before compiling.

</Callout>

**Step 3 — Compile and flash:**

```bash
cargo espflash flash --release --monitor
```

<Callout>

 - `--release` — compile with optimizations (smaller, faster firmware)
 - `--monitor` — after flashing, open the serial monitor to watch ESP32 logs

 The first build after `cargo clean` is long because it recompiles all of ESP-IDF (the C SDK). Subsequent builds are incremental and much faster.

</Callout>



**Do you always need to run erase-flash?** No. Only when:

 - You changed the partition table (`sdkconfig.defaults`)
 - The ESP32 had firmware with different partitions
 - You see weird behavior (boot loops, NVS corruption, startup crashes)
 - You want to start with a 100% clean flash

 For day-to-day builds, just `cargo espflash flash --release --monitor`.



## File: `secure_storage.rs` — Secure credentials in flash

### What problem does this file solve?

<Callout>
 We need to save credentials (WiFi, API key, device ID) that **survive reboots**. But we also need those credentials to **not be exposed in memory** after they're used.

 This file builds an **abstraction layer** over NVS that solves both problems. The plan is:

 1. **Define constants** for the NVS namespace and keys (avoid typos, centralize names)
 2. **Create a struct** `Credentials` that groups the 4 sensitive fields, with automatic memory wiping via `Zeroize`
 3. **Create a struct** `SecureStorage` that encapsulates NVS access and exposes a clear API: `is_provisioned()`, `store_credentials()`, `load_credentials()`, `clear_credentials()`

 **Rust pattern in use:** encapsulation via structs + derived traits. Instead of calling NVS directly from `main.rs` with loose strings, we build a typed API where the compiler helps us avoid mistakes. If we mishandle the credentials, Rust tells us at compile time (not in production).

</Callout>

### NVS in code — Namespace and keys

We already know NVS is a library that stores key-value pairs in a flash partition (see previous section). Now let's see how we use it.

The first thing we define in `secure_storage.rs` are the **constants** that identify where and what we save:

```rust
const NVS_NAMESPACE: &str = "credentials";
const KEY_WIFI_SSID: &str = "wifi_ssid";
const KEY_WIFI_PASS: &str = "wifi_pass";
const KEY_API_KEY: &str = "api_key";
const KEY_DEVICE_ID: &str = "device_id";
const KEY_PROVISIONED: &str = "provisioned";
```

<Callout>
 `NVS_NAMESPACE` is like a "folder" inside NVS that groups all our keys. Another module could use a different namespace (e.g. `"settings"`) without conflict.

 The `KEY_*` are the names we read and write each value with. They're constant `&str`s to avoid typos — if we misspell a key, the data simply isn't found.

</Callout>

### How does Rust process these constants?

Visually, these constants create the following structure in flash:

```plain text
Namespace: "credentials"
  ├── wifi_ssid    = "MyWifiNetwork"    (string)
  ├── wifi_pass    = "password123"      (string)
  ├── device_id    = "device-001"       (string)
  ├── api_key      = "sk-abc123..."     (string)
  └── provisioned  = 1                  (u8)
```

<Callout>
 Each key is stored as a **32-byte entry** in flash. Long strings use multiple entries. The `provisioned` flag is a simple `u8` (1 useful byte, 32 bytes per entry) — but efficiency doesn't matter here, persistence does.
</Callout>


### `Credentials` struct with secure wiping

```rust
use zeroize::{Zeroize, ZeroizeOnDrop};

#[derive(Debug, Default, Zeroize, ZeroizeOnDrop)]
pub struct Credentials {
    pub wifi_ssid: String,
    pub wifi_password: String,
    pub api_key: String,
    pub device_id: String,
}
```

### What does `#[derive(...)]` do?

<Callout>
 `#[derive(...)]` is a Rust attribute that **generates trait implementations at compile time**. Without it, you'd have to write each implementation by hand. On our `Credentials` we derive four:

 - **`Debug`** — implements the `Debug` trait so you can print the struct with `{:?}`. Useful for logs (be careful when logging real credentials in production).
 - **`Default`** — generates `Credentials::default()` returning a struct with every field at its default (`String::new()` = empty strings). We use it in `load_credentials()` to build an empty struct and then fill it from NVS.
 - **`Zeroize`** — adds the `.zeroize()` method that overwrites the fields with zeros (see below).
 - **`ZeroizeOnDrop`** — runs `.zeroize()` automatically when the struct goes out of scope. Defense in depth: even if the programmer forgets to call `.zeroize()` explicitly, the data is wiped on drop.

</Callout>

### What is `Zeroize` and why does it matter?

<Callout type="warning">
 When you save a password in a `String`, the bytes live on the heap until the allocator reuses them. In that interval, **any code that reads that memory region — a debugger, a core dump, a crash report sent to a server — sees the password in plaintext**.

 The `zeroize` crate solves this by overwriting the bytes with zeros before freeing the memory. The technical key: it uses `volatile_write` under the hood, so the compiler **does not optimize away** the write as "dead" (which would normally make sense, since nobody reads those bytes afterwards). The write actually reaches RAM.

 **Trade-off:** `Zeroize` adds a handful of instructions per byte at struct destruction. Negligible for 4 credential strings; expensive if you applied it to a 1 MB buffer. For sensitive data = obvious yes.

 **Usage rules in this step:**

 - `store_credentials()` calls `.zeroize()` **explicitly** right after writing to NVS — shrinks the exposure window to microseconds.
 - `load_credentials()` relies on `ZeroizeOnDrop` — the struct returned to the main loop is dropped when its use ends and the bytes wipe themselves.

</Callout>

### `SecureStorage` struct

```rust
pub struct SecureStorage {
    nvs: EspNvs<NvsDefault>,
}
```

<Callout>
 `EspNvs<NvsDefault>` is the handle to an NVS namespace. `NvsDefault` indicates that we use the ESP32's default NVS partition (others exist for OTA, PHY, etc.).

 The struct encapsulates the handle — every read/write operation goes through it. This centralizes access and makes it easier to add encryption in production.

 ### `impl SecureStorage` — Method implementation

 In Rust, a `struct` only defines **data** (fields). The **logic** (methods and associated functions) is defined in a separate `impl` block:

 ```rust
  // 1. The struct defines WHAT data it has
  pub struct SecureStorage {
      nvs: EspNvs<NvsDefault>,
  }

  // 2. The impl defines WHAT it can DO with that data
  impl SecureStorage {
      pub fn new(...) -> Result<Self> { ... }
      pub fn is_provisioned(&self) -> Result<bool> { ... }
      pub fn store_credentials(&mut self, ...) -> Result<()> { ... }
      pub fn load_credentials(&self) -> Result<Credentials> { ... }
      pub fn clear_credentials(&mut self) -> Result<()> { ... }
  }
  ```

 <Callout>
 **Why split struct from impl?** Because in Rust you can have **multiple** `impl` blocks for the same struct (even in different files). This lets you implement external traits, extend functionality, and organize code. The data/behavior split is fundamental to Rust.
 </Callout>



**Associated function vs method — the difference is the first parameter:**

 - **Method** → takes `self` (in some form: `&self`, `&mut self`, `self`). Called with **dot**: `storage.is``_provisioned()`
 - **Associated function** → does NOT take `self`. Called with **double colon**: `SecureStorage::new(nvs)`

 In our case, `new()` is the only **associated function** (equivalent to a "constructor" or "static method" in other languages). The other 4 (`is_provisioned`, `store_credentials`, `load_credentials`, `clear_credentials`) are **methods** because they operate on an existing instance of the struct.**



 Before we look at each method in detail, here's the **full map** — what each one does, why its parameters look that way, and what it returns:

 | **Method** | **Signature** | **What it does and why** |
 | --- | --- | --- |
 | `new()` | `pub fn new(nvs_partition: EspNvsPartition<NvsDefault>) -> Result<Self>` | **Constructor.** Takes the NVS partition handle and opens the `"credentials"` namespace. Returns `Result<Self>` because opening NVS can fail (corrupt partition, etc.). `Self` = `SecureStorage`. |
 | `is_provisioned()` | `pub fn is_provisioned(&self) -> Result<bool>` | **Query.** `&self` = read-only (doesn't mutate). Reads a `u8` flag from NVS. Returns `Result<bool>` — the read can fail, and the value is true/false. |
 | `store_credentials()` | `pub fn store_credentials(&mut self, mut creds: Credentials) -> Result<()>` | **Write.** `&mut self` = needs to mutate NVS (write data). `mut creds` = takes ownership AND permission to mutate (to call `.zeroize()` at the end). Returns `Result<()>` — can fail, but doesn't return data. |
 | `load_credentials()` | `pub fn load_credentials(&self) -> Result<Credentials>` | **Read.** `&self` = read-only NVS access. Returns `Result<Credentials>` — the full struct with all 4 fields, protected by `ZeroizeOnDrop`. |
 | `clear_credentials()` | `pub fn clear_credentials(&mut self) -> Result<()>` | **Factory reset.** `&mut self` = needs to mutate NVS (overwrite with empties). Wipes every key and sets `provisioned = 0`. Returns `Result<()>`. |

 ### Anatomy of a method signature in Rust

 Now let's look at each method in detail.

</Callout>


### `new()` method — Initialization

```rust
pub fn new(nvs_partition: EspNvsPartition<NvsDefault>) -> Result<Self> {
    let nvs = EspNvs::new(nvs_partition, NVS_NAMESPACE, true)?;
    Ok(Self { nvs })
}
```

<Callout>
 ** `EspNvs::new()` parameters:**

 - `nvs_partition` → handle to the NVS partition (obtained with `EspDefaultNvsPartition::take()`)
 - `NVS_NAMESPACE` → `"credentials"` — like a "schema" that groups our keys
 - `true` → create the namespace if it doesn't exist. `false` would return an error if it doesn't exist.

</Callout>


### `is_provisioned()` method — Are credentials already stored?

```rust
pub fn is_provisioned(&self) -> Result<bool> {
    match self.nvs.get_u8(KEY_PROVISIONED) {
        Ok(Some(val)) => Ok(val == 1),
        Ok(None) => Ok(false),
        Err(e) => {
            warn!("Error checking provisioned status: {:?}", e);
            Ok(false)
        }
    }
}
```

<Callout type="warning">
 `get_u8()` returns `Result<Option<u8>>`:

 - `Ok(Some(1))` → the key exists and equals 1 → provisioned
 - `Ok(None)` → the key doesn't exist → not provisioned (first boot)
 - `Err(e)` → NVS error → assume not provisioned (safe default)

 This "triple-state" pattern (`Result<Option<T>>`) is common in Rust when an operation can fail AND the value may not exist.

</Callout>


### `store_credentials()` method — Save to flash

```rust
pub fn store_credentials(&mut self, mut creds: Credentials) -> Result<()> {
    self.nvs.set_str(KEY_WIFI_SSID, &creds.wifi_ssid)?;
    self.nvs.set_str(KEY_WIFI_PASS, &creds.wifi_password)?;
    self.nvs.set_str(KEY_API_KEY, &creds.api_key)?;
    self.nvs.set_str(KEY_DEVICE_ID, &creds.device_id)?;
    self.nvs.set_u8(KEY_PROVISIONED, 1)?;

    creds.zeroize(); // Explicitly wipe credentials from memory
    Ok(())
}
```

<Callout type="warning">
 **Why** `mut creds`? Because `.zeroize()` needs to mutate the fields to overwrite them with zeros. The `mut` on the parameter allows this.

 **Why zeroize explicitly if it already has** `ZeroizeOnDrop`? Because `ZeroizeOnDrop` runs when the struct is dropped at the end of the function. With an explicit `.zeroize()`, we wipe memory **immediately** after saving — minimizing the window of exposure.

</Callout>


### `load_credentials()` method — Read from flash

```rust
pub fn load_credentials(&self) -> Result<Credentials> {
    if !self.is_provisioned()? {
        bail!("Device not provisioned. Run provisioning first.");
    }

    let mut creds = Credentials::default();
    let mut buf = [0u8; 256];

    if let Some(val) = self.nvs.get_str(KEY_WIFI_SSID, &mut buf)? {
        creds.wifi_ssid = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    if let Some(val) = self.nvs.get_str(KEY_WIFI_PASS, &mut buf)? {
        creds.wifi_password = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    if let Some(val) = self.nvs.get_str(KEY_API_KEY, &mut buf)? {
        creds.api_key = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    if let Some(val) = self.nvs.get_str(KEY_DEVICE_ID, &mut buf)? {
        creds.device_id = val.trim_end_matches('\0').to_string();
        buf.zeroize();
    }

    info!("Credentials loaded from NVS");
    Ok(creds)
}
```

<Callout type="warning">
 **Guard clause with** `bail!` — The first thing it does is check `is_provisioned()`. If there are no credentials, it returns an error immediately with `bail!` (an `anyhow` macro that does `return Err(...)`).

 This avoids reading non-existent keys and gives a clear error message to the caller.

</Callout>



** `Credentials::default()` — creates a struct with every field empty (`""`) thanks to the `Default` trait we derived. Then we fill each field from NVS.**



<Callout>
 **The** `buf` **buffer is zeroized after every read.** NVS needs a buffer to copy the read string into. If we don't wipe it, a copy of the password would linger on the stack.

 `.trim_end_matches('\0')` — NVS appends null terminators at the end of the string (a C heritage). We strip them before converting to a Rust `String`.

 `if let Some(val)` — `get_str()` returns `Option<&str>`. If the key doesn't exist, it returns `None` and the field stays empty. This is defensive: even if `is_provisioned()` says `true`, an individual key might be missing.

 **Why a 256-byte buffer?** It's the maximum size we support for API keys. Max SSID = 32, max password = 64. The buffer is reused for every field.

</Callout>


### `clear_credentials()` method — Factory reset

```rust
#[allow(dead_code)]
pub fn clear_credentials(&mut self) -> Result<()> {
    warn!("Clearing all credentials from NVS...");

    self.nvs.set_str(KEY_WIFI_SSID, "")?;
    self.nvs.set_str(KEY_WIFI_PASS, "")?;
    self.nvs.set_str(KEY_API_KEY, "")?;
    self.nvs.set_str(KEY_DEVICE_ID, "")?;
    self.nvs.set_u8(KEY_PROVISIONED, 0)?;

    info!("Credentials cleared");
    Ok(())
}
```

<Callout type="warning">
 **Why overwrite with empty strings instead of removing the keys?** Because NVS doesn't guarantee that `remove()` overwrites the physical bytes in flash — it may just mark the entry as invalid. Writing empty strings forces a real write that overwrites the previous data.
</Callout>



`#[allow(dead_code)]` — This method isn't used in the current code yet (there's no UI for factory reset). Without this attribute, the compiler would warn: "function is never used." We leave it ready for future steps where we'll add a physical button or remote reset command.

 `&mut self` — Needs a mutable reference because it writes to NVS (overwrites every key with empty values and sets `provisioned = 0`).

 `warn!` **instead of** `info!` — We use `warn!` because clearing credentials is a destructive action. In the serial monitor logs, warnings show up in yellow to grab attention.




## File: `provisioning.rs` — WiFi configuration portal

### What problem does this file solve?



The device needs WiFi credentials to connect to the internet, but **we can't hardcode them** in the code (every user has their own network). We also can't ask for them over serial (the device will be on a wall, with no computer attached).

 **Solution:** the ESP32 creates its **own temporary WiFi network** (Access Point), stands up an **HTTP server** with a web form, and the user configures everything from their phone. After saving the credentials, the ESP32 reboots and connects as a regular Station.

 This file implements that whole flow:

 1. **AP constants** — network name, password, channel
 2. **Inline HTML** — setup form and success page (live in `.rodata`, no RAM)
 3. ** `start_provisioning()` function** — configures the AP, brings up the HTTP server, waits for the user, restarts
 4. ** `urlencoding_decode()` helper** — decodes HTML form data

 **Key Rust pattern:** `move` closures to share state (`Arc<Mutex>`) between the HTTP handler (running in another thread) and the main loop.



### SoftAP vs Station

<Callout>
 The ESP32 can operate in two WiFi modes:

 **Station (STA):** connects to an existing router. It's a "client" of the network. This is how it works in normal operation.

 **Access Point (AP / SoftAP):** creates its own WiFi network. Other devices connect to it. This is how it works during provisioning — the ESP32 IS the router.

 **Why not use Station for provisioning?** Because we don't know which network to connect to — that's exactly what the user is going to configure! Chicken-and-egg problem. SoftAP solves it: it creates a temporary network to receive the configuration.
 </Callout>

### Signature of `start_provisioning()`

```rust
pub fn start_provisioning(
    modem: impl peripheral::Peripheral<P = esp_idf_svc::hal::modem::Modem> + 'static,
    sysloop: EspSystemEventLoop,
    storage: Arc<Mutex<SecureStorage>>,
) -> Result<()> {
```

It takes three things:

- `modem` — the WiFi peripheral (ownership, consumed here)
- `sysloop` — the system event bus (cloneable, cheap)
- `storage` — shared NVS access, wrapped in `Arc<Mutex<>>` because the HTTP handler runs in another thread

<Callout>
 **Why** `Arc<Mutex<SecureStorage>>`?

 The POST HTTP handler runs in a separate thread (created by the HTTP server). It needs storage access to save credentials. `Arc` lets us have multiple owners; `Mutex` guarantees exclusive access.

 - `Arc` → multiple ownership (reference counting)
 - `Mutex` → exclusive access (lock/unlock)
 - Without `Arc`: we can't share across threads
 - Without `Mutex`: data race (undefined behavior)

</Callout>

We'll walk through the function step by step, in the same order as the code.

### Step 4 — Prepare shared variables


Before creating the HTTP server, we set up the variables the POST handler needs:

- `provisioned` — `bool` flag wrapped in `Arc<Mutex<>>`. The POST handler sets it to `true` when the user completes the form. The main loop reads it to know when to restart.
- **Clones** — `provisioned_clone` and `storage_clone` are the ones we move into the POST closure with `move`. The originals stay around for the main loop.

<Callout type="warning">
 **Why clone before the closure?**

 A `move` closure takes ownership of everything it captures. If we passed `provisioned` directly into the closure, the main loop wouldn't be able to read it anymore. By cloning the `Arc`, both (closure and loop) hold their own reference to the same data.

</Callout>

### Step 6 — POST handler: receive, validate, save

```rust
server.fn_handler(
    "/provision",
    Method::Post,
    move |mut req| -> core::result::Result<(), esp_idf_svc::io::EspIOError> {
        // Read the POST body (URL-encoded form)
        let mut body = [0u8; 512];
        let len = req.read(&mut body)?;
        let body_str = std::str::from_utf8(&body[..len]).unwrap_or("");

        // Parse form fields
        let mut ssid = String::new();
        let mut password = String::new();
        let mut device_id = String::new();
        let mut api_key = String::new();

        for pair in body_str.split('&') {
            if let Some((key, value)) = pair.split_once('=') {
                let decoded = urlencoding_decode(value);
                match key {
                    "ssid" => ssid = decoded,
                    "password" => password = decoded,
                    "device_id" => device_id = decoded,
                    "api_key" => api_key = decoded,
                    _ => {}
                }
            }
        }

        // Validate required fields
        if ssid.is_empty() || password.is_empty() || device_id.is_empty() {
            error!("Missing required fields in provisioning form");
            let mut response = req.into_status_response(400)?;
            response.write_all(b"Missing required fields")?;
            return Ok(());
        }

        // Build credentials and save them in NVS
        let creds = Credentials {
            wifi_ssid: ssid,
            wifi_password: password,
            api_key,
            device_id,
        };

        if let Ok(mut storage) = storage_clone.lock() {
            if let Err(e) = storage.store_credentials(creds) {
                error!("Failed to store credentials: {:?}", e);
                let mut response = req.into_status_response(500)?;
                response.write_all(b"Failed to store credentials")?;
                return Ok(());
            }
        }

        // Mark provisioning as completed
        if let Ok(mut p) = provisioned_clone.lock() {
            *p = true;
        }

        // Send the success page
        let mut response = req.into_ok_response()?;
        response.write_all(HTML_SUCCESS.as_bytes())?;
        Ok(())
    },
)?;
```

This is the most important handler. The `move` keyword transfers ownership of `provisioned_clone` and `storage_clone` into the closure.

The full flow:

1. **Read body** — `req.read``(&mut body)` reads the form bytes. A 512-byte buffer is enough for the credentials
2. **Parse** — the body comes as `ssid=My+Net&password=abc%21`. Split on `&`, then on `=`, and decode URL encoding
3. **Validate** — if SSID, password, or device_id are missing, return 400
4. **Save** — build a `Credentials` and persist it in NVS via `storage_clone`
5. **Signal** — set `provisioned_clone` to `true` so the main loop knows it's time to reboot
6. **Respond** — send `HTML_SUCCESS` to the browser

<Callout type="warning">
 **What does** `move` **do in the closure?**

 **Without** `move`: the closure tries to take **references** to `provisioned_clone` and `storage_clone`. But those variables live in `start_provisioning()`, and the closure outlives them (in the HTTP server thread). The references would be invalid → **doesn't compile**.

 **With** `move`: the closure takes **ownership** of the clones. The data moves into the closure and lives as long as it does. The compiler is happy.

</Callout>

<Callout type="warning">
 **Why** `if let Ok(mut storage)` **and not** `.unwrap()`?

 In embedded, if the thread holding the lock panics, the Mutex becomes "poisoned." `.lock()` returns `Err` in that case. With `if let Ok(...)` the handler simply skips the save — much more robust than crashing the whole device with `.unwrap()`.

</Callout>

</Callout>


### urlencoding_decode() function — Decoding form data

<Callout>
 **What is URL encoding and why do we need to decode it?**

When the user fills out the form at `192.168.4.1` and hits "Save & Connect," the browser sends the data as `application/x-www-form-urlencoded`. This format has two escape rules:

- Spaces → `+` (the browser replaces every space with a plus sign)
- Special characters → `%XX` (where XX is the hexadecimal code of the byte)

For example, if the SSID is "My Net WiFi" and the password is "p@ssw0rd!", the browser sends:
`ssid=My+Net+WiFi&password=p%40ssw0rd%21`

Our function reverses this: `+` → space, `%40` → `@`, `%21` → `!`.
</Callout>

### The code

```rust
fn urlencoding_decode(input: &str) -> String {
    let mut result = String::with_capacity(input.len());
    let mut chars = input.chars().peekable();

    while let Some(c) = chars.next() {
        match c {
            '+' => result.push(' '),
            '%' => {
                let hex: String = chars.by_ref().take(2).collect();
                if let Ok(byte) = u8::from_str_radix(&hex, 16) {
                    result.push(byte as char);
                }
            }
            _ => result.push(c),
        }
    }

    result
}
```

### Code analysis

<Callout>
 **Pre-allocation and memory in embedded**

`String::with_capacity(input.len())` — Pre-allocates heap memory the size of the input. The decoded string is always ≤ the encoded one (the 3-char `%XX` sequences become 1 char), so this capacity is sufficient.

**Why does it matter on the ESP32?** The ESP32-C3 has only ~400KB of RAM. Without `with_capacity`, Rust would start with a small buffer and re-allocate every time it filled (doubling its size). Each re-allocation means: ask for new memory → copy everything → free the old one. On a microcontroller with limited heap, this can cause memory fragmentation. Pre-allocating eliminates that risk.
</Callout>



**Iterator pattern: while let + by_ref().take(2).collect()**

`input.chars().peekable()` — Creates a Unicode-character iterator. `peekable()` lets you look at the next character without consuming it (we don't use it directly here, but it's good practice for parsers).

`while let Some(c) = chars.next()` — Idiomatic Rust pattern: "while there's a next character, extract it into `c`." Combines iteration + destructuring.

`match c` — Pattern matching with three arms:
- `'+'` → push a space (HTML form encoding convention)
- `'%'` → take the next 2 chars as hex:
 `chars.by_ref().take(2).collect()` — Three operations:
 1. `by_ref()` borrows the iterator (no ownership transfer)
 2. `take(2)` limits to the next 2 characters
 3. `collect()` joins them into a `String`
- `_` → any other character is copied as-is.



<Callout>
 **Safe conversion and error handling without panics**

`u8::from_str_radix(&hex, 16)` — Converts the hex string to a byte (`u8`). Returns `Result<u8>`, NOT a u8 directly.

`if let Ok(byte)` — If the hex is invalid (e.g. `%ZZ`), it just ignores it instead of panicking. **In embedded we NEVER use.unwrap() with external data** — a panic reboots the chip.

`byte as char` — Cast from `u8` to `char`. It works because the URL-encoded values in WiFi forms are within the ASCII range (0-127).

**Why not use a crate?** `urlencoding` and `percent-encoding` exist, but adding a crate for 15 lines isn't worth it: more binary in limited flash, more bug surface, more compile time. Practical rule: if you can write it in under 20 lines and it covers your case, don't add a crate.
</Callout>

### Signature of `connect()`

```rust
pub fn connect(
    ssid: &str,
    password: &str,

    sysloop: EspSystemEventLoop,
```

### `impl Trait` in parameters

<Callout>
 `impl peripheral::Peripheral<P = ...Modem> + 'static` — Instead of taking a concrete type, we use `impl Trait`. This tells Rust: "I accept any type that implements this trait." It's syntactic sugar for generics — the compiler generates a concrete version at compile time (monomorphization).

**Why not a concrete type?** Because ESP-IDF can change the modem's internal type between versions. With `impl Trait` our code is flexible: it works with any type that satisfies the `Peripheral` contract.

`+ 'static` — Means the value contains no temporary references. It's required because the WiFi driver lives in a `Box` that may live indefinitely — it can't depend on data that's destroyed earlier.
</Callout>

### `Box<EspWifi<'static>>` — Why Box?



`Result<Box<EspWifi<'static>>>` — The function returns the WiFi driver wrapped in a `Box` (heap allocation). Why not return it directly on the stack?

**1.** `EspWifi` is a large struct (~200+ bytes) with internal buffers. Moving it through the stack means copying all those bytes every time it's returned or assigned.
**2.** With `Box`, only a pointer (4 bytes on the ESP32) is moved. The real struct lives on the heap and isn't copied.
**3.** `'static` indicates that the driver can live for the whole program — it isn't tied to any function's lifetime.**

RAII in action:** while the `Box` exists, the WiFi connection is held. If it's dropped (goes out of scope or you call `drop()`), `EspWifi`'s destructor turns the WiFi radio off automatically.



### `EspSystemEventLoop` — The event bus

<Callout>
 `EspSystemEventLoop` is the ESP-IDF system event bus. WiFi, IP, Bluetooth — everything goes through this bus. It's a singleton obtained with `EspSystemEventLoop::take()` in `main.rs` and passed in as a parameter.

**Why can it be passed to multiple functions?** Because internally it uses `Arc` (atomic reference-counted pointer). Each `.clone()` only increments a counter — it doesn't duplicate the bus. It's like having multiple "subscribers" to the same event bus.
</Callout>

### Build the WiFi driver

<Callout>
 `EspWifi::new()` **takes ownership of the modem.** After this line, `modem` no longer exists (moved). No one else can use the WiFi radio.

 **Parameters:**

 - `sysloop.clone()` → event-loop clone. `EspSystemEventLoop` uses `Arc` internally, so cloning is cheap (only increments an atomic counter, no data copy)

### BlockingWifi: the synchronous wrapper

```rust
let mut wifi = BlockingWifi::wrap(&mut esp_wifi, sysloop)?;
```

<Callout type="warning">
 `EspWifi` is **async** by nature — it fires events and doesn't block. `BlockingWifi` is a wrapper that:

 1. Subscribes to events on the event loop
 2. **Blocks** the current thread until the event arrives
 3. Gives you a synchronous API: `wifi.connect()` blocks until connected

 **Why block?** In our firmware, WiFi is sequential: we can't do anything until we have a connection. Async would be over-engineering for this case.

 **Borrow vs Move:** `&mut esp_wifi` is a **mutable borrow**. `BlockingWifi` does NOT take ownership — it just borrows. `esp_wifi` is still the owner of the driver. This matters because at the end we return `esp_wifi`, not the `BlockingWifi`.

</Callout>


### Initial setup and start

<Callout>
 **Why configure with default() if we change it afterwards?**

The ESP-IDF WiFi driver requires a configuration **before** calling `start()`, and we need `start()` before we can `scan()`. The flow is:

1. `set_configuration(default)` → gives the driver a minimal config so it can start
2. `start()` → turn on the WiFi radio (hardware)
3. `scan()` → scan for available networks (needs the radio on)
4. `set_configuration(real)` → reconfigure with SSID, password, and the discovered channel

`ClientConfiguration::default()` creates a struct with all fields at their defaults (empty SSID, no password, auto channel). It's enough for the radio to start and be able to scan.
</Callout>


### Network scan

```rust
let ap_infos = wifi.scan()?;
let target_ap = ap_infos.into_iter().find(|ap| ap.ssid == ssid);
let channel = target_ap.as_ref().map(|ap| ap.channel);
```

<Callout>
 **Why scan?** To get the AP's exact **channel**. If we hand the right channel to the driver, the connection is faster (it doesn't need to try all 13 channels).

 **Iterator breakdown:**

 - `.into_iter()` — **consumes** the Vec (move, no borrow). Each AP is moved out of the Vec.
 - `.find(|ap| ap.ssid == ssid)` — finds the first element matching the predicate. Returns `Option<AccessPointInfo>`.
 - `.as_ref()` — turns `&Option<T>` into `Option<&T>` (so we don't move the value)
 - `.map(|ap| ap.channel``)` — transforms the inner value if it's `Some`. Result: `Option<u8>` with the channel.

 **Closures:** `|ap| ap.ssid == ssid` is a closure (anonymous function). `|ap|` is the parameter, `ap.ssid == ssid` is the body. The closure **captures** `ssid` by reference from the outer scope.

</Callout>


### Configure with credentials

```rust
wifi.set_configuration(&Configuration::Client(ClientConfiguration {
    ssid: ssid.try_into().expect("SSID too long"),
    password: password.try_into().expect("Password too long"),
    channel,
    auth_method,
    ..Default::default()
}))?;
```

### `try_into()` — Fallible conversion

<Callout>
 `ssid.try_into().expect("SSID too long")` — `ClientConfiguration` expects a `heapless::String<32>` (a 32-byte max string with no heap). Our `ssid` is a regular `&str`, so we need to convert it.

`try_into()` returns `Result` because it can fail: if the SSID is more than 32 bytes, it doesn't fit and returns `Err`.

**Why .expect() here and not if let?** Because an SSID longer than 32 bytes is a programming error on our side (WiFi SSIDs are at most 32 bytes by IEEE 802.11). It's not malicious external input — it's a bug. In that case, `expect()` with a descriptive message is acceptable.
</Callout>

### `..Default::default()` — Struct update syntax

<Callout>
 `..Default::default()` — Struct update syntax. It means: "for every field I didn't list explicitly, use the default values." `ClientConfiguration` has many fields (bssid, scan_method, pmf_cfg, etc.), but we only care about 4: `ssid`, `password`, `channel`, and `auth_method`.

Without this syntax, we'd have to list every field manually — tedious and fragile when ESP-IDF updates. With `..` the code survives if ESP-IDF adds new fields in future versions.
</Callout>

### Connect and obtain an IP

```rust
wifi.connect()?;       // Block until connected to the AP
wifi.wait_netif_up()?; // Block until DHCP assigns an IP
```

<Callout>
 **Being "connected" to the AP doesn't mean having an IP.** Two steps:

 1. `connect()` → wait for the `WIFI_EVENT_STA_CONNECTED` event (successful authentication with the router)
 2. `wait_netif_up()` → wait for the `IP_EVENT_STA_GOT_IP` event (DHCP assigns IP)

 Without an IP you can't do anything on the network (no HTTP, no WebSocket, no DNS). That's why we wait for both events.

</Callout>


### Get the IP and return the driver

```rust
let ip_info = wifi.wifi().sta_netif().get_ip_info()?;

info!("WiFi connected!");
info!("IP: {}", ip_info.ip);
info!("Gateway: {}", ip_info.subnet.gateway);
info!("Mask: {}", ip_info.subnet.mask);

Ok(Box::new(esp_wifi))
```

<Callout>
 **Access chain and the final return**

`wifi.wifi().sta_netif().get_ip_info()` — Method chain to reach the IP:
- `wifi.wifi()` → access the `EspWifi` inside `BlockingWifi`
- `.sta_netif()` → get the Station network interface
- `.get_ip_info()` → read IP, gateway, and mask assigned by DHCP

`Ok(Box::new(esp_wifi))` — Something subtle happens here: `wifi` (the `BlockingWifi`) is dropped at function exit because we don't return it. But the WiFi connection **isn't lost** — `BlockingWifi` was just a synchronous wrapper around `esp_wifi` (it took `&mut`, not ownership). The real driver (`esp_wifi`) survives inside the `Box` and keeps the radio on and connected.
</Callout>


## File: `main.rs` — Evolution from Step 1

### Module declarations

```rust
mod provisioning;
mod secure_storage;
mod wifi;
```

<Callout>
 Each `mod X;` tells Rust: "include the file `src/X.rs` as a module called `X`." The module has its own **namespace** — to use its public functions, you write `wifi::connect()` or `provisioning::start_provisioning()`.

 In Rust, **everything is private by default**. Only functions marked `pub` are visible from outside the module.

</Callout>


### `main()` → `run()` pattern

```rust
fn main() {
    esp_idf_svc::sys::link_patches();
    esp_idf_svc::log::EspLogger::initialize_default();

    if let Err(e) = run() {
        error!("Fatal error: {:?}", e);
        std::thread::sleep(Duration::from_secs(10));
        unsafe { esp_idf_svc::sys::esp_restart(); }
    }
}

fn run() -> anyhow::Result<()> {
    // all the logic here, with ? freely available
}
```

<Callout type="warning">
 **Why split** `main()` **from** `run()`?

 `main()` without a return type can't use the `?` operator. By moving the logic into `run() -> Result<()>`, we can use `?` freely. If something fails, the error "bubbles" up to `main()`, which:

 1. Logs the full error (`{:?}` uses the `Debug` trait)
 2. Waits 10 seconds (so you can read the error on serial)
 3. Reboots the chip via `esp_restart()`

 **The parent firmware uses exactly this pattern.**

</Callout>


### System initialization — Peripherals, EventLoop, and LED

```rust
let peripherals = Peripherals::take()?;
let sysloop = EspSystemEventLoop::take()?;

let nvs_partition = EspDefaultNvsPartition::take()?;

let mut led = PinDriver::output(peripherals.pins.gpio8)?;
```

<Callout>
 **Three ESP-IDF system singletons**

Each `::take()` can only be called once in the entire program. If called twice, it returns `Err`. This pattern guarantees exclusive ownership:

- `Peripherals::take()` — Takes ownership of EVERY GPIO pin, the WiFi modem, the I2C/SPI buses, etc. The `peripherals.modem` field will be passed to `wifi::connect()` or `start_provisioning()` (move).
- `EspSystemEventLoop::take()` — The event bus. Internally it uses `Arc`, so `.clone()` is cheap.
- `EspDefaultNvsPartition::take()` — The default NVS partition in flash.

`PinDriver::output(peripherals.pins.gpio8)` — Inherited from Step 1. Configures GPIO8 as a digital output for the LED. Here it works as a heartbeat: if the LED blinks, the firmware is running.
</Callout>


### NVS initialization

```rust
let nvs_partition = EspDefaultNvsPartition::take()?;
```

<Callout>
 Another **singleton** — like `Peripherals::take()` and `EspSystemEventLoop::take()`. Can only be called once. Returns a handle to the ESP32's default NVS partition.

 This handle is passed to `SecureStorage::new()` to initialize NVS access. `.clone()` is cheap because internally it uses `Arc`.

</Callout>


### SecureStorage with `Arc<Mutex<>>`

```rust
let storage = SecureStorage::new(nvs_partition.clone())?;
let storage = Arc::new(Mutex::new(storage));
```

<Callout>
 **Shadowing + safe concurrency**

Line 1: `SecureStorage::new()` opens the NVS namespace and creates the handle. `nvs_partition.clone()` is required because `SecureStorage::new()` takes ownership of the partition, and we'll also need it for WiFi.

Line 2: `let storage = Arc::new(Mutex::new(storage))` — **Shadowing:** reusing the name `storage` is valid in Rust. The first variable is consumed (moved) and the name now points to the `Arc<Mutex<SecureStorage>>`.**

Why `Arc<Mutex<>>`?** Because `storage` is used in two contexts: here in `run()` (main thread) and inside the provisioning HTTP handler (another server thread). `Arc` allows shared ownership across threads, `Mutex` guarantees that only one accesses it at a time.
</Callout>


### Provisioning check

```rust
let is_provisioned = {
    let storage = storage.lock().unwrap();
    storage.is_provisioned()?
};
```

<Callout type="warning">
 **Why the `{ ... }` block?** So that the `MutexGuard` (returned by `.lock()`) is dropped at the end of the block. Otherwise, the lock would stay held and no other code could touch `storage` — deadlock.

 This "lock, operate, release" pattern is fundamental when using `Mutex`.

</Callout>


### Branch: Provisioning mode

```rust
if !is_provisioned {
    warn!("Device not provisioned!");
    info!("Starting provisioning mode...");
    info!("Connect to WiFi: 'Leonobitech-Setup' / Password: 'setup1234'");
    info!("Then open http://192.168.4.1 in your browser");

    // start_provisioning() NEVER returns — it reboots the chip on completion
    provisioning::start_provisioning(peripherals.modem, sysloop, storage)?;

    return Ok(());
}
```

<Callout>
 **The big branch: provisioning vs normal**

All of `main.rs` splits into two paths based on `is_provisioned`:

**If NOT provisioned:**
1. Log instructions for the user (SSID, password, URL)
2. Call `start_provisioning()` which creates the Access Point and HTTP server
3. `start_provisioning()` internally calls `esp_restart()` once credentials are saved — the chip reboots and next time `is_provisioned` will be `true`
4. `return Ok(())` is unreachable (never executed because the chip already restarted), but Rust needs it for the control flow to compile.

**Ownership:** Note that `peripherals.modem` is passed by move. If this branch runs, the modem is consumed here and the code after the `if` never runs. If it doesn't run (already provisioned), the modem stays available for `wifi::connect()`. Rust verifies this at compile time — the two branches are mutually exclusive.
</Callout>


### Branch: Normal mode — Load credentials

```rust
info!("Device is provisioned, loading credentials...");

let credentials = {
    let storage = storage.lock().unwrap();
    storage.load_credentials()?
};

info!("Device ID: {}", credentials.device_id);
info!("Connecting to WiFi: {}", credentials.wifi_ssid);
```

<Callout>
 **Same block pattern for the Mutex**

Again the `{ let guard = mutex.lock(); ... }` pattern — the block limits the `MutexGuard`'s scope so it's released immediately after reading. Without the block, the lock would stay held until the end of `run()` — blocking any other access to storage.

After the block, `credentials` is an owned `Credentials` struct (not a reference). It contains copies of the WiFi SSID, password, API key, and device ID strings read from flash. Logging the device ID and SSID is safe — the password is never printed.
</Callout>

### `_wifi` variable — Keep alive without using

```rust
let _wifi = wifi::connect(
    &credentials.wifi_ssid,
    &credentials.wifi_password,
    peripherals.modem,
    sysloop,
)?;
```

<Callout type="warning">
 **CRITICAL CONCEPT.** The `_` prefix indicates that we don't use the variable, but we **DO need to keep it alive**. If it's dropped (goes out of scope), the WiFi connection is lost.

 **If you wrote:**

 ```rust
  wifi::connect(...)?;  // unassigned
  ```

 The `Box<EspWifi>` would be dropped **immediately** and you'd lose WiFi. RAII (Resource Acquisition Is Initialization) in action: when the owner dies, the resource is released.

 **Common embedded pattern:** "I don't need to interact with this value, but I need it to exist so that the underlying resource stays active."

</Callout>


### Explicit drop of credentials

```rust
drop(credentials);
info!("Credentials zeroized from memory");
```

<Callout>
 `drop()` destroys the value immediately (doesn't wait for it to go out of scope). Since `Credentials` implements `ZeroizeOnDrop`, the drop overwrites every field with zeros before freeing the memory.

 **Why not let it drop on its own?** Because `credentials` would live until the end of `run()` — which is an infinite loop. The credentials would stay in memory indefinitely. With explicit `drop()`, we wipe them as soon as we're done.

</Callout>


### `unsafe` block — FFI to C

```rust
unsafe {
    esp_idf_svc::sys::esp_restart();
}
```

<Callout type="warning">
 `unsafe` doesn't mean "dangerous." It means: "the compiler can't verify the safety of this code; I (the programmer) guarantee it's correct."

 `esp_restart()` is a C function called via FFI (Foreign Function Interface). Rust can't analyze C code, so it requires `unsafe` for any direct FFI call.

 In practice, `esp_restart()` is perfectly safe — it just reboots the chip cleanly. The `unsafe` is a formality of the type system.

</Callout>


### Main loop — LED heartbeat

```rust
info!("Entering main loop...");
loop {
    led.set_high()?;
    info!("LED ON");
    FreeRtos::delay_ms(500);

    led.set_low()?;
    info!("LED OFF");
    FreeRtos::delay_ms(500);
}
```

<Callout>
 **The infinite loop and why run() never returns Ok**

This `loop {}` has no `break` — it's an intentional infinite loop. In embedded, the firmware never "ends": there's always a main loop running. If `run()` exits, it's because a `?` propagated an error (e.g. GPIO failure), and `main()` catches it, waits 10s, and reboots the chip.

`FreeRtos::delay_ms(500)` — Not a busy-wait like `std::thread::sleep()`. FreeRTOS is the ESP32's real-time operating system. `delay_ms()` yields the CPU to the scheduler so other tasks (like the WiFi stack) can run during those 500ms. If we used a busy-wait, we'd block WiFi.

**Inherited from Step 1:** for now the LED only blinks as a heartbeat ("I'm alive"). In future steps, this loop will handle WebSocket messages, telemetry, and scheduling.
</Callout>

## Step 2 reflection — What you really learned

In Step 1 you turned on an LED. It seemed like a small thing, but you set up the toolchain, compiled for RISC-V, and wrote your first `unsafe` block.

**In Step 2, your ESP32 stopped being a toy and became a real device.** It now has persistence (NVS), security (Zeroize), connectivity (WiFi), and a provisioning flow any user can complete from their phone. That's not a tutorial — that's production firmware.

But the most important thing isn't what you built — it's how you built it. Look at the patterns that showed up over and over:



**Patterns that are now part of your vocabulary**

 **Ownership and move** — The modem moves to `connect()` or `start_provisioning()`, never to both. Rust forces you to decide who's the owner. This isn't compiler bureaucracy — it's the reason your firmware doesn't have race conditions.

 **RAII** — `_wifi` keeps the connection alive. `drop(credentials)` wipes passwords from memory. There's no `cleanup()`, no `free()`, no chance to forget. The destructor always runs.

 **Result + ?** — Every operation that can fail returns `Result`. The `?` propagates errors upward without `try/catch`. If something fails, `main()` catches it, waits, and reboots. Zero silent exceptions.

 **`Arc<Mutex<T>>`** — Sharing data between threads safely. The HTTP handler and the main loop access the same `SecureStorage` without data races. In C this would be a global with a semaphore someone forgets to take. In Rust, the compiler doesn't let you compile without the Mutex.





**The embedded philosophy you're absorbing**

Without realizing it, you're already thinking like an embedded developer:

- You don't add crates for what you can solve in 15 lines
- You pre-allocate memory instead of letting the allocator decide
- You never print credentials in logs
- You use `if let` instead of `.unwrap()` when input is external
- You zeroize sensitive data instead of trusting the garbage collector to clean up (spoiler: there's no garbage collector)
- You understand that `FreeRtos::delay_ms()` yields the CPU and `thread::sleep()` doesn't

These decisions are the difference between firmware that works on the desk and firmware that survives 24/7 on a real device with 400KB of RAM.

### Question 1: What does `mod wifi;` do in `main.rs`?

<details>
 <summary>Show answer</summary>

 It tells Rust to include the file `src/wifi.rs` as a module named `wifi`. The module has its own namespace — to use its public functions, you write `wifi::connect()`. In Rust, everything is **private by default**: only functions marked `pub` are visible from outside the module.

</details>

### Question 2: What is NVS and what do we use it for?

<details>
 <summary>Show answer</summary>

 **NVS (Non-Volatile Storage)** is a key-value store that lives on a flash partition of the ESP32. It survives reboots and power cycles. We use it to save WiFi credentials, device ID, API key, and a provisioning flag. It's like a tiny persistent database.

</details>

### Question 3: What do `Zeroize` and `ZeroizeOnDrop` do?

<details>
 <summary>Show answer</summary>

 `Zeroize` adds the `.zeroize()` method that overwrites a value's bytes with zeros. `ZeroizeOnDrop` automatically calls `.zeroize()` when the struct goes out of scope (via the `Drop` trait). This prevents credentials from lingering in memory after use — an attacker with memory access wouldn't be able to read them.

</details>

### Question 4: Why do we use `Arc<Mutex<SecureStorage>>` and not just `SecureStorage`?

<details>
 <summary>Show answer</summary>

 Because the HTTP server runs handlers in threads separate from the main thread. Both need access to `SecureStorage`. `Arc` allows shared ownership across threads (each `.clone()` only increments an atomic counter). `Mutex` guarantees that only one thread accesses it at a time (mutual exclusion). Without `Arc` we couldn't share, without `Mutex` we couldn't mutate safely.

</details>

### Question 5: What is SoftAP and why do we need it for provisioning?

<details>
 <summary>Show answer</summary>

 SoftAP (Software Access Point) is when the ESP32 creates its own WiFi network. We need it because we don't know which network to connect to — that's exactly what the user is going to configure. It's a chicken-and-egg problem: we need a network to receive the network config. SoftAP solves it by creating a temporary network for configuration.

</details>

### Question 6: What does `move` do in a closure?

<details>
 <summary>Show answer</summary>

 Without `move`, the closure captures variables by reference. With `move`, the closure takes **ownership** of captured variables — they're moved into the closure. It's required when the closure outlives the function where it's defined (like the HTTP handlers that keep running after `start_provisioning()` enters its loop). Without `move`, the references would be invalid and Rust won't compile.

</details>

### Question 7: What does `impl Peripheral<P = Modem> + 'static` mean?

<details>
 <summary>Show answer</summary>

 It's an inline generic that says: "I accept any type that implements the `Peripheral` trait with associated parameter `Modem`, and that has lifetime `'static`." It's equivalent to writing `fn connect<M: Peripheral<P=Modem> + 'static>(modem: M)`. The `'static` guarantees the value contains no temporary references — it lives for the whole program execution.

</details>

### Question 8: Why do we return `Box<EspWifi<'static>>` and not just `EspWifi<'static>`?

<details>
 <summary>Show answer</summary>

 Because `EspWifi` is a large struct (buffers, driver state, hardware references). `Box` moves it to the **heap** for: 1) Avoiding copying a big struct on the stack at return, 2) Having a known fixed size (a pointer = 4 bytes on 32-bit), 3) Letting the caller store the driver with a `'static` lifetime. When the `Box` goes out of scope, Rust frees the memory automatically (RAII + Drop).

</details>

### Question 9: What's the difference between `WifiDriver` and `EspWifi`?

<details>
 <summary>Show answer</summary>

 `WifiDriver` is the low-level driver — it gives you more control over configuration. `EspWifi` is a high-level wrapper that bundles the driver with the network interfaces (STA and AP). In `wifi.rs` we use `EspWifi::new()` directly (defaults). In `provisioning.rs` we need `WifiDriver` + `EspWifi::wrap_all()` to configure the AP interface with a static IP and DHCP server before assembling everything.

</details>

### Question 10: What's the difference between `connect()` and `wait_netif_up()`?

<details>
 <summary>Show answer</summary>

 They're two distinct steps: `connect()` → authenticate with the router (waits for `WIFI_EVENT_STA_CONNECTED`). `wait_netif_up()` → obtain an IP via DHCP (waits for `IP_EVENT_STA_GOT_IP`). Being "connected" to the AP **doesn't mean having an IP**. Without an IP you can't do anything on the network (no HTTP, no WebSocket, no DNS).

</details>

### Question 11: What happens if we write `wifi::connect(...)?;` without assigning to `_wifi`?

<details>
 <summary>Show answer</summary>

 The `Box<EspWifi>` is **dropped immediately** because it has no owner. RAII in action: when the owner dies, the resource is released. We'd lose the WiFi connection instantly. That's why we use `let _wifi = ...` — the `_` prefix indicates we don't use the variable directly, but we keep it alive in scope.

</details>

### Question 12: Why do we call `drop(credentials)` explicitly?

<details>
 <summary>Show answer</summary>

 Because `credentials` would live until the end of `run()` — which is an infinite loop. The credentials would linger in memory indefinitely. With explicit `drop()`, we destroy them right after using them. Since `Credentials` implements `ZeroizeOnDrop`, the drop overwrites every field with zeros before freeing the memory.

</details>

### Question 13: Why does the `{ let storage = storage.lock().unwrap(); ... }` block have braces?

<details>
 <summary>Show answer</summary>

 So that the `MutexGuard` (returned by `.lock()`) is dropped at the end of the block. Without braces, the lock would stay held and no other code could touch `storage` — causing a deadlock. The "lock, operate, release" pattern is fundamental with `Mutex`.

</details>

### Question 14: Why does `esp_restart()` need `unsafe`?

<details>
 <summary>Show answer</summary>

 Because it's a C function called via **FFI** (Foreign Function Interface). Rust can't analyze C code, so it requires `unsafe` for any direct FFI call. `unsafe` doesn't mean "dangerous" — it means "the compiler can't verify safety, I (the programmer) guarantee it's correct." In practice, `esp_restart()` is perfectly safe.

</details>

### Question 15: Why is `sysloop.clone()` "cheap"?

<details>
 <summary>Show answer</summary>

 `EspSystemEventLoop` uses `Arc` (Atomic Reference Counted) internally. Cloning an `Arc` only increments an **atomic counter** — it doesn't copy data. It's O(1) and costs a few nanoseconds. Multiple parts of the code can hold their own reference to the same event loop without overhead.

</details>

### Question 16: What's the difference between `BlockingWifi` and `EspWifi`?

<details>
 <summary>Show answer</summary>

 `EspWifi` is the real driver — async by nature, fires events and doesn't block. `BlockingWifi` is a **synchronous wrapper** that subscribes to events on the event loop and blocks the thread until the event arrives. It gives you a simple API: `wifi.connect()` blocks until connected. `BlockingWifi` **does not take ownership** of `EspWifi` — it only holds a `&mut` (mutable borrow).

</details>

### Question 17: Why do we clone `provisioned` and `storage` before the `move` closure?

<details>
 <summary>Show answer</summary>

 Because `move` transfers ownership of captured variables into the closure. If we moved the originals, we couldn't use them afterwards (in the wait loop). We clone the `Arc`s (which is cheap — just an atomic counter increment) and move the clones into the closure. The originals stay alive for the main loop.

</details>

### Question 18: If the HTTP server runs in another thread, how does the main loop know provisioning finished?

<details>
 <summary>Show answer</summary>

 Via `Arc<Mutex<bool>>`. The HTTP handler sets `*provisioned_clone.lock().unwrap() = true` on completion. The main loop checks `provisioned.lock()` every second. `Arc` allows sharing the bool across threads, `Mutex` synchronizes access. It's the simplest pattern for thread-to-thread communication.

</details>
