### `.gitkeep` vs `__init__.py`

Git no trackea directorios vacíos. La solución son archivos placeholder:

```bash
# Mantener carpetas vacías en git
touch data/raw/.gitkeep data/processed/.gitkeep
touch results/baselines/.gitkeep results/contextual/.gitkeep
touch results/late_chunking/.gitkeep results/finetuned/.gitkeep
touch notebooks/.gitkeep tests/.gitkeep scripts/.gitkeep docs/.gitkeep

# Convertir src/ y subdirectorios en paquetes Python importables
touch src/__init__.py src/chunking/__init__.py src/embeddings/__init__.py
touch src/eval/__init__.py src/utils/__init__.py tests/__init__.py
```

<Callout>
  Distinción clave: `.gitkeep` es **convención** (le dice a git "trackea este directorio aunque esté vacío"). `__init__.py` es **funcional** (Python lo necesita para tratar la carpeta como paquete importable, e.g. `from src.chunking import naive`).
</Callout>

## Instalar dependencias en batches

### Concepto: dependencias transitivas

Cuando instalás un paquete, ese paquete trae automáticamente otros que él necesita. Es como pedir una pizza: viene con queso, el queso viene con leche, la leche viene de la vaca.

```
sentence-transformers (lo que pediste)
   └── transformers (viene de regalo)
          └── torch (viene de regalo del regalo)
                 └── numpy (transitivo del transitivo)
```

¿Por qué entonces listo varios explícitos? Porque **vos fijás la versión** que querés, no la que el otro paquete decida. Ser explícito y controlado gana sobre dejar al azar.

### Batch 1 — Core ML (PyTorch + HuggingFace)

```bash
uv add torch transformers sentence-transformers datasets
```

| Paquete | Para qué |
|---|---|
| `torch` | Backend de tensores y GPU |
| `transformers` | Modelos de HuggingFace (`AutoModel.from_pretrained`) |
| `sentence-transformers` | Wrappers especializados para embeddings |
| `datasets` | Loader de datasets de HuggingFace |

<Callout type="warning">
  Mac vs GPU NVIDIA — En Apple Silicon, torch se instala con MPS (Metal Performance Shaders). Para inferencia y prototipo va fino. Cuando rentes GPU NVIDIA en la nube, necesitás torch con CUDA: es como cambiar de gasolina a diesel, el motor del carro tiene que cambiar.
</Callout>

### Batch 2 — Data + plotting + estadística

```bash
uv add numpy pandas scikit-learn scipy matplotlib seaborn tqdm
```

| Paquete | Para qué (en este proyecto) |
|---|---|
| `numpy` | Arrays — los embeddings son arrays de numpy |
| `pandas` | Cargar las 150 QA pairs de FinanceBench como DataFrame |
| `scikit-learn` | Métricas (NDCG, MRR) y splits |
| `scipy` | Clave: `scipy.stats.bootstrap` da los CIs del README |
| `matplotlib` | Base de todos los plots |
| `seaborn` | Plots bonitos (heatmaps de Recall@k) |
| `tqdm` | Barras de progreso |

### Batch 3 — PDF + APIs de embedders

```bash
uv add pdfplumber openai voyageai cohere anthropic
```

| Paquete | Para qué |
|---|---|
| `pdfplumber` | Parsea 10-K filings (preserva tablas financieras) |
| `openai` | Cliente para `text-embedding-3-large` |
| `voyageai` | Cliente para `voyage-finance-2` |
| `cohere` | Cliente para Rerank v3.5 |
| `anthropic` | Cliente para Claude Haiku → Contextual Retrieval |

<Callout type="warning">
  ¿Por qué `pdfplumber` y NO `pypdf`? Los 10-K filings son documentos brutales con estados financieros multi-columna, footnotes pegadas a tablas, layout denso. `pypdf` es básico — saca texto plano pero **destroza tablas** convirtiéndolas en sopa de números. `pdfplumber` preserva el layout y tiene `extract_tables()` estructurado. Si los embeddings ven tablas mal parseadas, todo el eval se contamina desde la base. La elección del parser no es cosmética: define la baseline de calidad de TODOS los embeddings que se evalúan después. Decisión documentada en `Decision Log #002`.
</Callout>

## Smoke test de hardware

Antes de seguir, queremos confirmar que el stack respira end-to-end: torch + GPU + sentence-transformers + cosine similarity. Si algo se rompe acá, mejor saberlo antes de invertir tiempo en los Stages siguientes.

<details>
<summary>Ver `scripts/verify_hardware.py` completo</summary>

```python
"""Smoke test: torch + sentence-transformers + cosine similarity."""

import time

import torch
from sentence_transformers import SentenceTransformer
from torch.nn.functional import cosine_similarity

# 1. Detectar device
if torch.cuda.is_available():
    device = "cuda"
    print(f"✅ CUDA available ({torch.cuda.get_device_name(0)})")
elif torch.backends.mps.is_available():
    device = "mps"
    print("✅ MPS available (Apple Silicon)")
else:
    device = "cpu"
    print("⚠️  Falling back to CPU")

# 2. Cargar modelo en device
print(f"Loading 'all-MiniLM-L6-v2' on {device}...")
t0 = time.perf_counter()
model = SentenceTransformer("all-MiniLM-L6-v2", device=device)
print(f"   Loaded in {time.perf_counter() - t0:.1f}s")

# 3. Embeddear 3 frases de prueba
sentences = [
    "Apple reported record revenue this quarter.",
    "Microsoft revenue grew significantly in cloud services.",
    "10-K filings disclose material risks to investors.",
]
t0 = time.perf_counter()
embeddings = model.encode(sentences, convert_to_tensor=True)
print(f"   Embedded {len(sentences)} sentences in {(time.perf_counter() - t0) * 1000:.0f}ms")
print(f"   Embedding shape: {tuple(embeddings.shape)}")
print(f"   Embedding device: {embeddings.device}")

# 4. Cosine similarity manual
sim_apple_msft = cosine_similarity(embeddings[0:1], embeddings[1:2]).item()
sim_apple_10k = cosine_similarity(embeddings[0:1], embeddings[2:3]).item()
print()
print(f"   Cosine sim (Apple vs Microsoft revenue): {sim_apple_msft:.3f}")
print(f"   Cosine sim (Apple revenue vs 10-K filing): {sim_apple_10k:.3f}")
print("   ✅ Smoke test passed.")
```

</details>

Correr el script:

```bash
uv run python scripts/verify_hardware.py
```

Output esperado en la primera corrida (Apple Silicon, MPS):

```
✅ MPS available (Apple Silicon)
Loading 'all-MiniLM-L6-v2' on mps...
[descarga del modelo, ~90MB]
   Loaded in ~9s
   Embedded 3 sentences in ~1500ms
   Embedding shape: (3, 384)
   Embedding device: mps:0

   Cosine sim (Apple vs Microsoft revenue): 0.326
   Cosine sim (Apple revenue vs 10-K filing): 0.079
   ✅ Smoke test passed.
```

<Callout>
  ¿Los valores tienen sentido? **Apple revenue vs Microsoft revenue → 0.326** (similar, ambas hablan de revenue de big tech). **Apple revenue vs 10-K filing → 0.079** (más bajo, cosas distintas semánticamente). Si te sale 0.99 entre las tres o valores negativos, algo está roto en el modelo o en el device.
</Callout>

<Callout type="warning">
  Primera corrida lenta — warmup de MPS. La primera vez que llamás `model.encode()` en MPS, la GPU compila los kernels al vuelo (~1500ms). La segunda corrida (modelo cacheado, kernels compilados) baja a ~80ms — un 18× más rápido. Esto es normal, no es bug.
</Callout>

<Callout type="warning">
  Warning de Hugging Face en la primera corrida: `You are sending unauthenticated requests to the HF Hub. Please set a HF_TOKEN to enable higher rate limits and faster downloads.` Es informativo, no error. HuggingFace limita requests anónimas. Lo arreglamos abajo en la sección del token.
</Callout>

¿Dónde se descargó el modelo? En `~/.cache/huggingface/hub/`. HuggingFace usa un cache global compartido — si bajás el mismo modelo en otro proyecto, no se vuelve a descargar. Cache deduplicado con hard-links, no te ensucia el repo (los modelos son enormes).

## Workflow Git: feature branches + PRs + Conventional Commits

A partir de acá, **NUNCA más commit directo en `main`**. Cada cambio va por:

```
main (estable)
  ↑
  └── feature/branch  →  push  →  Pull Request  →  squash-merge  →  main
       └── commit(s)
```

Por qué este workflow:

| Razón | Detalle |
|---|---|
| Historia limpia | Cada PR squash-mergeado = 1 commit en `main` = fácil de leer |
| Review checkpoint | Revisás el diff del PR antes de mergear |
| Reverso simple | Si algo se rompió, `git revert` sobre el merge commit |
| Profesional | Es el workflow estándar en empresas serias |

### Branch naming: `<type>/<short-kebab-description>`

```
feat/initial-scaffold           ← nueva feature
fix/bge-tokenizer-error         ← bug fix
chore/add-vector-store          ← mantenimiento (deps, configs)
docs/improve-readme             ← solo documentación
refactor/eval-pipeline          ← reescritura sin cambiar comportamiento
test/chunking-edge-cases        ← tests
```

### Commit message — Conventional Commits

Formato: `<type>(<scope>): <subject imperativo, max 72 chars>`, opcionalmente seguido de un body explicando el "por qué". Tipos válidos: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`, `build`, `ci`.

### Squash-merge desde la GitHub UI

1. Andá al PR en GitHub.
2. Revisá el diff.
3. Click **Squash and merge** (NO "Create merge commit", NO "Rebase and merge").
4. Click **Confirm squash and merge**.
5. Click **Delete branch**.

<Callout>
  ¿Por qué squash y no merge commit? Squash colapsa todos los commits del PR en uno solo en `main`. Historia lineal, cada commit en `main` = 1 feature/fix completo, revert simple.
</Callout>

### Sincronizar local después del merge

```bash
git checkout main
git pull
git branch -d feat/initial-scaffold
```

## Vector store: qdrant vs chroma

Comparación entre **ChromaDB** y **Qdrant**:

| Criterio | ChromaDB | Qdrant |
|---|---|---|
| Setup | 1 línea, embebido | 1 línea, embedded mode |
| BM25 / hybrid nativo | No (necesita `rank-bm25` aparte) | Sí, built-in |
| Sparse vectors | No | Sí |
| Producción cuando escale | Limitado | Battle-tested |

```bash
git checkout -b chore/add-vector-store
uv add qdrant-client
```

<Callout>
  Decisión: **qdrant**, porque vamos a hacer hybrid retrieval (BM25 + dense + RRF) en el Stage 2 (Comparative) y qdrant lo da nativo. Si elegimos Chroma, gastamos tiempo orquestando BM25 a mano y eso introduce ruido en la comparación dense-vs-hybrid. Decisión documentada en `Decision Log #004`. Quote para entrevistas: "chose qdrant over chroma because hybrid retrieval was a first-class requirement for the eval and qdrant supports BM25 + dense + RRF natively".
</Callout>

### Modos de uso (FYI, no se usan en este Stage)

```python
from qdrant_client import QdrantClient

# Modo 1 — In-memory (para tests)
client = QdrantClient(":memory:")

# Modo 2 — Persistent local (para research/dev)

# Modo 3 — Server (Docker/cloud)
client = QdrantClient(host="localhost", port=6333)
```

Para el proyecto vamos con el **modo 2** (persistent local). No requiere Docker, los datos persisten entre sesiones, ideal para research reproducible.

  ¿Por qué `--dev`? Estas tools NO van a producción — solo durante desarrollo. El flag `--dev` las pone en grupo separado en `pyproject.toml`. Cuando alguien hace deploy, puede excluirlas con `uv sync --no-dev`.
</Callout>

### Configuración base en `pyproject.toml`

Agregar al final de `pyproject.toml` cuatro secciones de configuración:

- **`[tool.ruff]`** — line-length 100, target Python 3.12.
- **`[tool.ruff.lint]`** — selecciona reglas: pycodestyle errors/warnings (`E`, `W`), pyflakes (`F`), isort (`I`), pyupgrade (`UP`), flake8-bugbear (`B`), flake8-simplify (`SIM`). Ignora `E501` (line-too-long) porque el formatter lo maneja.
- **`[tool.mypy]`** — strict mode sobre `src/`, con overrides de `ignore_missing_imports = true` para librerías sin stubs (`sentence_transformers.*`, `qdrant_client.*`, `voyageai.*`, `pdfplumber.*`).
- **`[tool.pytest.ini_options]`** — `testpaths = ["tests"]`, `addopts = "-v --tb=short --strict-markers"`, patrones estándar para `python_files`, `python_classes`, `python_functions`.

## Cuestionario — Valida lo aprendido

Respondé mentalmente o en voz alta antes de abrir cada respuesta.

### Pregunta 1: ¿Por qué `uv` y no `pip` o `poetry`?

<details>
<summary>Ver respuesta</summary>

Porque `uv` está escrito en Rust por Astral (los mismos de `ruff`) y resuelve el grafo de deps 10-100× más rápido que pip/poetry. Trae lockfile automático para reproducibilidad cross-machine. Adoptado por proyectos serios de 2025-2026 (Anthropic, HuggingFace, Allen AI). Decisión #001 en el Decision Log.

</details>

### Pregunta 2: ¿Qué pasa si no incluís `__init__.py` en `src/` y subcarpetas?

<details>
<summary>Ver respuesta</summary>

Python no trata `src/` como paquete importable. Llamadas como `from src.chunking import naive` fallan con `ModuleNotFoundError`. `__init__.py` es **funcional** (el runtime de Python lo necesita); `.gitkeep` es **convención** (le dice a git "preservá esta carpeta aunque esté vacía"). Son dos placeholders distintos con propósitos distintos.

</details>

### Pregunta 3: En el `.gitignore`, ¿qué hace `data/raw/*` seguido de `!data/raw/.gitkeep`?

<details>
<summary>Ver respuesta</summary>

Ignora todos los archivos dentro de `data/raw/` (los PDFs de FinanceBench son enormes, no van al repo) **excepto** `.gitkeep`. Sin la excepción, git no trackea la carpeta y al clonar el repo en otra máquina la carpeta no existe — los scripts que la asumen revientan.

</details>

### Pregunta 4: ¿Por qué `pdfplumber` y no `pypdf` para los 10-K filings?

<details>
<summary>Ver respuesta</summary>

Los 10-K son documentos densos con tablas financieras multi-columna y footnotes pegadas a tablas. `pypdf` saca texto plano pero rompe el layout, convirtiendo tablas en sopa de números. Si los embeddings ven tablas mal parseadas, todo el eval downstream se contamina desde la base. `pdfplumber` preserva layout y tiene `extract_tables()` estructurado. La elección del parser no es cosmética — define la baseline de calidad de TODOS los embeddings que se evalúan después. Decisión #002 en el Decision Log.

</details>

### Pregunta 5: La primera corrida de `scripts/verify_hardware.py` tarda ~1500ms en `model.encode()`. ¿Bug o feature?

<details>
<summary>Ver respuesta</summary>

Feature. La primera vez que corrés `model.encode()` en MPS, la GPU compila los kernels al vuelo. La segunda corrida (modelo cacheado, kernels compilados) baja a ~80ms — un 18× más rápido. Es el warmup esperado de Metal Performance Shaders. Si la lentitud persiste en corridas subsiguientes, ahí sí hay algo raro.

</details>

### Pregunta 6: ¿Por qué el smoke test usa `all-MiniLM-L6-v2` y no uno de los embedders del estudio (BGE-M3, OpenAI, etc.)?

<details>
<summary>Ver respuesta</summary>

Porque `all-MiniLM-L6-v2` pesa solo ~90MB y tiene 384 dimensiones — descarga rápida, encoding rápido, ideal para validar que el stack respira **end-to-end** sin invertir tiempo ni dinero. Los embedders del estudio (BGE-M3, `text-embedding-3-large`, etc.) tienen 1024-3072 dimensiones y pesos de cientos de MB; rentan API o piden GPU. El smoke test elige el modelo más liviano que ejercite el path completo (load → encode → compute similarity).

</details>

### Pregunta 7: ¿Por qué `qdrant` y no `chromadb` como vector store?

<details>
<summary>Ver respuesta</summary>

Porque hybrid retrieval (BM25 + dense + RRF) es first-class requirement de la eval suite. Qdrant lo soporta nativo en embedded mode persistent. Chroma no — habría que sumar `rank-bm25` aparte y orquestar la fusión a mano, introduciendo ruido en la comparación dense-vs-hybrid del Stage 2 (Comparative). Decisión #004 en el Decision Log.

</details>

### Pregunta 8: ¿Qué diferencia hay entre `git tag v0.1.0` y `git tag -a v0.1.0 -m "..."`?

<details>
<summary>Ver respuesta</summary>

`git tag v0.1.0` (lightweight) es solo un nombre que apunta a un commit. `git tag -a v0.1.0 -m "..."` (anotado) lleva autor, fecha y mensaje completo, y es lo que GitHub muestra en la página de Releases con changelog. Para releases públicos siempre `-a` — los lightweight son marcadores internos de uso personal.

</details>

### Pregunta 9: ¿Por qué `--public` y `--description` al crear el repo, en vez de quedarte con los defaults?

<details>
<summary>Ver respuesta</summary>

`--public` porque el repo es para portfolio — un repo privado no se puede compartir con un hiring manager. `--description` porque la primera frase que ve cualquiera (en GitHub search, en cards de redes sociales, en JSON-LD) es esa descripción. Una buena: "Rigorous evaluation of contextual retrieval techniques on FinanceBench". Una mala: vacía.

</details>

### Pregunta 10: ¿Por qué `uv add --dev pytest ruff mypy ipykernel` y no `uv add` directamente?

<details>
<summary>Ver respuesta</summary>

Porque pytest, ruff, mypy e ipykernel **no van a producción** — solo son útiles en desarrollo. El flag `--dev` las pone en un grupo separado en `pyproject.toml`. Cuando alguien hace deploy puede excluirlas con `uv sync --no-dev`, ahorrando tamaño de imagen y tiempo de install. La separación es disciplina básica de packaging.

</details>

### Pregunta 11: ¿Qué señala un repo con releases por Stage (`v0.1.0`, `v0.2.0`, ..., `v1.0.0`) vs uno con tags por commit?

<details>
<summary>Ver respuesta</summary>

Un repo con 5 releases con changelogs sustantivos señala madurez y disciplina técnica — cada Stage cierra un bloque defendible. Un repo con 50 tags triviales señala ruido y falta de criterio. La convención `v0.x.y` durante desarrollo y `v1.0.0` al cerrar comunica el estado del proyecto: "pre-release / API en evolución" vs "publicado y estable". Patrón estándar de uv, ruff, sentence-transformers. Decisión #005 en el Decision Log.

</details>

### Pregunta 12: Si en el smoke test la cosine similarity entre las dos frases de revenue te da `-0.05`, ¿qué interpretás?

<details>
<summary>Ver respuesta</summary>

Algo está roto. Cosine similarity ∈ [-1, 1], pero embeddings de `all-MiniLM-L6-v2` sobre frases de inglés natural casi nunca dan negativos — el modelo está entrenado con loss contrastiva sobre pares positivos, así que valores negativos sugieren un bug (frases vacías, encoding fallido, device equivocado, modelo corrupto). Comparado con el output esperado (`Apple revenue` vs `Microsoft revenue` ≈ 0.326), un -0.05 dice "no estoy comparando lo que creo que comparo". Vale debuggear.

</details>

### Pregunta 13: En el workflow Git, ¿qué problema resuelve "squash and merge" sobre "create a merge commit"?

<details>
<summary>Ver respuesta</summary>

"Create a merge commit" preserva todos los commits del PR en `main`, lo que ensucia la historia con WIPs, fix-typo, fix-fix-typo. "Squash and merge" colapsa todos los commits del PR en uno solo (con el título del PR), dejando `main` lineal y leíble. Cada commit en `main` corresponde a 1 feature/fix completo. El revert es trivial (`git revert <merge-sha>`). La historia se vuelve documentación.

</details>

### Pregunta 14: ¿Qué pasa si tu `.gitignore` lista `.env` pero no `.env.local` ni `.env.production`?

<details>
<summary>Ver respuesta</summary>

`.env.local` y `.env.production` no quedan ignorados, y al hacer `git add .` los podés commitear con secrets adentro — y los secrets en una historia git pública son un incidente de seguridad costoso (rotar todas las keys + force-push de la historia, con pérdida potencial). El patrón correcto es `.env.*` con `!.env.example` para excepcionar el template. Defensa en profundidad: además del `.gitignore`, no escribir secrets en archivos que no estén bajo `.env`.

</details>

### Pregunta 15: ¿Qué viene después del Stage 0? ¿Y por qué ese orden?

<details>
<summary>Ver respuesta</summary>

Stage 1 — Baselines (`v0.2.0`). Empieza con PyTorch Fundamentals: tensors, autograd conceptual y `AutoModel.from_pretrained` para cargar un modelo de HuggingFace y generar embeddings manualmente. Después aterriza el FinanceBench loader, el eval pipeline (Recall@k, MRR, NDCG, MAP con bootstrap CIs) y los dos primeros baselines (OpenAI `text-embedding-3-large` + BGE-M3 self-hosted). El orden importa: sin saber medir rigurosamente con bootstrap CIs, las baselines no son defendibles. Por eso evaluación va antes que comparación multi-modelo (Stage 2) y antes que fine-tuning (Stage 3). El próximo paso del curso es **PyTorch Fundamentals — Tensors + AutoModel**.

</details>
