## Initialize the project with `uv`

```bash
mkdir ~/financebench-rag-eval
cd ~/financebench-rag-eval
uv init --python 3.12
```

Expected output:

```
Initialized project `financebench-rag-eval` at `/Users/your-user/financebench-rag-eval`
```

`uv init` creates five files: `pyproject.toml` (project manifest, the equivalent of Node's `package.json`), `.python-version` (pin of the Python version), `main.py` (placeholder), `README.md` (empty) and a `.gitignore` with basic rules. It also initializes a local git repo.

`main.py` won't be used — delete it:

```bash
rm main.py
```

<Callout>
  Location convention: projects live under `~/`, `~/code/` or `~/projects/`. A consistent scheme saves future headaches with scripts and absolute paths.
</Callout>

### `.gitkeep` vs `__init__.py`

Git does not track empty directories. The fix is placeholder files:

```bash
# Keep empty folders in git
touch data/raw/.gitkeep data/processed/.gitkeep
touch results/baselines/.gitkeep results/contextual/.gitkeep
touch results/late_chunking/.gitkeep results/finetuned/.gitkeep
touch notebooks/.gitkeep tests/.gitkeep scripts/.gitkeep docs/.gitkeep

# Make src/ and subdirs importable Python packages
touch src/__init__.py src/chunking/__init__.py src/embeddings/__init__.py
touch src/eval/__init__.py src/utils/__init__.py tests/__init__.py
```

<Callout>
  Key distinction: `.gitkeep` is a **convention** (it tells git "track this folder even if empty"). `__init__.py` is **functional** (Python needs it to treat the folder as an importable package, e.g. `from src.chunking import naive`).
</Callout>

## Install dependencies in batches

### Concept: transitive dependencies

When you install a package, that package automatically pulls in others it depends on. Think of ordering a pizza: it comes with cheese, the cheese comes with milk, the milk comes from a cow.

```
sentence-transformers (what you asked for)
   └── transformers (free with the order)
          └── torch (free with the free order)
                 └── numpy (transitive of the transitive)
```

Why list several explicit ones then? Because **you pin the version** you want, instead of letting another package pick it. Explicit and controlled wins over leaving it to chance.

### Batch 1 — Core ML (PyTorch + HuggingFace)

```bash
uv add torch transformers sentence-transformers datasets
```

| Package | Used for |
|---|---|
| `torch` | Tensor and GPU backend |
| `transformers` | HuggingFace models (`AutoModel.from_pretrained`) |
| `sentence-transformers` | Specialized wrappers for embeddings |
| `datasets` | HuggingFace dataset loader |

<Callout type="warning">
  Mac vs NVIDIA GPU — On Apple Silicon, torch installs with MPS (Metal Performance Shaders). For inference and prototyping it's fine. When you rent an NVIDIA GPU in the cloud, you need torch with CUDA: it's like switching from gas to diesel — the engine has to change.
</Callout>

### Batch 2 — Data + plotting + statistics

```bash
uv add numpy pandas scikit-learn scipy matplotlib seaborn tqdm
```

| Package | Used for (in this project) |
|---|---|
| `numpy` | Arrays — embeddings are numpy arrays |
| `pandas` | Loads FinanceBench's 150 QA pairs as a DataFrame |
| `scikit-learn` | Metrics (NDCG, MRR) and splits |
| `scipy` | Key: `scipy.stats.bootstrap` produces the README's CIs |
| `matplotlib` | Base for every plot |
| `seaborn` | Polished plots (Recall@k heatmaps) |
| `tqdm` | Progress bars |

### Batch 3 — PDF + embedder APIs

```bash
uv add pdfplumber openai voyageai cohere anthropic
```

| Package | Used for |
|---|---|
| `pdfplumber` | Parses 10-K filings (preserves financial tables) |
| `openai` | Client for `text-embedding-3-large` |
| `voyageai` | Client for `voyage-finance-2` |
| `cohere` | Client for Rerank v3.5 |
| `anthropic` | Client for Claude Haiku → Contextual Retrieval |

<Callout type="warning">
  Why `pdfplumber` and NOT `pypdf`? 10-K filings are brutal documents with multi-column financial statements, footnotes glued to tables, dense layouts. `pypdf` is basic — it extracts plain text but **destroys tables**, turning them into number soup. `pdfplumber` preserves layout and exposes a structured `extract_tables()`. If embeddings see broken tables, the entire eval is contaminated from the base. The parser choice is not cosmetic: it sets the quality baseline for every embedding evaluated downstream. Documented in `Decision Log #002`.
</Callout>

## Hardware smoke test

Before going further, we want to confirm the stack breathes end-to-end: torch + GPU + sentence-transformers + cosine similarity. If something breaks here, better know it now than after sinking time into the next Stages.

<details>
<summary>See full `scripts/verify_hardware.py`</summary>

```python
"""Smoke test: torch + sentence-transformers + cosine similarity."""

import time

import torch
from sentence_transformers import SentenceTransformer
from torch.nn.functional import cosine_similarity

# 1. Detect device
if torch.cuda.is_available():
    device = "cuda"
    print(f"✅ CUDA available ({torch.cuda.get_device_name(0)})")
elif torch.backends.mps.is_available():
    device = "mps"
    print("✅ MPS available (Apple Silicon)")
else:
    device = "cpu"
    print("⚠️  Falling back to CPU")

# 2. Load model on device
print(f"Loading 'all-MiniLM-L6-v2' on {device}...")
t0 = time.perf_counter()
model = SentenceTransformer("all-MiniLM-L6-v2", device=device)
print(f"   Loaded in {time.perf_counter() - t0:.1f}s")

# 3. Embed 3 test sentences
sentences = [
    "Apple reported record revenue this quarter.",
    "Microsoft revenue grew significantly in cloud services.",
    "10-K filings disclose material risks to investors.",
]
t0 = time.perf_counter()
embeddings = model.encode(sentences, convert_to_tensor=True)
print(f"   Embedded {len(sentences)} sentences in {(time.perf_counter() - t0) * 1000:.0f}ms")
print(f"   Embedding shape: {tuple(embeddings.shape)}")
print(f"   Embedding device: {embeddings.device}")

# 4. Manual cosine similarity
sim_apple_msft = cosine_similarity(embeddings[0:1], embeddings[1:2]).item()
sim_apple_10k = cosine_similarity(embeddings[0:1], embeddings[2:3]).item()
print()
print(f"   Cosine sim (Apple vs Microsoft revenue): {sim_apple_msft:.3f}")
print(f"   Cosine sim (Apple revenue vs 10-K filing): {sim_apple_10k:.3f}")
print("   ✅ Smoke test passed.")
```

</details>

Run it:

```bash
uv run python scripts/verify_hardware.py
```

Expected output on the first run (Apple Silicon, MPS):

```
✅ MPS available (Apple Silicon)
Loading 'all-MiniLM-L6-v2' on mps...
[model download, ~90MB]
   Loaded in ~9s
   Embedded 3 sentences in ~1500ms
   Embedding shape: (3, 384)
   Embedding device: mps:0

   Cosine sim (Apple vs Microsoft revenue): 0.326
   Cosine sim (Apple revenue vs 10-K filing): 0.079
   ✅ Smoke test passed.
```

<Callout>
  Do those values make sense? **Apple revenue vs Microsoft revenue → 0.326** (similar — both about big-tech revenue). **Apple revenue vs 10-K filing → 0.079** (lower — semantically different things). If you get 0.99 across the three or negative values, something is broken in the model or the device.
</Callout>

<Callout type="warning">
  First run is slow — MPS warmup. The first time you call `model.encode()` on MPS, the GPU compiles the kernels on the fly (~1500ms). The second run (cached model, compiled kernels) drops to ~80ms — an 18× speedup. This is normal, not a bug.
</Callout>

<Callout type="warning">
  HuggingFace warning on the first run: `You are sending unauthenticated requests to the HF Hub. Please set a HF_TOKEN to enable higher rate limits and faster downloads.` Informational, not an error. HuggingFace rate-limits anonymous requests. We fix it below in the token section.
</Callout>

Where did the model download to? `~/.cache/huggingface/hub/`. HuggingFace uses a global shared cache — if you download the same model in another project, it isn't re-downloaded. Cache deduplicated with hard-links, doesn't pollute the repo (models are huge).

## Git workflow: feature branches + PRs + Conventional Commits

From here on, **NEVER commit directly to `main` again**. Every change goes through:

```
main (stable)
  ↑
  └── feature/branch  →  push  →  Pull Request  →  squash-merge  →  main
       └── commit(s)
```

Why this workflow:

| Reason | Detail |
|---|---|
| Clean history | Each squash-merged PR = 1 commit on `main` = easy to read |
| Review checkpoint | You review the PR diff before merging |
| Easy rollback | If something broke, `git revert` the merge commit |
| Professional | This is the standard workflow at serious companies |

### Branch naming: `<type>/<short-kebab-description>`

```
feat/initial-scaffold           ← new feature
fix/bge-tokenizer-error         ← bug fix
chore/add-vector-store          ← maintenance (deps, configs)
docs/improve-readme             ← documentation only
refactor/eval-pipeline          ← rewrite without changing behavior
test/chunking-edge-cases        ← tests
```

### Commit message — Conventional Commits

Format: `<type>(<scope>): <imperative subject, max 72 chars>`, optionally followed by a body explaining the "why". Valid types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`, `build`, `ci`.

### Squash-merge from the GitHub UI

1. Go to the PR on GitHub.
2. Review the diff.
3. Click **Squash and merge** (NOT "Create merge commit", NOT "Rebase and merge").
4. Click **Confirm squash and merge**.
5. Click **Delete branch**.

<Callout>
  Why squash and not merge commit? Squash collapses every commit in the PR into a single one on `main`. Linear history, each commit on `main` = 1 complete feature/fix, simple revert.
</Callout>

### Sync local after the merge

```bash
git checkout main
git pull
git branch -d feat/initial-scaffold
```

## Vector store: qdrant vs chroma

Side-by-side of **ChromaDB** and **Qdrant**:

| Criterion | ChromaDB | Qdrant |
|---|---|---|
| Setup | 1 line, embedded | 1 line, embedded mode |
| Native BM25 / hybrid | No (needs `rank-bm25` extra) | Yes, built-in |
| Sparse vectors | No | Yes |
| Production at scale | Limited | Battle-tested |

```bash
git checkout -b chore/add-vector-store
uv add qdrant-client
```

<Callout>
  Decision: **qdrant**, because we are going to run hybrid retrieval (BM25 + dense + RRF) in Stage 2 (Comparative) and qdrant supports it natively. Picking Chroma would mean spending research time orchestrating BM25 by hand, which adds noise to the dense-vs-hybrid comparison. Documented in `Decision Log #004`. Quote for interviews: "chose qdrant over chroma because hybrid retrieval was a first-class requirement for the eval and qdrant supports BM25 + dense + RRF natively".
</Callout>

### Base configuration in `pyproject.toml`

Append four configuration sections at the end of `pyproject.toml`:

- **`[tool.ruff]`** — line-length 100, target Python 3.12.
- **`[tool.ruff.lint]`** — selects rules: pycodestyle errors/warnings (`E`, `W`), pyflakes (`F`), isort (`I`), pyupgrade (`UP`), flake8-bugbear (`B`), flake8-simplify (`SIM`). Ignore `E501` (line-too-long) — the formatter handles it.
- **`[tool.mypy]`** — strict mode on `src/`, with overrides setting `ignore_missing_imports = true` for libraries without stubs (`sentence_transformers.*`, `qdrant_client.*`, `voyageai.*`, `pdfplumber.*`).
- **`[tool.pytest.ini_options]`** — `testpaths = ["tests"]`, `addopts = "-v --tb=short --strict-markers"`, standard patterns for `python_files`, `python_classes`, `python_functions`.

## Closing Stage 0 — Tag `v0.1.0` + Release

To mark the close of Stage 0 as a reproducible milestone, merging PRs is not enough — we need a **tag** and a **GitHub Release**. This gives the repo an anchor any hiring manager (or you, later) can `git checkout v0.1.0` and reproduce the exact state of.

### Tag strategy by Stage

`v0.x.y` tags (semver) mark **technical Stages**, not small increments:

| Tag | Stage | Status |
|---|---|---|
| `v0.1.0` | Stage 0 — Foundation | Released |
| `v0.2.0` | Stage 1 — Baselines | Pending |
| `v0.3.0` | Stage 2 — Comparative | Pending |
| `v0.4.0` | Stage 3 — Custom | Pending |
| `v1.0.0` | Stage 4 — Publication | Pending |

<Callout>
  Why `v0.x.y` and not `v1.x.y` from day one? In semver, `v0.x.y` means "pre-release / API may change". `v1.0.0` is reserved for when the project is published and stable. This is the convention of serious open-source projects (uv, ruff, sentence-transformers all use it). Documented in `Decision Log #005`.
</Callout>

### Sync local with `main`

```bash
git checkout main
git pull
git branch -d chore/add-dev-tools
```

### Create an annotated tag

```bash
git tag -a v0.1.0 -m "v0.1.0 — Stage 0: project setup complete

- Project scaffold (paper-quality structure)
- 16 production deps + 4 dev deps via uv
- Hardware smoke test (MPS verified)
- HuggingFace token configured
- qdrant-client for vector store
- ruff + mypy + pytest + ipykernel with strict configs
- 2 environment smoke tests passing
- Git workflow: feature branches + Conventional Commits + squash-merge"
```

Annotated vs lightweight tag:

| | Lightweight (`git tag v0.1.0`) | Annotated (`git tag -a v0.1.0 -m "..."`) |
|---|---|---|
| Metadata | Name only | Author, date, full message |
| Used for | Internal marker | Public releases |
| Best practice | No | Yes |

Always use `-a` for releases.

### Push the tag

```bash
git push origin v0.1.0
```

<Callout type="warning">
  Gotcha — pre-commit hook blocking the tag push. If you have a personal hook that blocks pushes to `main` to enforce the PR workflow, make sure it explicitly excludes tags. `git push origin v0.1.0` pushes a **tag**, not the branch — the hook should detect that and let it through (`git push --tags` or a regex `git push <remote> v[0-9]...`).
</Callout>

### Create the GitHub Release

```bash
gh release create v0.1.0 --title "v0.1.0 — Stage 0: Project Setup Complete" \
  --notes "..."
```

The `--notes` field carries the changelog in Markdown. Recommended structure:

- **What's in this release** — bullets of the highlights.
- **Production dependencies (16)** — list.
- **Dev dependencies (4)** — list.
- **Tooling** — `uv`, `ruff`, `mypy`, `pytest`, `ipykernel`.
- **Verified on** — Apple Silicon (MPS), Python 3.12.
- **Test plan** — commands to reproduce.
- **What's next** — link to the next Stage / `v0.2.0`.

<Callout>
  Tag vs Release: a **tag** is a pointer to a specific commit. A **release** is a GitHub page wrapping a tag with a changelog, downloadable assets, and a slot in the repo's "Releases" section. For portfolio, **always create a release**, not just a tag.
</Callout>

Final release URL:

```
https://github.com/FMFigueroa/financebench-rag-eval/releases/tag/v0.1.0
```

That is the first thing a hiring manager sees when they open the repo.

## Quiz — Validate what you learned

Answer each one mentally before opening the response.

### Question 1: Why `uv` and not `pip` or `poetry`?

<details>
<summary>See answer</summary>

Because `uv` is written in Rust by Astral (the team behind `ruff`) and resolves the dep graph 10-100× faster than pip/poetry. It ships with an automatic lockfile for cross-machine reproducibility. Adopted by serious 2025-2026 projects (Anthropic, HuggingFace, Allen AI). Decision #001 in the Decision Log.

</details>

### Question 2: What happens if you don't include `__init__.py` in `src/` and its subfolders?

<details>
<summary>See answer</summary>

Python doesn't treat `src/` as an importable package. Calls like `from src.chunking import naive` fail with `ModuleNotFoundError`. `__init__.py` is **functional** (the Python runtime needs it); `.gitkeep` is **convention** (it tells git "preserve this folder even if empty"). Two different placeholders for two different jobs.

</details>

### Question 3: In the `.gitignore`, what does `data/raw/*` followed by `!data/raw/.gitkeep` do?

<details>
<summary>See answer</summary>

It ignores every file inside `data/raw/` (FinanceBench PDFs are huge, they don't go to the repo) **except** `.gitkeep`. Without the exception git wouldn't track the folder, and a clone on another machine would not have the folder — every script that assumes it explodes.

</details>

### Question 4: Why `pdfplumber` and not `pypdf` for the 10-K filings?

<details>
<summary>See answer</summary>

10-K filings are dense documents with multi-column financial tables and footnotes glued to tables. `pypdf` extracts plain text but breaks layout, turning tables into number soup. If embeddings see broken tables, every downstream eval is contaminated from the base. `pdfplumber` preserves layout and exposes a structured `extract_tables()`. Parser choice is not cosmetic — it sets the quality baseline for every embedding evaluated downstream. Decision #002 in the Decision Log.

</details>

### Question 5: The first run of `scripts/verify_hardware.py` takes ~1500ms in `model.encode()`. Bug or feature?

<details>
<summary>See answer</summary>

Feature. The first time you call `model.encode()` on MPS, the GPU compiles the kernels on the fly. The second run (cached model, compiled kernels) drops to ~80ms — an 18× speedup. It's the expected Metal Performance Shaders warmup. If the slowness persists across subsequent runs, that's a real problem.

</details>

### Question 6: Why does the smoke test use `all-MiniLM-L6-v2` instead of one of the study's embedders (BGE-M3, OpenAI, etc.)?

<details>
<summary>See answer</summary>

Because `all-MiniLM-L6-v2` weighs only ~90MB and has 384 dimensions — fast download, fast encoding, ideal for validating that the stack breathes **end-to-end** without sinking time or money. The study's embedders (BGE-M3, `text-embedding-3-large`, etc.) have 1024-3072 dimensions and weights in the hundreds of MB; they need API budget or GPU. The smoke test picks the lightest model that exercises the full path (load → encode → compute similarity).

</details>

### Question 7: Why `qdrant` and not `chromadb` as vector store?

<details>
<summary>See answer</summary>

Because hybrid retrieval (BM25 + dense + RRF) is a first-class requirement of the eval suite. Qdrant supports it natively in persistent embedded mode. Chroma does not — you'd add `rank-bm25` separately and orchestrate the fusion by hand, adding noise to the dense-vs-hybrid comparison in Stage 2 (Comparative). Decision #004 in the Decision Log.

</details>

### Question 8: What's the difference between `git tag v0.1.0` and `git tag -a v0.1.0 -m "..."`?

<details>
<summary>See answer</summary>

`git tag v0.1.0` (lightweight) is just a name pointing at a commit. `git tag -a v0.1.0 -m "..."` (annotated) carries author, date and a full message, and is what GitHub displays on the Releases page with changelog. For public releases always `-a` — lightweight tags are personal-use markers.

</details>

### Question 9: Why `--public` and `--description` when creating the repo, instead of taking the defaults?

<details>
<summary>See answer</summary>

`--public` because the repo is portfolio — a private repo can't be shared with a hiring manager. `--description` because the first sentence anyone sees (in GitHub search, social card embeds, JSON-LD) is that description. A good one: "Rigorous evaluation of contextual retrieval techniques on FinanceBench". A bad one: empty.

</details>

### Question 10: Why `uv add --dev pytest ruff mypy ipykernel` instead of `uv add` directly?

<details>
<summary>See answer</summary>

Because pytest, ruff, mypy and ipykernel **don't ship to production** — they only earn their keep during development. `--dev` puts them in a separate group in `pyproject.toml`. On deploy, you can exclude them with `uv sync --no-dev`, saving image size and install time. The split is basic packaging hygiene.

</details>

### Question 11: What does a repo with releases per Stage (`v0.1.0`, `v0.2.0`, ..., `v1.0.0`) signal vs a repo with one tag per commit?

<details>
<summary>See answer</summary>

A repo with 5 releases and substantive changelogs signals technical maturity and discipline — every Stage closes a defensible block. A repo with 50 trivial tags signals noise and lack of judgment. The convention `v0.x.y` during development and `v1.0.0` on close communicates the project's state: "pre-release / API in motion" vs "published and stable". Standard pattern from uv, ruff, sentence-transformers. Decision #005 in the Decision Log.

</details>

### Question 12: If on the smoke test the cosine similarity between the two revenue sentences gives `-0.05`, what do you read into that?

<details>
<summary>See answer</summary>

Something is broken. Cosine similarity ∈ [-1, 1], but `all-MiniLM-L6-v2` embeddings on natural English sentences almost never go negative — the model is trained with contrastive loss on positive pairs, so a negative value points at a bug (empty inputs, failed encoding, wrong device, corrupted model). Versus the expected output (`Apple revenue` vs `Microsoft revenue` ≈ 0.326), a -0.05 is screaming "I am not comparing what you think I am comparing". Worth debugging.

</details>

### Question 13: In the Git workflow, what problem does "squash and merge" solve over "create a merge commit"?

<details>
<summary>See answer</summary>

"Create a merge commit" preserves every commit in the PR on `main`, dirtying the history with WIPs, fix-typo, fix-fix-typo. "Squash and merge" collapses every commit in the PR into a single one (with the PR title), keeping `main` linear and readable. Each commit on `main` corresponds to 1 complete feature/fix. Reverts are trivial (`git revert <merge-sha>`). The history becomes documentation.

</details>

### Question 14: What happens if your `.gitignore` lists `.env` but not `.env.local` or `.env.production`?

<details>
<summary>See answer</summary>

`.env.local` and `.env.production` are not ignored, and on `git add .` you can commit them with secrets inside — and secrets in a public git history are a costly security incident (rotate every key + force-push the history, with potential losses). The right pattern is `.env.*` plus `!.env.example` to spare the template. Defense in depth: beyond `.gitignore`, never write secrets in files that are not `.env`.

</details>

### Question 15: What comes after Stage 0? And why that order?

<details>
<summary>See answer</summary>

Stage 1 — Baselines (`v0.2.0`). Starts with PyTorch Fundamentals: tensors, conceptual autograd, and `AutoModel.from_pretrained` to load a HuggingFace model and generate embeddings manually. Then the FinanceBench loader, the eval pipeline (Recall@k, MRR, NDCG, MAP with bootstrap CIs), and the first two baselines (OpenAI `text-embedding-3-large` + BGE-M3 self-hosted). The order matters: without rigorous measurement using bootstrap CIs, baselines are not defensible. That's why evaluation comes before multi-model comparison (Stage 2) and before fine-tuning (Stage 3). The next step in the course is **PyTorch Fundamentals — Tensors + AutoModel**.

</details>
