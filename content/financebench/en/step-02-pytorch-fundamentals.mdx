## 1. What is a tensor?

A **tensor** is the **atomic unit of PyTorch**: an N-dimensional array of numbers with three superpowers a `list` or a `numpy.array` does not have.

### Anatomy and dimensions

| Name | Dims | Example |
|---|---|---|
| Scalar | 0-D | `5.7` (the temperature) |
| Vector | 1-D | `[1.2, 3.4, 5.6]` (a small embedding) |
| Matrix | 2-D | `[[1,2],[3,4]]` (a table) |
| 3-D tensor | 3-D | `(batch, tokens, embed_dim)` — the typical BERT output |
| N-D tensor | N-D | 4-D images `(batch, channels, height, width)`, 5-D video, etc. |

In this project you will see a lot of:

- `(batch_size, seq_len, hidden_dim)` → BERT-base output, e.g. `(1, 12, 768)` for 12 tokens
- `(num_chunks, embed_dim)` → embedding matrix for your corpus, e.g. `(10000, 1024)` with BGE-M3

<Callout>
  Shape convention: the first number is **rows**, the second is **columns**. `torch.randn(2, 3)` = 2 rows × 3 columns. Read the arguments as **"height × width"**, like a `1080 × 1920` photo.
</Callout>

### The 3 superpowers (vs `numpy.array`)

**Superpower 1 — Vectorization**. An operation over the entire tensor runs in optimized C/CUDA, not in a Python `for`. Multiplying two 1M-element tensors takes ~1 ms; a Python loop takes ~5 seconds. 5000× difference. The technical reason: contiguous memory + uniform types enable **SIMD** (Single Instruction Multiple Data). `numpy` also vectorizes on CPU — the difference with PyTorch comes from the next superpower.

**Superpower 2 — GPU / MPS native**. You move the tensor to the GPU with `.to(device)` and every operation after that runs there. Numpy CANNOT do this. This is why modern ML = tensors, not numpy.

```python
device = "mps" if torch.backends.mps.is_available() else "cpu"
x = x.to(device)   # reassigning is CRITICAL — .to() returns a new tensor
```

<Callout type="warning">
  Key gotcha: `.to(device)` does **NOT mutate in-place**. If you only write `x.to(device)` (without reassigning), the result is lost and `x` stays on CPU. General rule: every PyTorch method **without a trailing underscore** returns a new tensor; methods **with a trailing underscore** (`.add_()`, `.zero_()`) do mutate in-place.
</Callout>

**Superpower 3 — Autograd**. PyTorch keeps an **automatic computational graph**: every operation gets recorded and you can later compute gradients with `.backward()`. This is what makes neural network training possible.

```python
x = torch.tensor([2.0], requires_grad=True)
y = x ** 2 + 3 * x
y.backward()
print(x.grad)   # tensor([7.])  → which is 2x+3 with x=2
```

PyTorch built the graph, computed the derivative symbolically and evaluated it — by itself. Without autograd you would have to compute derivatives by hand for every model (impossible in networks with millions of parameters).

<Callout>
  Why this is the most important: vectorization and GPU exist in other libraries (TensorFlow, JAX). The **quality and ergonomics of PyTorch's autograd** are what made it the dominant framework after 2018. Numpy has no autograd and never will — its scope is numerical computation, not automatic differentiation.
</Callout>

### 5 ways to create tensors

```python
import torch

t1 = torch.tensor([1.0, 2.0, 3.0])    # from explicit values
t2 = torch.zeros(2, 3)                 # zeros with a specific shape
t3 = torch.randn(2, 3)                 # random normal (weight init)

import numpy as np
arr = np.array([1, 2, 3])
t4 = torch.from_numpy(arr)             # from numpy

t5 = torch.arange(0, 10)               # sequential (handy for debugging)
```

### The 3 attributes you'll read all the time

```python
t = torch.randn(2, 3, 4)
t.shape   # torch.Size([2, 3, 4])  → dimensional structure
t.dtype   # torch.float32           → data type (default float32)
t.device  # device(type='cpu')      → CPU or MPS or CUDA
```

<Callout>
  `shape` is the first thing you ever look at. 80% of ML bugs are shape mismatches. When something does not work, the first question is: *what shape does this tensor have?*. That's why you see `print(x.shape)` in every ML notebook.
</Callout>

### Gotchas that bite the first time

1. The default `dtype` is `float32`, not `float64` like numpy. In ML 99% of the time you want float32 (faster on GPU, enough precision).
2. `.numpy()` requires CPU. If the tensor is on MPS: `t.cpu().numpy()` (not direct `t.numpy()`).
3. `torch.tensor()` vs `torch.Tensor()` — use the **lowercase**. Uppercase is legacy and has confusing behavior with dtypes.
4. `torch.from_numpy(np_arr)` inherits the numpy dtype (default `float64`). Fix: `.float()` or `arr.astype(np.float32)`.

## Math detour — Cosine similarity vs distance

This section is the conceptual heart of the project's eval. Worth documenting properly because the two metrics get mixed up.

### The 3 concepts that get tangled

| Concept | What it is | Range |
|---|---|---|
| **Model weights** | Trainable internal numbers (transformer matrices) | approx `[-3, 3]` |
| **Embedding** (the vector) | List of N numbers representing a sentence/chunk | each component approx `[-1, 1]` |
| **Cosine similarity** | Result of comparing 2 embeddings | `[-1, +1]` |

### The actual formula

Cosine similarity is NOT an NLP invention — it's pure geometry, derived from the dot product:

```
                    A · B
cosine_sim(A, B) = ─────────
                   ‖A‖ · ‖B‖
```

Comes from `A · B = ‖A‖ · ‖B‖ · cos(θ)`. Solving: `cos(θ) = (A · B) / (‖A‖ · ‖B‖)`. **Cosine similarity LITERALLY IS cos(θ) between two vectors.**

| Angle θ | cos(θ) | Interpretation |
|---|---|---|
| 0° | +1 | same direction (identical) |
| 90° | 0 | perpendicular (no relation) |
| 180° | −1 | opposite directions |

### Cosine vs euclidean

**Cosine does NOT measure distance between points** — it measures **angular alignment**. Example: `A = [1, 1]` and `B = [10, 10]` are **far in euclidean distance** (~12.7) but **identical in cosine** (1.0) — they point in the same direction.

That's why we use cosine in NLP: the embedding **magnitude** can vary for spurious reasons (longer sentences, model normalization), but the **semantic direction** captures the meaning.

### Similarity vs distance — the sort flip that bites

```
COSINE SIMILARITY              COSINE DISTANCE  (= 1 − similarity)
←———•———•———→                  •———•———→
-1   0  +1                      0   1   2
opposite ⊥ same ✨             same ✨  ⊥  opposite
```

Modern vector DBs expose both depending on the API:

- **Qdrant** — API exposes `Distance.COSINE`, internally ranks by **similarity** (larger to smaller).
- **FAISS** — uses **distance** or **inner product** depending on the index type.
- **scikit-learn** — `cosine_similarity()` → similarity. `pairwise_distances(metric='cosine')` → distance.
- **OpenAI / Voyage / Cohere SDKs** — return raw embeddings, you compute the metric.

<Callout type="warning">
  Eval gotcha: when measuring Recall@k in Stage 1, you'll sort results from larger to smaller (similarity) or smaller to larger (distance) depending on what the library returns. **If you sort the wrong way, your Recall@5 will be ~0%** and you'll think your model is trash when it's actually just a sort flip. Everyone hits this the first time.
</Callout>

### Why in practice you almost always see values in `[0, 1]`

Modern embedders (OpenAI, Voyage, BGE-M3, sentence-transformers) produce vectors that mostly live in the **positive hemisphere of space**. It is not a mathematical guarantee — it is an **emergent property of contrastive training**: the model learned to put related sentences close together (cos near 1) and unrelated ones perpendicular (cos ≈ 0). They almost never learn genuinely "opposite" representations.

<Callout>
  Interview bonus: if asked *"why cosine and not euclidean?"*, you can answer with the mathematical WHY (angular alignment vs distance, magnitude-insensitive) **and** mention that with modern contrastive embedders the effective range is `[0,1]`. That's a senior answer, not junior.
</Callout>

## Inference vs training — why `torch.no_grad()` matters

~85-90% of the project is **pure inference**. Worth pinning the difference.

| Mode | What you do | When |
|---|---|---|
| **Training** | Adjust the model weights (forward + loss + backward + step) | Stage 3 (BGE-M3 fine-tuning) |
| **Inference** | Pass input → get output. **Frozen weights, no learning** | Stages 1, 2 and 4 (anything eval) |

<Callout>
  Analogy: training is a chef learning (you teach them recipes, they try, fail, correct). Inference is the trained chef taking orders at the restaurant: they execute, they don't learn. **Your customer doesn't want the chef learning on their plate.**
</Callout>

### Why `torch.no_grad()` matters in inference

When PyTorch sees operations, **by default it builds the computational graph** (because it doesn't know whether you'll call `.backward()` later). That graph:

- Takes extra memory
- Slows the math down a bit
- Is **completely unnecessary if you only want the output**

```python
# Without no_grad — PyTorch builds an unnecessary graph
embeddings = model.encode(texts)

# With no_grad — only forward pass
with torch.no_grad():
    embeddings = model.encode(texts)
# Memory: ~50% | Speed: ~110-130%
```

<Callout type="warning">
  Project gotcha: when you process the 150 FinanceBench documents and embed them with BGE-M3, without `no_grad()` you can OOM on MPS because the graph grows with every batch. With `no_grad()` the problem disappears. Difference between "the script finishes in 3 minutes" and "the script crashes halfway".
</Callout>

**Modern bonus (PyTorch 2.0+)**: `torch.inference_mode()` is even more aggressive than `no_grad`. Strictly better when you are 100% sure you don't need gradients — which is our case in Stages 1 and 2.

```python
with torch.inference_mode():
    embeddings = model.encode(texts)
```

## 2. Tensor operations

The four operations you'll do every day with embeddings.

### 2.1 Indexing and slicing

Practically the same as numpy. If you know numpy, you already know 90% of this.

```python
t = torch.tensor([[10,11,12,13],[20,21,22,23],[30,31,32,33]])

t[0, 0]            # tensor(10)
t[:, 1]            # tensor([11, 21, 31])     ← entire column 1
t[0:2, 1:3]        # sub-matrix rows 0-1, cols 1-2
t[t > 20]          # boolean mask → FLAT tensor
```

**Ellipsis `...`** — handy for tensors with many dims. In NLP with `(batch, seq_len, hidden_dim)`, getting the last token: `output[..., -1, :]`.

**Gotchas**:

1. Indexing with an integer **REDUCES** the dim, slicing does not: `t[0].shape == (4,)` vs `t[0:1].shape == (1, 4)`. To preserve the dim: `t[i:i+1]` or `t[i].unsqueeze(0)`.
2. `tensor[i]` returns a **view**, not a copy (same as numpy). Mutating the view mutates the original. For an independent copy: `.clone()`.
3. Boolean masks return a **flat (1-D)** tensor regardless of original shape.

### 2.2 Reshaping

**Reshape = the same number of values, organized differently.** The product of the dimensions must match the total element count. If not, `RuntimeError`.

```python
t = torch.arange(0, 12)
t.view(3, 4)             # 0..11 organized as 3 rows × 4 cols
t.view(3, -1).shape      # torch.Size([3, 4])  ← -1 inferred by PyTorch
t.view(-1).shape         # torch.Size([12])    ← total flatten
```

**You can only use ONE `-1`** per reshape. PyTorch infers ONE dim, not two.

**`.view()` vs `.reshape()`**:

- `.view()` — requires a **contiguous** tensor in memory. Otherwise, error. Faster (only changes metadata).
- `.reshape()` — more flexible. If contiguous, behaves like view; otherwise, copies data automatically. **Always works.**

`.transpose()` and `.permute()` break contiguous. **Pragmatic rule**: when in doubt, use `.reshape()`.

**`.squeeze()` and `.unsqueeze()`** — add/remove size-1 dimensions.

```python
emb = torch.randn(1024)             # (1024,)
emb.unsqueeze(0).shape              # (1, 1024)  ← batch of 1

batch = torch.randn(1, 1024)
batch.squeeze(0).shape              # (1024,)
```

<Callout type="warning">
  `.squeeze()` WITHOUT arguments removes ALL size-1 dims. If your tensor is `(1, 32, 768)` and you call `.squeeze()`, you lose the batch dim. Fix: always specify the dim → `.squeeze(0)`.
</Callout>

### 2.3 Broadcasting + batched dot product

**Broadcasting** = operating between tensors of different shapes without loops, as if they had the same shape. The rules:

1. Dimensions align **from the right**.
2. Two dimensions are compatible if they are **equal**, one is **1**, or one **does not exist** (assumed to be 1).
3. The size-1 dim "stretches" virtually to the size of the other. PyTorch **does NOT copy data** — it's just a view.

**The real project case**: 1 query against 150 chunks in one vectorized operation.

```python
query_emb = torch.randn(1024)              # (1024,)
corpus    = torch.randn(150, 1024)         # (150, 1024)
similarities = (corpus * query_emb).sum(dim=1)   # (150,)
```

Step A — `corpus * query_emb` with broadcasting: PyTorch aligns from the right, "stretches" `query_emb` virtually to `(150, 1024)` and multiplies element-wise. Each row of the result is a chunk multiplied by the query.

Step B — `.sum(dim=1)` reduces dim 1, leaving 150 dot products. **Without loops.**

**Mental rule**: `sum(dim=N)` **eliminates** dim N. The rest of the shape stays intact.

### Vector norm — generalized Pythagoras

```
‖A‖ = √(a₁² + a₂² + ... + aₙ²)
```

3-4-5 triangle: `A = [3, 4]` → `‖A‖ = √25 = 5`. The formula is **identical** in 1024 dimensions — you just add more squared terms.

```python
torch.tensor([3.0, 4.0]).norm()    # tensor(5.)
corpus.norm(dim=1).shape           # torch.Size([150]) — norm per chunk
```

`.norm()` defaults to **L2 norm** (the euclidean one). In ML 99% of the time you want L2.

### Normalized vectors → cosine = dot product

A vector is **normalized** when `‖A‖ = 1`. If both vectors are normalized:

```
cos(θ) = (A · B) / (1 · 1) = A · B
```

**The dot product IS the cosine similarity directly.** Modern embedders (BGE-M3, OpenAI, Voyage) return normalized vectors by default. In real project code you'll see:

```python
similarities = corpus @ query_emb   # dot product = cosine sim
```

Without dividing by norms. The `@` operator does the same thing as `(corpus * query).sum(dim=1)` but more readably — it's the PyTorch matmul.

### 2.4 Concat and stack

- **`torch.cat()`** — **concatenates** tensors along an **EXISTING** dimension.
- **`torch.stack()`** — **stacks** tensors creating a **NEW** dimension.

<Callout>
  Quick analogy: if you have 3 A4 sheets...
  - `cat` = tape them together to make one long sheet (sums along the length)
  - `stack` = pile them like a sandwich (creates a new "height" dimension)
</Callout>

```python
A = torch.tensor([1, 2, 3])     # shape (3,)
B = torch.tensor([4, 5, 6])     # shape (3,)

torch.cat([A, B])               # tensor([1,2,3,4,5,6])   shape (6,)
torch.stack([A, B])             # tensor([[1,2,3],[4,5,6]]) shape (2, 3)
```

**Shape rules**:

- For `cat`: the dims you do NOT concatenate must be **identical**. The dim you concatenate can differ.
- For `stack`: ALL tensors must have **exactly the same shape**.

**Project case**: when processing the 150 chunks in batches:

```python
batch_1 = model.encode(chunks[0:32])     # (32, 1024)
batch_2 = model.encode(chunks[32:64])    # (32, 1024)
all_embeddings = torch.cat([batch_1, batch_2, ...], dim=0)
# (150, 1024)
```

If you want to create a new dim, **always `stack`. Never `cat`.**

## 3. Conceptual autograd

**Autograd = the system that automatically computes derivatives over any sequence of operations.** It is what makes neural networks able to learn. **We do NOT use it in Stages 1-2** (inference only), we do in Stage 3 (fine-tuning).

### What is a gradient?

**The derivative of a function at a point** — it tells you **in which direction and with what magnitude** the function changes if you tweak its inputs.

If you stand at some point on a hill:

- Sign of the gradient → direction of the slope (positive goes up, negative goes down)
- Magnitude → how steep the slope is

```
f(x) = x²,  derivative: df/dx = 2x

at x = 3  → gradient = 6   (positive slope, strong)
at x = 0  → gradient = 0   (minimum, no slope)
at x = -2 → gradient = -4  (negative slope)
```

**Training a model = finding the parameters that minimize an error function (loss).** The gradient tells you "in which direction to tweak my parameters to walk down the loss hill". This is **gradient descent** and is the basis of ALL training.

### The computational graph

PyTorch automatically builds a graph as you write operations, and can later compute gradients by walking it backward (hence `.backward()`).

```python
x = torch.tensor([2.0], requires_grad=True)
y = x ** 2 + 3 * x
y.backward()
print(x.grad)
# tensor([7.])
```

Manual check: `y = x² + 3x`, derivative `dy/dx = 2x + 3`. At `x = 2`: `2*2 + 3 = 7`.

### The 3-piece API

| Piece | What it does |
|---|---|
| `requires_grad=True` | Tells PyTorch "track this tensor in the graph" |
| `.backward()` | Computes gradients walking the graph (this IS backpropagation) |
| `.grad` | Attribute that stores the computed gradient |

### Full hierarchy

```
GRADIENT DESCENT (the optimization algorithm)
   ↓ needs
GRADIENT VECTOR (vector of partial derivatives)
   ↓ computed via
BACKPROPAGATION (chain rule walking the graph)
   ↓ implemented in PyTorch as
AUTOGRAD (torch.autograd module)
   ↓ accessed via
.backward() and .grad
```

In code:

```python
loss.backward()       # this IS backpropagation
parameter.grad        # this is the gradient vector
optimizer.step()      # this IS gradient descent (adjusts parameters)
```

### Scale — why autograd is revolutionary

| Model | Parameters | Gradient vector size |
|---|---|---|
| BERT-base | 124M | 124 million partial derivatives |
| BGE-M3 | 568M | 568M partial derivatives |
| GPT-3 | 175B | 175 billion partial derivatives |

Every parameter has its OWN partial derivative. Backpropagation computes them ALL in a single pass. **Without backprop, doing this by hand would be impossible.**

<Callout>
  In this project you do NOT write `.backward()` by hand. In Stage 3, `sentence-transformers` does it under the hood during fine-tuning. Knowing **conceptually** what's happening inside is what lets you **debug when things go wrong** (loss not going down, exploding gradients, etc.).
</Callout>

## 4. HuggingFace `AutoModel` — anatomy of the cache

A HuggingFace model is **NOT a file** — it's a **Git-style organized directory**.

```
~/.cache/huggingface/hub/models--sentence-transformers--all-MiniLM-L6-v2/
│
├── blobs/                        ← raw content, named by SHA256 hash
│   ├── 53aa51172d142c89d9...     87 MB    ← model.safetensors (the weights)
│   ├── cb202bfe2e3c986450...    456 KB    ← tokenizer.json
│   ├── fb140275c155a9c7c5...    228 KB    ← vocab.txt
│   └── ...                                ← JSON configs, README
│
├── snapshots/                    ← human-readable views via symlinks
│   └── c9745ed1d9f207416b.../   ← SHA of one HF repo commit
│       ├── config.json            → ../../blobs/72b987...
│       ├── model.safetensors      → ../../blobs/53aa51...
│       ├── tokenizer.json         → ../../blobs/cb2028...
│       └── ...
│
└── refs/                         ← Git "branch"-style pointers
    └── main                       ← contains the SHA of the live snapshot
```

**The 3 folders**:

- `blobs/` — stores content **once**, deduplicated by SHA256 hash. If you download another version that shares an identical file, it is NOT duplicated: it points at the same blob.
- `snapshots/<commit-sha>/` — the "human" view: files with readable names (`config.json`, `model.safetensors`) that are **symlinks** to blobs.
- `refs/main` — says which snapshot is "live". If the model repo is updated on HuggingFace, a new snapshot is downloaded and `refs/main` points at the new SHA.

<Callout>
  This is **literally Git mechanics applied to models** — content-addressed storage + commits + branches. That's why HuggingFace talks about "model commits", "revisions", etc.
</Callout>

## 5. Tokenization — text → IDs

Tokenization is **converting raw text into numeric IDs** the model understands. `AutoTokenizer` uses 4 files from the tokenizer directory.

| File | What it's for |
|---|---|
| `tokenizer_config.json` | Defines the **class** (`BertTokenizer`) and options (`do_lower_case=true`, `max_length=512`). `AutoTokenizer` reads it first to know what to instantiate. |
| `vocab.txt` | The **flat vocabulary**: 30,522 lines, one token per line, **line number = ID**. |
| `tokenizer.json` | Everything serialized into a single JSON (rules + vocab + config). Used by "fast tokenizers" in Rust — faster than parsing `vocab.txt`. |
| `special_tokens_map.json` | Mapping of special tokens (`cls_token=[CLS]`, `sep_token=[SEP]`, etc.). |

### Anatomy of `vocab.txt`

```
Line 0:    [PAD]      ← padding
Line 100:  [UNK]      ← unknown
Line 101:  [CLS]      ← classification (sequence start)
Line 102:  [SEP]      ← separator (sequence end)
Line 103:  [MASK]     ← masked language modeling
```

That's why `[101, 7592, 2088, 102]` decomposes as `[CLS]`, two real words, `[SEP]`.

### Full flow

```
"Hello world"
   ↓ do_lower_case   (from tokenizer_config.json)
"hello world"
   ↓ WordPiece       (rules in tokenizer.json)
["hello", "world"]
   ↓ lookup IDs      (in vocab.txt)
[7592, 2088]
   ↓ wrap [CLS]/[SEP] (from special_tokens_map.json)
[101, 7592, 2088, 102]   ← input ready for the model
```

### How the model USES the IDs

`vocab.txt` only translates text → IDs. **But the model does not use `vocab.txt`** — it has its own **internal embedding matrix** of shape `(vocab_size, hidden_dim)` = `(30522, 384)` in MiniLM, **learned during training**.

```
embedding_matrix  →  shape (30522, 384)

ID = 7592
    ↓
matrix[7592]      ← extracts row 7592 (a vector of 384 numbers)
    ↓
vector = [0.12, -0.45, 0.78, ..., -0.03]   (384 values)
```

**No computation, no search** — the ID IS the row number. O(1) operation, instant (same as `list[index]` in Python). Each row was learned during training: tokens with similar meaning (`"dog"`, `"canine"`) ended up with nearby vectors in 384-D space.

### From the static vector to the contextualized one — the attention layers

The vector that comes out of `embedding_matrix` is **static**: the word `"bank"` has ONE single vector in the matrix, regardless of whether the sentence is:

- `"the park bank"` (a seat)
- `"Santander bank"` (a financial institution)

The **6 attention layers** of MiniLM are what refine that vector using context from neighboring words. Each layer = one pass of the puzzle:

- **Layer 1**: each token looks at its close neighbors and adjusts its vector with that local context
- **Layers 2-5**: context expands — vectors capture more global relationships
- **Layer 6**: each token has a vector reflecting its meaning across the WHOLE sentence

<Callout>
  Size trade-off: 6 layers is small compared with BERT-base (12) or BGE-M3 (24). MiniLM is ~3-4× faster but loses expressive power. For fine retrieval over 10-K filings, BGE-M3 will outperform it.
</Callout>

### Genealogy: why the vocab has `[CLS] [SEP] [PAD]`

- **2017** — Google publishes *"Attention Is All You Need"*: the **Transformer** is born.
- **2018, October** — Google ships **BERT**: bidirectional, understands text, beats 11 NLP benchmarks.
- **2019-2020** — Cambrian explosion: RoBERTa, DistilBERT, ALBERT, ELECTRA, DeBERTa.
- **2021-2022** — paths diverge: encoders (BERT family) → embeddings; decoders (GPT family) → generation. **sentence-transformers** are born (Reimers & Gurevych): BERT fine-tuned to produce embeddings comparable with cosine similarity.
- **2023-2026** — **BGE-M3, E5, Voyage, Cohere**: direct descendants of BERT with mandatory L2 normalization + hard negative mining.

That's why when `vocab.txt` shows up with `[CLS] [SEP] [PAD] [UNK] [MASK]`, you are looking at **BERT's genealogy from 2018, intact**.

## 6. Manual embeddings — forward pass + mean pooling

Generating embeddings = passing IDs through the model (**forward pass**) + reducing tokens to 1 vector per sentence (**pooling**). Two atomic steps.

### Forward pass

```python
with torch.no_grad():
    outputs = model(**batch)

last_hidden = outputs.last_hidden_state
# shape: (batch, seq_len, hidden_dim) = (3, 11, 384)
```

- `**batch` unpacks the tokenizer dict: `input_ids` + `attention_mask` + `token_type_ids` are passed as kwargs
- `torch.no_grad()` disables autograd graph construction → less memory, faster
- `outputs.last_hidden_state`: tensor with one contextualized vector per token

### `model.eval()` vs `torch.no_grad()` — NOT the same thing

| Flag | What it does | When |
|---|---|---|
| `model.eval()` | Disables **dropout** and switches **batch-norm** to running stats | Once on model load |
| `torch.no_grad()` | Disables construction of the **autograd graph** | Wraps every inference call |

You need **both** for clean inference. They are not interchangeable — they live on different planes.

### Mean pooling — line by line

```python
def mean_pooling(token_embeddings, attention_mask):
    mask = attention_mask.unsqueeze(-1).expand(token_embeddings.size()).float()
    summed = (token_embeddings * mask).sum(dim=1)
    counts = mask.sum(dim=1).clamp(min=1e-9)
    return summed / counts
```

1. **Prepare the mask**: `unsqueeze(-1)` → `(3,11,1)`, `expand` → `(3,11,384)`, `.float()` converts 0/1 to 0.0/1.0
2. **Sum real tokens**: element-wise product (padding stays 0) + `.sum(dim=1)` → `(3, 384)`
3. **Count real tokens**: `mask.sum(dim=1)` with `.clamp(min=1e-9)` to avoid division by zero
4. **Average**: `summed / counts` → 1 average vector per sentence

Output: `embeddings.shape = (3, 384)` — 3 sentences × 1 vector of 384 dim each.

### Why mean pooling and not [CLS] pooling?

Depends on the model: **MiniLM** and **BGE-small** were trained with **mean pooling** as the official recipe. **BGE-M3** uses **[CLS] pooling** for its dense vector. **Always read the model card.**

<Callout>
  Pooling metaphor: mean pooling = **street poll** (every token votes equally, average it). [CLS] pooling = **director's editorial** (the `[CLS]` token listens to everyone during the attention layers and emits a single summary).
</Callout>

## 7. Manual cosine similarity

Comparing embeddings = **matmul after normalizing**. When vectors are unit-length (norm = 1), the dot product **is** cosine similarity directly.

### Step 1 — L2 normalization

```python
import torch.nn.functional as F

embeddings = F.normalize(embeddings, p=2, dim=1)
print([round(n, 4) for n in embeddings.norm(dim=1).tolist()])
# → [1.0, 1.0, 1.0]
```

- `p=2`: uses L2 norm (the euclidean one, generalized Pythagoras)
- `dim=1`: normalizes row by row (each sentence has its own unit vector)
- Each vector ends with magnitude 1 → **unit vector**

### Step 2 — Similarity matrix with ONE matmul

```python
sim_matrix = embeddings @ embeddings.T
print(sim_matrix.cpu().numpy().round(3))
```

- `embeddings @ embeddings.T`: `(3, 384) @ (384, 3) = (3, 3)`
- Cell `[i, j]` = dot product between sentence `i` and sentence `j`
- Since the vectors are **normalized**, dot product = cosine similarity

### Expected notebook output

The 3 smoke-test sentences: two about Apple, one about a cat.

```
                   sent_0   sent_1   sent_2
sent_0 (Apple-1)   [1.000   ~0.85    ~0.30 ]
sent_1 (Apple-2)   [~0.85   1.000    ~0.30 ]
sent_2 (cat)       [~0.30   ~0.30    1.000 ]
```

- **Diagonal = 1.0** (each sentence with itself)
- `[0,1]` and `[1,0]` high (both about Apple)
- `[0,2]`, `[1,2]` low (the cat one is unrelated)

If the matrix matches that pattern, the embedder is working as it should.

### Connection with the project

**This is exactly the retrieval operation at scale.**

- **Vector DBs** (Qdrant, FAISS) store normalized embeddings precisely so nearest-neighbor by dot product is equivalent to nearest-neighbor by cosine similarity.
- In the upcoming Stage 1 lessons we'll use the same pattern (`encode → normalize → matmul`) to evaluate 5 embedders against FinanceBench's 150 QA pairs.
- The similarity matrix is the foundation of the **Recall@k**, **MRR**, **NDCG** computations.

## Quiz — Validate what you learned

Answer each one mentally before opening the response.

### Question 1: What does `.to(device)` do on a tensor and what is the classic gotcha?

<details>
<summary>See answer</summary>

`.to(device)` moves the tensor to CPU, MPS or CUDA. **The gotcha**: it does NOT mutate in-place — it returns a new tensor. If you write `x.to(device)` without reassigning, the result is lost and `x` stays where it was. You have to write `x = x.to(device)`. General rule: any PyTorch method without a trailing underscore returns a new tensor; methods with a trailing underscore (`.add_()`, `.zero_()`) do mutate.

</details>

### Question 2: If a tensor has shape `(2, 3, 4)`, what shape does `.sum(dim=1)` produce?

<details>
<summary>See answer</summary>

`(2, 4)`. The rule: `sum(dim=N)` **eliminates** dim `N`, the rest of the shape stays intact. It's the same operation used to collapse the token dim in embeddings: `(batch, seq_len, hidden_dim).sum(dim=1)` → `(batch, hidden_dim)`. Mean pooling is exactly that, divided by the number of valid tokens per the attention mask.

</details>

### Question 3: Why does `.numpy()` fail when the tensor is on MPS?

<details>
<summary>See answer</summary>

Because `numpy` only operates on CPU — it does not understand GPU/MPS memory. You have to move the tensor first: `t.cpu().numpy()`. It's the logical order of the flow: copy to RAM (CPU) and then interpret the bytes as an `np.ndarray`. PyTorch raises `TypeError: can't convert mps tensor to numpy` so you don't accidentally move data without realizing it.

</details>

### Question 4: In cosine similarity, why do `[1, 1]` and `[10, 10]` give 1.0 even though they are far in euclidean distance?

<details>
<summary>See answer</summary>

Because cosine **does NOT measure distance between points** — it measures **angular alignment**. The two vectors point in exactly the same direction (45° relative to the axes), so the angle between them is 0° and `cos(0°) = 1`. The magnitude (one is √2 long, the other 10√2) "cancels" when dividing by the norms. That's why we use cosine in NLP: embedding magnitude can vary for spurious reasons, but the semantic direction is what captures the meaning.

</details>

### Question 5: If your embedder returns `Distance.COSINE` via API and you sort **smallest to largest**, what happens to your Recall@5?

<details>
<summary>See answer</summary>

You get ~0% — you'd be picking the 5 **least** similar chunks as your top-5 instead of the most similar. It's the classic **sort flip**: similarity ranks largest-to-smallest, distance ranks smallest-to-largest. Everyone hits this the first time. Before declaring "my model is trash", verify the sort order. Qdrant internally ranks by similarity even though its API says "Distance"; OpenAI/Voyage return raw embeddings and you compute the metric.

</details>

### Question 6: What is the difference between `model.eval()` and `torch.no_grad()` and why do you need both for inference?

<details>
<summary>See answer</summary>

`model.eval()` changes the behavior of **layers with modes** — turns off dropout and makes batch-norm use running stats instead of the batch's. `torch.no_grad()` disables construction of the **autograd computational graph** — operations still happen, but PyTorch does not store references for an eventual `.backward()`. Different planes: without `eval()` the model applies dropout and returns embeddings with random noise (non-reproducible); without `no_grad()` the graph grows with every batch and you can OOM on MPS while processing 150 chunks. In PyTorch 2.0+ `torch.inference_mode()` is even more aggressive than `no_grad`.

</details>

### Question 7: Why is the first iteration of an MPS block ~10× slower than the next ones?

<details>
<summary>See answer</summary>

It's the **Metal Performance Shaders warmup**: the first time the GPU sees a new operation, it **compiles it on the fly** to Metal kernels. That compilation takes time. Subsequent calls reuse the cached kernels and drop to the expected speed. Same concept as CUDA kernels on NVIDIA and compute shaders in Vulkan. That's why in MPS inference benchmarks you discard the first run and average from the second onward.

</details>

### Question 8: In the HuggingFace cache, what is stored in `blobs/`, what in `snapshots/<sha>/`, and what in `refs/main`?

<details>
<summary>See answer</summary>

`blobs/` stores each file's content **once**, deduplicated by SHA256 hash — it's the actual storage. `snapshots/<sha>/` is the "human view": files with readable names (`config.json`, `model.safetensors`) that are **symlinks** pointing to the blobs of a specific HF repo commit. `refs/main` is the "branch"-style pointer — it contains the SHA of the live snapshot. It's **literally Git applied to models**: content-addressed storage + commits + branches. That's why HuggingFace talks about "model commits", "revisions", etc.

</details>

### Question 9: In `vocab.txt`, what's the relationship between line number and token ID? What are `[CLS]`, `[SEP]`, `[PAD]`?

<details>
<summary>See answer</summary>

**The line number (0-indexed) IS the ID** of the token. Line 7592 is the token with ID 7592 — direct lookup, O(1). `[CLS]` (line 101) marks the start of a sequence (from "classification"); `[SEP]` (line 102) separates sequences or marks the end; `[PAD]` (line 0) fills when batching so all sequences have the same length. They are **special tokens** inherited from BERT (2018) — that's why they appear in MiniLM, BGE-M3, E5, Voyage, etc.

</details>

### Question 10: If embeddings are already normalized (norm = 1), what simplification does the cosine similarity formula allow?

<details>
<summary>See answer</summary>

`cos(θ) = (A · B) / (‖A‖ · ‖B‖) = (A · B) / (1 · 1) = A · B`. **The dot product IS cosine similarity directly.** That's why modern embedders (BGE-M3, OpenAI, Voyage) return normalized vectors by default — they let the eval computation be a single `embeddings @ embeddings.T` without dividing by norms. Vector DBs (Qdrant, FAISS) store normalized embeddings for the same reason: nearest-neighbor by dot product = nearest-neighbor by cosine similarity.

</details>

### Question 11: Why mean pooling and not `[CLS]` pooling in MiniLM and BGE-small?

<details>
<summary>See answer</summary>

Because **they were trained with mean pooling as the official recipe**. The choice is not free — it depends on how the model was trained. MiniLM and BGE-small fine-tune with mean pooling on the output, so the sentence's representative vector is the average of all tokens' contextualized vectors (filtered by attention mask to ignore padding). BGE-M3, on the other hand, uses `[CLS]` pooling for its dense vector. **Rule**: always read the model card before choosing the pooling. If you flip it, retrieval degrades even though the code "runs without error".

</details>

### Question 12: Why does the diagonal of the matrix `embeddings @ embeddings.T` give exactly 1.0?

<details>
<summary>See answer</summary>

Because cell `[i, i]` is the dot product of sentence `i` with itself: `A · A = ‖A‖²`. Since the embeddings are **L2-normalized** (`‖A‖ = 1`), then `A · A = 1² = 1`. It's a perfect sanity check: if the diagonal does NOT give 1.0, it's a sign you forgot to normalize or the model didn't return unit vectors. Off-diagonal cells `[i, j]` with `i ≠ j` are the actual cosine similarities between different sentences — those are what drive the retrieval ranking.

</details>
