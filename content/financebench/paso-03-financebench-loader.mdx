## Sub-bloque 1 — Descarga del dataset desde HuggingFace

### El one-liner que hace todo

```python
from datasets import load_dataset

ds = load_dataset("PatronusAI/financebench")
```

Eso es todo. Un import, una llamada, y el dataset queda en disco listo para queryarlo. Pero ese one-liner esconde tres capas de maquinaria que vale entender — porque las vas a usar en **todos** los datasets del proyecto.

### Qué retorna `load_dataset` exactamente

`load_dataset()` retorna un `DatasetDict` — un container dict-like donde:

- **Las keys** son los *splits* del dataset (`"train"`, `"validation"`, `"test"`)
- **Los values** son objetos `Dataset` (la clase propia de HuggingFace, optimizada con Apache Arrow por debajo)

```python
print(type(ds).__name__)        # DatasetDict
print(list(ds.keys()))           # ['train']

train = ds["train"]
print(type(train).__name__)      # Dataset
print(train.num_rows)            # 150
print(train.column_names)        # ['financebench_id', 'company', ...]
```

### Qué es un "split"

Los splits son un concepto fundamental en ML:

| Split | Propósito | Analogía |
|---|---|---|
| `train` | Entrenar el modelo | Estudiar para un examen |
| `validation` | Tunear hyperparámetros sin tocar `test` | Pruebas tipo simulacro |
| `test` | Medir performance final **una sola vez** | El examen real |

**¿Por qué dividir?** Si entrenas y mides en los mismos datos, el modelo memoriza en vez de aprender. La métrica se ve excelente en papel pero colapsa con datos nuevos — eso es **overfitting**. Separar los splits fuerza al modelo a generalizar.

### Por qué FinanceBench solo trae `train`

FinanceBench es **evaluation-only**. Patronus no lo diseñó para entrenar nada — lo diseñaron para que midas qué tan bueno es tu sistema RAG **ya construido**. Por convención de la librería `datasets`, cualquier split único se llama `train` aunque su uso real sea evaluación. **Es un quirk del naming, nada más.**

<Callout type="warning">
  Decisión fijada para todo el proyecto: **NO se hacen splits sobre FinanceBench**. Las 150 QA pairs quedan intactas como eval set sagrado. En el Stage 3 (fine-tuning de BGE-M3) usamos **otra fuente** de 10-K y de ahí construimos splits propios. Así garantizamos zero contamination — el modelo nunca ve las 150 QAs durante el entrenamiento, y la métrica final es honesta.
</Callout>

> 💡 **Regla de oro en ML**: el test set jamás se toca durante el desarrollo. Ni para inspección, ni para "una mirada nada más", ni para sanity-check. En el momento que tuneas algo basado en el comportamiento del test set, las métricas reportadas se vuelven marketing, no ciencia.

### Apache Arrow vs Parquet — qué hay realmente debajo

`load_dataset()` lee **archivos Parquet** desde disco y los expone como **Apache Arrow** en memoria. Son dos cosas distintas:

| | Parquet | Apache Arrow |
|---|---|---|
| Dónde vive | En **disco** | En **memoria** |
| Para qué | Almacenamiento eficiente (columnar, comprimido) | Procesamiento rápido sin copias |
| Quién lo lee | `load_dataset` lo lee del cache | El objeto `Dataset` lo expone en RAM |

La librería los combina: descarga Parquet del Hub → lo materializa como Arrow en memoria → tú accedes con la API de `Dataset`.

#### Por qué Parquet (vs CSV/JSON)

- **Columnar**: si solo necesitas la columna `question`, lee solo ese bloque del disco (no las 15 columnas).
- **Comprimido**: archivos 5-10× más chicos.
- **Tipado**: `int`, `float`, `list`, `struct` se guardan nativos (sin parseo manual de strings).

#### Por qué Arrow (vs listas Python o pandas)

Tres razones técnicas que justifican todo el diseño:

**1. Memory-mapped I/O — datasets más grandes que tu RAM.**
Arrow puede mapear archivos a memoria sin cargarlos. Si tienes un dataset de 50 GB pero solo 16 GB de RAM, Arrow lee bajo demanda — solo las filas/columnas que tocas viven en RAM en cada momento. Listas Python o pandas DataFrames cargan **todo** de una. Para datasets como Common Crawl o LAION-5B, pandas simplemente no es opción.

> **Analogía**: pandas es leer un libro entero antes de buscar una palabra. Arrow es ir directo al índice y abrir solo la página que necesitas.

**2. Layout columnar — operaciones tipo ML son rapidísimas.**
ML data prep es fundamentalmente "tocar columnas enteras": filtrar por `doc_period == 2018`, tokenizar todos los `question`, calcular longitudes de cada `evidence`. Arrow guarda cada columna contigua en memoria, así el CPU vectoriza con instrucciones SIMD (operando sobre 8-16 valores en paralelo). **10-100× más rápido** que iterar listas Python.

**3. Zero-copy + estándar cross-language.**
Cuando haces `train.map(fn)` o pasas el dataset a PyTorch / TensorFlow, el dataset **no se copia** — se comparte el buffer Arrow. Listas Python tienen que serializar/deserializar (formato binario propio de Python) en cada paso, lo que se vuelve cuello de botella en un training loop. Arrow además es estándar abierto y agnóstico de lenguaje: Python, Rust, Java, JavaScript y R leen el mismo archivo sin conversión.

### La abstracción se puede romper — y necesitas saber cómo

HuggingFace te da todo eso "gratis": escribes una línea y recibes los beneficios transparentemente. **Pero en el momento que rompes la abstracción sin querer, lo devuelves todo.**

```python
# MAL — convierte el Dataset a una lista Python de dicts
data = list(train)  # ahora todo está en RAM como objetos Python

# MAL — itera y acumula resultados en una lista
chunks = []
for row in train:
    chunks.append(process(row))  # backed por listas, no Arrow

# BIEN — usa la API nativa, mantiene Arrow por debajo
processed = train.map(process, batched=True)
```

**Por qué entender esto importa aunque HuggingFace lo abstraiga**:

- Cuando veas un OOM con un dataset grande → vas a saber que escribiste `list(...)` o `to_pandas()` en algún lado y rompiste el lazy loading.
- Cuando entrenes con PyTorch → vas a usar `train.with_format("torch")` para zero-copy a tensors (10× más rápido que convertir a mano).
- Cuando optimices → vas a alcanzar `.filter()` y `.map()` en vez de comprehensions.

> **TL;DR**: HuggingFace te da el poder gratis, pero **solo si juegas con su API**. Salirte de ella = volver a velocidad pandas.

### Cache local — la primera descarga es lenta, el resto es instantáneo

La primera llamada a `load_dataset("PatronusAI/financebench")` descarga los archivos Parquet en `~/.cache/huggingface/datasets/`. Las llamadas siguientes son **instantáneas** — leen del cache local. Lo verificas re-corriendo la celda: la barra de progreso desaparece.

Si necesitas liberar espacio o forzar una re-descarga, el cache es solo una carpeta que puedes borrar:

```bash
rm -rf ~/.cache/huggingface/datasets/patronus_ai___financebench
```

### Gotchas que vale flaggear de entrada

1. **`HF_TOKEN` no es necesario para este dataset** — es público. Si el token está en tu env, la librería lo usa; si no, irrelevante acá. Los tokens son obligatorios para modelos **gated** (ej: LLaMA), no para datasets públicos.
2. **La primera descarga muestra `Generating train split: 0%|...|`** — son los Parquet materializándose en Arrow. La segunda corrida ya no la muestra (cache hit).
3. **`"train"` es convención**, no intención. FinanceBench es eval-only sin importar el nombre del split.
4. **No itres el dataset entero a RAM si fuera grande** — usa `train[i]` o slicing. Arrow lo maneja lazy y la memoria queda plana.

### 2.0 Schema del dataset — los 15 campos, por rol

Antes de mirar los campos, vale situar el dataset en la **escala de granularidad de texto** que usamos a lo largo del proyecto:

| Unidad | Qué es |
|---|---|
| **Documento** | Un archivo entero (ej: un 10-K de 200 páginas) |
| **Página** | Una página individual del documento |
| **Passage** | Uno o varios párrafos relacionados (ej: la sección que explica CapEx) |
| **Token** | Palabra o sub-palabra (la unidad atómica del modelo, ~3-4 chars en promedio) |

El **passage** es el sweet spot del retrieval: chico para que un embedding lo represente bien, suficientemente grande para contener una respuesta completa. Si retrieveamos a nivel de documento (200 páginas), el embedder no puede procesarlos (capean en ~512 tokens) y la búsqueda se vuelve imprecisa. Si retrieveamos a nivel de palabras, perdemos contexto. **FinanceBench se define en la unidad pasaje** — y es por eso que los 15 campos del dataset orbitan alrededor del campo `evidence`.

Un record de FinanceBench tiene **15 campos**. No son igualmente importantes — tres son el corazón de toda evaluation de retrieval, el resto es metadata.

#### Núcleo RAG (los que se evalúan de verdad)

| Campo | Tipo | Qué es |
|---|---|---|
| `question` | string (44-592 chars) | La pregunta financiera |
| `answer` | string (1-609 chars) | La respuesta correcta |
| **`evidence`** | list (1-3 items) | Passages del PDF que contienen la respuesta · ⭐ **GROUND TRUTH** |
| `justification` | string (nullable) | Explicación de cómo se derivó la respuesta |

#### Identificación del documento

`doc_name` (ej: `"3M_2018_10K"`) · `doc_link` (URL al PDF en SEC) · `doc_type` (`10k` / `10q` / `8k` / `Earnings`) · `doc_period` (año fiscal, 2015-2024) · `company` (ej: `"3M"`, `"JPMorgan"`)

#### Metadata / filtros

`financebench_id` (ID único de 21 chars) · `question_type` (3 categorías) · `question_reasoning` (3 tipos de razonamiento) · `domain_question_num` · `dataset_subset_label` (siempre `"OPEN_SOURCE"`) · `gics_sector` (uno de 9)

#### Sample row — la unidad de evaluación

```json
{
  "financebench_id": "financebench_id_03029",
  "company": "3M",
  "doc_name": "3M_2018_10K",
  "question": "What is the FY2018 capital expenditure amount (USD millions) for 3M?",
  "answer": "$1577.00",
  "evidence": [
    {
      "evidence_text": "Consolidated Statement of Cash Flows ... Purchases of PP&E ...",
      "evidence_text_full_page": "...",
      "doc_name": "3M_2018_10K",
      "evidence_page_num": 47
    }
  ],
  "doc_period": 2018,
  "doc_type": "10k"
}
```

`evidence` es una lista de **dicts**, no de strings. Cada dict tiene 4 sub-campos: `evidence_text` (el passage), `evidence_text_full_page` (la página entera donde vive el passage — usado en la sección 2.4), `doc_name` y `evidence_page_num`. Esa estructura más rica es el scaffolding del ground truth para cada métrica de retrieval que vamos a computar.

**Por qué `evidence` es el campo más importante del dataset.** Cada item del array es un *passage* — un fragmento del PDF, sized para ser pedagógicamente significativo por sí mismo (típicamente unos pocos párrafos relacionados). Esos pasajes no son automáticos: annotators humanos de Patronus leyeron cada 10-K y seleccionaron a mano qué fragmento justifica cada respuesta. Eso los convierte en el **ground truth** para evaluar retrieval. **Sin ellos la evaluación sería a ciegas** — no podrías decir "el chunk que recuperaste contiene la respuesta correcta". Con ellos, computas Recall@k chequeando si el chunk recuperado se solapa con el `evidence_text`. Este único campo es lo que convierte a FinanceBench de curiosidad a benchmark riguroso.

**Pasaje vs chunk** — distinción que va a aparecer toda la lección: en FinanceBench los `evidence_text` son **pasajes curados por humanos**. En nuestro sistema RAG vamos a generar **chunks automáticamente** con distintas estrategias y compararlos contra esos pasajes humanos — así las métricas miden si el chunking captura la misma información que los humanos seleccionaron. Una *chunking strategy ganadora* es aquella cuyos chunks generados matchean consistentemente los pasajes curados.

### 2.1 Diversidad documental — ¿el eval set es robusto o nicho?

**La pregunta que auditamos**: ¿el dataset cubre suficientes compañías, sectores y años para que las métricas generalicen, o tiene sesgo hacia un slice estrecho de la economía?

Si las 150 QA pairs fueran sobre una sola compañía o sector (ej: solo Tech), nuestros embedders podrían "memorizar" el vocabulario de ese nicho y producir métricas infladas que no reflejan el performance real en otros dominios financieros.

#### Qué medimos

- Número de compañías distintas (más = mejor diversidad)
- Número de sectores GICS cubiertos (de 11 posibles)
- Rango temporal (años fiscales)
- Mix de tipos de documento (10-K, 10-Q, 8-K, earnings releases)

#### Hallazgos

**Cobertura sólida:**

- **32 compañías distintas** sin un outlier dominante (max: PepsiCo con 11/150 = 7.3%) — las métricas no están capturando idiosincrasias de una sola firma.
- **9 de 11 sectores GICS** representados — los únicos ausentes son Energy y Real Estate. Si los embedders generalizan acá, deberían generalizar en casi todo el corporate finance.
- **10 años cubiertos (2015-2024)** — ventana temporal amplia.

**Cosas que documentar como limitaciones:**

| Tipo de documento | # | % | Naturaleza |
|---|---:|---:|---|
| `10k` | 112 | 74.7% | Annual reports (audited, formal SEC filings) |
| `10q` | 15 | 10.0% | Quarterly reports (unaudited) |
| `Earnings` | 14 | 9.3% | **Voluntary investor communications, NO son SEC filings formales** |
| `8k` | 9 | 6.0% | Event-driven reports |

El 9.3% de earnings releases importa: no siguen formato SEC-mandated y reflejan más el "spin" del management que el regulatory scrutiny. Documentamos esa heterogeneidad como feature del dataset en vez de filtrarla.

**Sesgo temporal**: 2022-2023 acumulan ~64% combinados. Los modelos van a evaluarse sobre filings recientes, lo cual es realista para producción pero vale flaggear en el README.

> **Decisión fijada**: mantener los **150 records mezclados** sin filtrar. Razón: FinanceBench es un eval set público estándar de la literatura; modificarlo haría nuestros resultados incomparables con otros papers. La heterogeneidad de fuentes se documenta como feature, no como bug.

### 2.2 Tipos de pregunta — ¿qué razonamiento necesita el sistema?

**La pregunta que auditamos**: ¿cada QA es un lookup simple (*"what was the revenue?"*) o una inferencia multi-step (*"did margin grow faster than CapEx?"*)? Esto define qué tan desafiante es el dataset y qué tiene que hacer el LLM downstream **después** del retrieval.

Patronus etiquetó cada pregunta con **dos labels independientes**:

- **`question_type`** → de dónde nace la pregunta (origen y estilo)
- **`question_reasoning`** → qué razonamiento exige para responderla

**Independientes** significa que ninguno determina al otro — no son una jerarquía de dificultad. Una `metrics-generated` puede ser fácil o difícil; una `novel-generated` puede ser trivial o compleja. La diferencia está en el **origen**, no en la complejidad inherente.

**¿Por qué estas categorías específicas?** Porque el PDF de un 10-K tiene **3 tipos distintos de contenido**, y cada `question_type` refleja uno. La taxonomía no es arbitraria — sigue la estructura real del documento.

#### `question_type` — el origen de cada pregunta

##### `metrics-generated` (~50/150)

**De dónde nace**: los **estados financieros estandarizados** del 10-K. Tres tablas auditadas, reguladas por SEC GAAP, estructuralmente idénticas en todas las compañías:

- **Income Statement** — revenue, costs, profitability (`Net sales`, `Cost of goods sold`, `Operating income`)
- **Balance Sheet** — assets, liabilities, equity (`Total assets`, `Long-term debt`, `Stockholders' equity`)
- **Cash Flow Statement** — cash inflows/outflows (`Capital expenditures`, `Operating cash flow`)

**Cómo Patronus las construyó**: tomó una lista de métricas estándar de estas tablas y aplicó plantillas rígidas tipo *"What is the {metric} for {company} in fiscal year {year}?"* — una sola plantilla genera 50+ preguntas variando los parámetros.

> **Ejemplo real**: 3M 2018 10-K, página 59 (Statement of Cash Flows) → línea `Purchases of property, plant and equipment ... $(1,577)` → la plantilla genera *"What is the FY2018 capital expenditure amount in USD millions for 3M?"* → respuesta literal **$1,577M**.

##### `domain-relevant` (~50/150)

**De dónde nace**: las **secciones narrativas obligatorias** del 10-K. Más allá de las tablas auditadas, cada 10-K tiene prosa libre regulada por la estructura de Items SEC:

- **Item 1** — Business (operations, products, markets, competition)
- **Item 1A** — Risk Factors (riesgos específicos que la compañía identifica)
- **Item 7** — MD&A (Management's Discussion and Analysis)

Estas secciones son **predecibles en su existencia** (todo 10-K las tiene) pero **variables en contenido** (cada compañía describe sus propios riesgos). Humanos escribieron las preguntas con foco claro en estas secciones, formato libre.

> **Ejemplo real**: 3M 2018 10-K, Item 1A → varios párrafos sobre PFAS contamination, litigation y supply chain disruption → pregunta *"What are the main risks 3M identifies?"* exige leer y sintetizar prosa, no hay un único valor que extraer.

##### `novel-generated` (~50/150)

**De dónde nace**: información que **cruza secciones** del 10-K. En vez de quedarse dentro de una sola tabla o un solo Item, las preguntas combinan signals de múltiples partes del documento:

- Una **métrica** de los estados financieros
- Una **narrativa** de los items cualitativos (Item 1 Business / Item 7 MD&A)
- Un **riesgo** de Item 1A

Humanos las escribieron completamente abiertas, simulando cómo un analista real explora un 10-K sin agenda predefinida. Su impredecibilidad las convierte en el **stress test** del benchmark.

> **Ejemplo real**: Pfizer 2022 10-K → pregunta *"How is Pfizer's R&D pipeline positioning the company in oncology?"* → exige cruzar 3 secciones: R&D spend en Income Statement + pipeline narrative en Item 1 Business + competitive risks en Item 1A.

**Por qué importa para evaluación**: un sistema RAG puede ganar en `metrics-generated` (plantillas predecibles) y **fallar feo en `novel-generated`** — eso revela que el sistema está overfitted al estilo del prompt en vez de entender la semántica. La categoría novel fuerza generalización real.

#### `question_reasoning` — qué razonamiento exige (dónde vive la respuesta)

Mientras `question_type` te dice de qué parte del PDF viene la pregunta, `question_reasoning` te dice qué tan elaborado tiene que ser el procesamiento *después* de recuperar el chunk.

##### `Information extraction` (~21%)

**Dónde vive la respuesta**: literal en una celda de tabla o en una mención directa de la prosa. El sistema solo necesita encontrar el chunk correcto y devolver el texto. Sin matemáticas, sin inferencia.

> **Ejemplo real**: 3M 2018 Income Statement → línea `Net sales ... $32,765` → pregunta *"What was 3M's revenue in FY2018?"* → respuesta **$32,765M** literal en el PDF.

##### `Numerical reasoning` (~29%)

**Dónde vive la respuesta**: **NO** está literal en el PDF. El documento contiene los inputs (2 o más valores), pero el resultado lo computa el LLM downstream después del retrieval.

> **Ejemplo real**: 3M 2017 Net sales = $31,657M, 2018 Net sales = $32,765M → pregunta *"What was the YoY growth rate?"* → la respuesta NO está en el PDF, hay que computar `(32,765 - 31,657) / 31,657 = 3.5%`. El sistema RAG debe recuperar un chunk que contenga ambos números (idealmente la tabla completa de comparación interanual).

##### `Logical reasoning (multi-step)` (~17%)

**Dónde vive la respuesta**: requiere combinar piezas de información, posiblemente de **secciones distintas** del PDF. Extraer datos, computar ratios intermedios, comparar, emitir un juicio.

> **Ejemplo real**: pregunta *"Did 3M's CapEx grow faster than its revenue between 2017 and 2018?"* → los inputs son 4 números de 2 secciones distintas: Income Statement (Revenue 2017 + 2018) y Cash Flow Statement (CapEx 2017 + 2018). Proceso: extraer → computar 2 ratios de growth → comparar → emitir yes/no. El sistema RAG **debe recuperar chunks de 2 secciones distintas** del 10-K. Esta es exactamente la razón por la que ~23% del dataset tiene 2-3 evidences distintos (lo veremos en §2.4).

> **Caveat sobre `None`**: ~33% de las preguntas (todas las `novel-generated`) tienen `question_reasoning = None`. **NO significa que requieran más procesamiento** — Patronus simplemente no las clasificó. Una pregunta novel puede ser un lookup simple o un cálculo complejo; las dejaron free-form porque su diversidad las hace difíciles de meter en categorías fijas.

#### Hallazgos

- **`question_type` está perfectamente balanceado** (50/50/50): metrics-generated, domain-relevant, novel-generated. Patronus diseñó el dataset con cuotas iguales — no hay sesgo de tipo que corregir.
- **33% del dataset tiene `question_reasoning = None`** — todas las novel-generated, por la razón explicada arriba.
- **Cross-tab insight**: metrics-generated se divide en 72% Numerical + 28% Information extraction (las plantillas son cálculos o lookups). Domain-relevant tiene más diversidad, incluyendo los casos híbridos multi-step.

#### Implicación para el proyecto

El subset de ~29% Numerical reasoning significa que el sistema RAG necesita **dos competencias**: recuperar el chunk correcto, **y** que el LLM downstream haga el cálculo. El Stage 1 de este proyecto mide **retrieval only** — el cálculo es responsabilidad del Eslabón 2 (un body of work separado). Dividir la responsabilidad limpiamente nos permite shipear una baseline de retrieval honesta sin mezclar dos failure modes.

Y el ~17% de Logical multi-step explica el **origen del ~23% multi-evidence** que veremos en §2.4: cuando una pregunta exige combinar piezas de 2 secciones distintas del 10-K, Patronus anotó 2 (o 3) `evidence` items en lugar de 1.

### 2.3 Distribuciones de longitud — ¿questions, answers y evidence caben en chunks estándar?

**La pregunta que auditamos**: ¿qué tan largos son los textos del dataset? Esto define dos cosas críticas para el sistema RAG:

1. **Token budget del embedder** — los embedders modernos típicamente capean en 512 tokens. Si los chunks o queries lo exceden, o truncamos (perdiendo info) o cambiamos modelo.
2. **Chunking strategy apropiada** — si los passages de `evidence` son muy largos, necesitamos chunks más grandes **o** retrieval que devuelva múltiples chunks para cubrir el evidence completo.

#### Mini-primer: cómo leer una distribución

Cuando reportamos longitudes (o métricas tipo Recall@k más adelante), no reportamos un solo número — reportamos la **forma** de la distribución. Para eso usamos 7 términos:

| Término | Qué es | Cuándo lo usas |
|---|---|---|
| **min** | El valor **más chico** del dataset | Detectar piso / outliers bajos |
| **max** | El valor **más grande** del dataset | Detectar techo / outliers altos |
| **mean** | **Promedio aritmético** (suma / count) | Centro de la distribución, **pero sensible a outliers** |
| **median** (= **p50**) | **Valor del medio**: 50% abajo, 50% arriba | Centro **robusto**, ignora outliers |
| **p25** | 25% del dataset queda por debajo | Borde inferior del cuerpo central |
| **p75** | 75% del dataset queda por debajo | Borde superior del cuerpo central |
| **p95** | 95% del dataset queda por debajo | Cola larga / casos extremos pero no-outlier |

**Analogía**: imagínate una fila de 100 personas ordenadas por altura. `min` = la más baja (posición 1). `p25` = la altura de la persona en posición 25. `median` (p50) = la persona en posición 50. `p75` = posición 75. `p95` = posición 95 (casi al final). `max` = la más alta (posición 100). `mean` = el promedio de las 100 alturas sumadas y divididas.

**Por qué reportamos los 7 juntos**: la diferencia entre `mean` y `median` es lo que te dice si la distribución es simétrica o tiene cola larga.

| Si... | Significa... |
|---|---|
| `mean ≈ median` | Distribución **simétrica** (campana clásica, pocas sorpresas) |
| `mean >> median` | **Cola larga a la derecha** (algunos outliers altos tiran el promedio para arriba) |
| `mean << median` | Cola larga a la izquierda (raro pero posible) |

#### Distribución en chars (y tokens aproximados, a ~4 chars/token en inglés)

| Campo | Longitud (median · p95 · max) | Tokens (median) |
|---|---|---|
| `question` | 137 · 356 · 592 chars | ~34 |
| `answer` | 50 · 285 · 609 chars | ~12 |
| `justification` | 100 · 488 · 703 chars | ~25 |
| **`evidence` (suma per QA)** | **1,450 · 4,194 · 12,123 chars** | **~362** |

#### Hallazgos

- Questions y answers son chiquitas — caben sobradas en cualquier embedder.
- `justification` puede estar vacía en ~25% del dataset (`p25 = 0`).
- `evidence` tiene **cola muy larga a la derecha**: el outlier llega a 12,123 chars (~3,030 tokens), 8× el median. El mean (1,712) por encima del median (1,450) confirma la cola.
- **76.7% de las QAs tienen 1 evidence**, 20.7% tienen 2, 2.7% tienen 3 — así que ~23% del dataset es multi-evidence.

#### Visualizando la cola larga

```text
Frecuencia (cuántos evidences caen en cada rango)
  │
  │  ▓▓▓▓▓▓▓
  │  ▓▓▓▓▓▓▓▓▓▓
  │  ▓▓▓▓▓▓▓▓▓▓▓▓▓
  │  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░░ ........... ... ..    .       .
  └──────────────────────────────────────────────────────────────────
     0     1k    2k    3k    4k    5k   ...   8k   ...   10k  ...  12k  chars
        ↑ median ↑ mean         ↑ p95                              ↑ max
        (1,450) (1,712)        (4,194)                          (12,123)
```

La "cola" es la parte derecha que se extiende: pocos casos pero muy grandes. Técnicamente se llama **right-skewed distribution**.

> 💡 **Si solo midiéramos el mean (1,712 chars)**, perderíamos la cola larga. Diseñaríamos un sistema asumiendo que el evidence típico es ~1,712 chars y los outliers nos sorprenderían. Por eso **siempre median + percentiles**, nunca solo mean.

#### La implicación crítica para chunking

Si chunkeamos a **512 tokens (~2,048 chars)** — la convención estándar para embedders BERT-family:

| Percentil de evidence | Tokens (est.) | ¿Cabe en 1 chunk? |
|---|---:|---|
| Median | ~362 | Plenty of room |
| p75 | ~566 | Apenas se pasa → necesita 2 chunks |
| p95 | ~1,048 | Necesita 2-3 chunks |
| Max | ~3,030 | Outlier — necesita 6+ chunks |

**Para ~25-30% del dataset, el evidence completo no entra en 1 chunk.** Esa es la observación que justifica dos design choices: (1) reportar Recall@k en múltiples valores de `k` (1, 3, 5, 10), y (2) incluir MAP, que premia recuperar múltiples chunks relevantes para la misma query.

#### La cadena operativa que conecta longitud → diseño del sistema

Las longitudes no son un dato académico — **manejan decisiones técnicas concretas** del RAG. El razonamiento encadenado:

```text
1. Longitud del evidence (chars)         ← lo que acabamos de medir
   ↓ (regla de pulgar: ~4 chars ≈ 1 token en inglés)
2. Tokens estimados                      ← cuántos tokens tendría el evidence si lo embedeas
   ↓
3. Comparar contra el max_tokens del embedder (típicamente 512)
   ↓
4. ¿Cabe en 1 chunk?
     ├─ SÍ → puedes hacer retrieval con k=1, las métricas van a ser optimistas
     └─ NO  → necesitas chunking + retrieval con k>1
   ↓
5. Chunking strategy + valor de k apropiado
```

#### Regla de oro

Cuando la distribución tiene cola larga (típico de datos del mundo real: salarios, tamaños de archivo, longitudes de texto, scores de retrieval), **siempre reportar median + percentiles**, no solo mean. El periodismo honesto usa "salario mediano", no "salario promedio", por exactamente esta razón.

Esto también explica por qué en stages siguientes vamos a reportar **bootstrap confidence intervals** para nuestras métricas de retrieval — en lugar de un solo número tipo "Recall@5 = 0.80", vamos a reportar algo como "Recall@5 = 0.80 [0.74, 0.85] (95% CI)". El intervalo captura la incertidumbre del estimate, computado resampleando el eval set 1,000 veces.

### 2.4 Distribución del evidence — ¿dónde viven realmente las respuestas en un 10-K?

**La pregunta que auditamos**: cuando el sistema RAG hace retrieval, ¿qué tan profundo en el 10-K tiene que mirar? ¿Las respuestas están en el executive summary, en el MD&A, en los financial statements, o dispersas?

Medimos tres sub-dimensiones:

1. Distribución de `evidence_page_num` — ¿en qué páginas viven los evidences?
2. Locality del multi-evidence — cuando hay 2-3 evidences, ¿están cerca o lejos?
3. Ratio `evidence_text` / `evidence_text_full_page` — ¿el evidence es casi toda la página o un fragmento chico?

#### Hallazgo 1 — ~85% de los evidences viven en las primeras 75 páginas del 10-K

| Rango de páginas | % | Mapea a sección típica del 10-K |
|---|---:|---|
| pp. 0-25 | 27.5% | Item 1 (Business) + Item 1A (Risk Factors) |
| pp. 26-50 | 19.0% | Item 7 (MD&A) inicial |
| **pp. 51-75** ⭐ | **38.1%** | **Item 8 (Financial Statements)** — el corazón del 10-K |
| pp. 76-100 | 7.9% | Notes to Financial Statements |
| pp. 101+ | 7.5% | Appendices y exhibits |

`mean ≈ median` (ambos = 51) → distribución bastante simétrica, sin colas distorsionando el centro. El 10-K tiene una geografía predecible: la mayoría de las respuestas viven en la sección de financial statements, y el resto se concentra alrededor de MD&A y risk factors.

#### Hallazgo 2 — Multi-evidence es local: 91% dentro de ≤5 páginas

Cuando una QA tiene 2-3 evidences, casi siempre vienen de la **misma sección** del 10-K (ej: ambos del Cash Flow Statement, o ambos del Balance Sheet).

**Implicación técnica**: **NO** necesitamos MMR (Maximal Marginal Relevance) ni reranking diversity-aware para el caso típico. Los chunks relevantes están clusterizados, así que el top-k vanilla con cosine similarity los captura bien. Solo el ~9% outlier (multi-evidence cross-section) se beneficiaría de un paso diversity-aware — y la movida correcta de ingeniería es shipear la baseline simple primero y solo agregar complejidad si el failure analysis muestra que ese 9% está arrastrando la métrica desproporcionadamente.

#### Hallazgo 3 — Distribución bimodal del ratio evidence/página

¿Qué fracción de la página es el evidence?

- **46% de los evidences cubren casi toda la página** (>90%) — típico cuando la página es una tabla financiera completa (Income Statement, Balance Sheet, Cash Flow Statement).
- **29% son fragmentos chicos** (menos del 25% de la página) — típico cuando la página cubre múltiples temas y el evidence es un párrafo específico.
- El 25% restante queda en el medio.

**Esta es una distribución bimodal**, y es la observación más importante de la auditoría. Revela un **trade-off que ninguna chunking strategy puede esquivar**:

| Strategy | Performance |
|---|---|
| Chunk = página completa | Perfecto para el 46% (tablas) · Diluye fragmentos chicos |
| Chunk = 512 tokens | Captura fragmentos chicos · Puede partir tablas grandes |
| **Semantic chunking** | Respeta semantic boundaries — cubre ambos regímenes parcialmente |
| **Late chunking** | Embedea el documento completo primero, decide los chunks después |

Esta es **la razón por la que el proyecto compara 4 chunking strategies** en vez de comprometerse con una: cada una está optimizada para una parte distinta de la distribución. Medimos las cuatro y dejamos que los datos decidan. El write-up formal aterriza en `docs/chunking_decisions.md` cuando cierre el sub-bloque 6.

## Sub-bloque 3 — Descarga de los PDFs source

Los sub-bloques 1 y 2 nos dieron las **preguntas** y los **passages de ground truth** (el campo `evidence`). Para correr retrieval real, todavía nos falta el **corpus desde el cual recuperamos**: los PDFs de 10-K, 10-Q, 8-K y earnings de los que esas preguntas y passages vienen.

El campo `evidence_text` ya contiene el texto relevante, pero es un **excerpt de 2-3 párrafos** sacado de un documento de 100-300 páginas. Para evaluar retrieval honestamente necesitamos chunkear e indexar los **PDFs completos** — si no, le estaríamos dando al sistema un corpus chiquito pre-filtrado y las métricas serían irrealmente optimistas. Eso es trabajo de los sub-bloques 4 (parsing) y 5 (chunking). Este sub-bloque aterriza la **receta** para tener los PDFs en disco.

### La decisión: receta, no redistribución

Los 84 PDFs únicos suman ~165 MB. Podríamos commitearlos al repo. No lo hacemos. Tres razones manejaron esa decisión, y cada una vale entender porque el mismo trade-off aparece en cada proyecto de ML que toca data third-party:

#### 1. Higiene del repo — git no es un binary store

Git guarda cada revisión de cada archivo trackeado en su historia para siempre. Agregás 165 MB de binarios hoy y los clones quedan 165 MB más pesados **para cada developer, siempre**. Reemplazás un PDF dentro de un año y la historia agrega otros 165 MB. Después de algunas iteraciones el repo es multi-gigabyte, cada clone es lento, cada CI run paga el costo, y `git log` contra los blobs binarios es inutilizable.

> **Regla general**: cualquier cosa más grande que ~1 MB que cambie ocasionalmente probablemente no pertenece a git. Usá git-lfs, S3, un release artifact, o — como hacemos acá — convertilo en un input derivado que los users bajan con un script.

#### 2. Licensing — FinanceBench es `CC-BY-NC-4.0`

Patronus AI distribuye FinanceBench bajo Creative Commons Attribution-NonCommercial 4.0. Re-hostear los mismos PDFs desde un repo público distinto entra en territorio legalmente ambiguo: es una redistribución del dataset licenciado de otro bajo la sombrilla de un repo distinto. Apuntar a los users al source upstream — `patronus-ai/financebench` en GitHub — es inequívocamente OK. **La receta no redistribuye nada; solo le dice a `requests` qué URLs golpear.**

#### 3. Single source of truth — el upstream es canónico

Si Patronus actualiza un PDF (re-emite un 10-K corregido, arregla un error de metadata), el repo upstream recibe el fix. Una copia en nuestro repo no, y ahora estamos silenciosamente corriendo el eval contra documentos viejos. El patrón de receta esquiva esto del todo — cada clone fresco baja lo que upstream tenga en `main` en ese momento.

<Callout>
  Este patrón se llama **"recipe, not redistribution"** y es estándar para proyectos de ML que dependen de corpora third-party (ej: las dataset cards de HuggingFace linkean al source original en vez de re-hostear). Llamarlo explícitamente en el README + `docs/reproducibility.md` es lo que lo convierte de "no nos molestamos en commitear los datos" en una decisión deliberada y defendible de ingeniería.
</Callout>

### Descarga selectiva — 84 PDFs, no 368

El repo upstream `patronus-ai/financebench` tiene **368 PDFs** (~705 MB). El split open-source de FinanceBench solo referencia **84 de ellos** (las QAs que usamos). Bajar los otros 284 desperdicia bandwidth, disco y tiempo. La receta extrae los valores únicos de `doc_name` desde el dataset mismo y solo baja lo que efectivamente está referenciado:

```python
def get_unique_doc_names() -> list[str]:
    """Load the FinanceBench dataset and return the unique doc_name values."""
    ds = load_dataset("PatronusAI/financebench")
    train = ds["train"]
    # set() collapses duplicates; sorted() makes the order deterministic.
    return sorted(set(train["doc_name"]))
```

`set()` colapsa duplicados (múltiples QAs referencian el mismo `doc_name`); `sorted()` hace que el orden sea determinista para que la barra de progreso y cualquier logging sea reproducible entre runs. **150 QA pairs → 84 PDFs únicos.**

### Idempotente y paralelo — los dos no-negociables

Dos propiedades que cualquier script de data-fetch en un proyecto serio debería tener:

#### Idempotente — re-correr el script tiene que ser safe

Si el script crasheó en el PDF #50 de 84, re-correrlo no debería empezar de cero. El patrón es muy simple: chequear si el destino existe y es non-empty, skipear si sí:

```python
def download_one(doc_name: str) -> tuple[str, str, int]:
    dest = DEST_DIR / f"{doc_name}.pdf"

    # Idempotency: skip if already present and non-empty.
    if dest.exists() and dest.stat().st_size > 0:
        return (doc_name, "skipped", dest.stat().st_size)
    ...
```

El check es sobre **archivo non-empty**, no solo sobre existencia del archivo — porque una descarga parcial (la red murió a mitad de stream) deja un placeholder de 0 bytes. Skipear un archivo de 0 bytes significaría que el corpus queda incompleto para siempre. El script también unlinkea el destino ante cualquier fallo de descarga, así el siguiente run ve la mesa limpia.

> 💡 **Por qué la idempotencia importa en la práctica**: en un setting de curso u onboarding, los estudiantes inevitablemente van a darle Ctrl+C a la mitad. En un setting CI/research, el runner es preempted. En un setting de equipo, dos personas corren el script al mismo tiempo. Un script idempotente tolera los tres; uno no-idempotente corrompe el corpus y deja al siguiente debugeando.

#### Paralelo — pero con buenas maneras

84 requests HTTP secuenciales a ~1-2 s cada uno = ~2 minutos. Con 8 workers paralelos es ~10 segundos. El patrón desde la stdlib es `ThreadPoolExecutor` — threads (no processes) son el modelo correcto acá porque el cuello de botella es **I/O de red**, no CPU:

```python
with ThreadPoolExecutor(max_workers=MAX_WORKERS) as pool:
    futures = {pool.submit(download_one, name): name for name in doc_names}
    with tqdm(total=len(futures), desc="PDFs", unit="file") as pbar:
        for future in as_completed(futures):
            name, status, size = future.result()
            results[status] += 1
            ...
            pbar.update(1)
```

Tres piezas que vale resaltar:

1. **`max_workers=8`**: un default educado. Valores más altos pueden disparar el rate-limiter de GitHub. 8 es suficiente para saturar una conexión hogareña típica sin antagonizar al CDN.
2. **`as_completed(futures)`**: el loop tickea **a medida que cada descarga termina**, no en orden de submission. La barra de progreso se mantiene actualizándose de forma pareja aunque algunos PDFs sean de 1 MB y otros de 8 MB.
3. **`stream=True` + `iter_content(chunk_size=8192)`** dentro de `download_one`: previene cargar un PDF entero de varios MB en memoria antes de escribirlo a disco. Importante cuando 8 workers están bajando en paralelo — sin streaming, la RAM peak escala linealmente con el número de workers.

### Lo que NO hicimos, y por qué

- **Sin git-lfs**. LFS resuelve un problema distinto (versionado de binarios grandes) y agrega overhead operativo (LFS tokens, quotas de bandwidth) que no se justifica acá.
- **Sin upload a HuggingFace dataset**. Podríamos espejear los PDFs como un HF dataset, pero es redistribución bajo otra jurisdicción — mismo concern de licensing que commitearlos a git.
- **Sin pinning del commit upstream todavía**. El script golpea `patronus-ai/financebench/main` en vez de un SHA específico. Para trabajo del día a día está bien; para un release `v1.0.0` paper-quality vamos a pinnear el SHA así un reviewer corriendo la receta dentro de un año recibe PDFs bit-idénticos. Esa hardening aterriza una vez que el eval pipeline esté publicado. (Ver `docs/reproducibility.md` § *Corpus integrity* para el plan completo.)

## Sub-bloque 4 — Parsing PDFs con `pdfplumber`

Los sub-bloques 1, 2 y 3 nos dejaron un dataset auditado y los 84 PDFs source en disco. **Pero el embedder no come PDFs binarios**: necesita texto estructurado. Sub-bloque 4 cierra ese cuello de botella — convierte los 84 PDFs en JSONL indexable, página por página, con tablas extraídas como matrices 2D limpias.

El bloque sigue un **scaffold pedagógico de 3 niveles**: leer la teoría densa primero, materializarla en demos ejecutables, y después escalar al corpus completo.

### 1. PRE-READ obligatorio — `docs/CONTEXT.md` § 13

Antes de tocar código, lee la sección **§ 13 — Parsing PDFs: from positioned glyphs to indexable structure** del [`docs/CONTEXT.md`](https://github.com/FMFigueroa/financebench-rag-eval/blob/main/docs/CONTEXT.md#13-parsing-pdfs--from-positioned-glyphs-to-indexable-structure) del repo proyecto. Son ~220 líneas distribuidas en 6 sub-secciones:

- **§ 13.1** — Por qué un PDF es hostil al ML (la herencia PostScript: el archivo describe glyphs en coordenadas, no estructura semántica). Sienta el modelo mental: parsear un PDF es **layout reverse-engineering**, no leer un text file.
- **§ 13.2** — Head-to-head de 5 librerías (`pypdf`, `pdfplumber`, `PyMuPDF`, `tabula-py`, `Marker`/`Docling`). Justifica por qué `pdfplumber` gana **para este corpus** (10-K con tablas con bordes visibles, exposición de coordenadas raw, MIT license, zero setup friction).
- **§ 13.3** — El gotcha del default `extract_tables()`: la estrategia `"lines"` (default) ve las separator lines del Income Statement como bordes de mini-tablas y fragmenta una matriz de 33 filas en ~12 sub-tablas de 1 fila. El fix: cambiar a `"text"` y dejar que clusterice por proximidad espacial.
- **§ 13.4** — Tres técnicas de chunking de tablas con la matemática del trade-off: whole-table-as-one-chunk, header-repetition (la del baseline), row-as-sentence linearization. Cada una preserva y pierde cosas distintas.
- **§ 13.5** — Anclaje cuantitativo del corpus: 84 PDFs × ~141 páginas avg = 11,853 páginas / 19.4M tokens / ~63K chunks / ~$2.52 por embedder pass. La tabla que sostiene el budget de $40-50 USD del proyecto entero.
- **§ 13.6** — Payoff arquitectónico: la **boundary parser/chunker**. Persistir el output del parser como JSONL hace que cada strategy de chunking se pueda evaluar contra el mismo corpus parseado, en segundos en lugar de re-parsear los 84 PDFs cada vez (~26 min).

## Síntesis — los 4 beats narrativos

Después de cinco sub-secciones de mediciones, la auditoría cuenta una sola historia coherente. Estos son los takeaways a internalizar antes de seguir:

### Beat 1 — FinanceBench es un benchmark serio, no un toy example

Las 150 preguntas son queries reales que un junior analyst preguntaría en un investment bank o fund:

- **32 compañías distintas** en **9 sectores económicos** (todo menos Energy y Real Estate)
- **10 años de cobertura** (2015-2024), con sesgo hacia 2022-2023
- Mix balanceado en tres categorías de pregunta (templated / domain-broad / novel)

> **Traducción**: si un sistema RAG funciona acá, hay un prior fuerte de que va a funcionar en cualquier dominio financiero corporativo del mundo real.

### Beat 2 — El sistema no solo tiene que **encontrar** — tiene que **entender**

El campo `question_reasoning` revela tres niveles de dificultad:

- **~21% lookup directo** — el sistema solo necesita encontrar el número.
- **~29% requieren cálculo** — encontrar dos números y operar.
- **~17% razonamiento multi-step** — múltiples extracciones, ratios, comparaciones.

> **Traducción**: este proyecto cubre solo la primera mitad del trabajo (encontrar el chunk correcto). El cálculo y el razonamiento viven en el LLM downstream (Eslabón 2 del roadmap más amplio). Dividir la responsabilidad limpiamente significa que la baseline de retrieval es honesta sobre lo que sí promete y lo que no.

### Beat 3 — Hay un patrón geográfico claro dentro del 10-K

Los evidences no están dispersos al azar — viven en su mayoría en una zona específica del documento:

- **85% en las primeras 75 páginas** del 10-K
- Concentración fuerte en **páginas 51-75** (financial statements)
- Multi-evidences están **91% dentro de ≤5 páginas** entre sí

> **Traducción**: no hace falta leer el 10-K entero para responder. La mayoría de las respuestas viven en una zona predecible. Esto justifica optimizaciones futuras (ej: indexar solo páginas 0-150) y permite retrieval simple (top-k vanilla) en vez de algoritmos complejos de diversity.

### Beat 4 — El dataset es honesto sobre su propia complejidad

La distribución bimodal del ratio evidence/página revela un trade-off fundamental:

- **46% evidences full-page** (tablas financieras)
- **29% fragmentos chicos** (párrafos específicos en páginas densas)

> **Traducción**: NO existe **una chunking strategy** que gane todos los casos. Si chunkeamos a página completa, fallamos con fragmentos. Si chunkeamos chico, partimos las tablas. **Por eso el proyecto compara 4 strategies** — cada una está optimizada para un perfil distinto del dataset.

## Decisiones técnicas justificadas por esta auditoría

Esto es el ROI del bloque: cada design choice que sigue se puede defender con **números específicos** de los datos, no con opinión o convención.

| Decisión técnica | Justificación (desde la auditoría / sub-bloque 3) |
|---|---|
| Reportar **4 métricas** (Recall@k, MRR, NDCG, MAP) | Multi-evidence existe en ~23% de las QAs; MRR sola engaña, MAP captura todos los relevants per query |
| Reportar **múltiples valores de `k`** (k=1, 3, 5, 10) | Evidence puede no caber en 1 chunk (long-tail de longitudes); k=5,10 mide "al menos está cerca" |
| Comparar **4 chunking strategies** | Ratio evidence/página bimodal (46% full-page vs 29% fragments) — no hay one-size-fits-all |
| Mantener el dataset **mezclado** (no filtrar a solo 10-K) | Filtrar nos haría incomparables con la literatura académica que usa FinanceBench as-is |
| **Saltarse MMR / diverse retrieval** en la baseline | Multi-evidences están clusterizados (≤5 páginas) en 91% de los casos — la diversity penalty es la movida equivocada |
| **Receta, no redistribución** para los PDFs source | Higiene del repo + licensing `CC-BY-NC-4.0` + single source of truth (repo upstream de Patronus) |
| **Descarga selectiva** (84 de 368 PDFs upstream) | Solo el split open-source se usa; los otros 284 PDFs gastarían ~540 MB y quedarían sin leer |

## Cuestionario — valida lo aprendido

Trata de responder cada pregunta antes de abrir la respuesta.

### Pregunta 1 — ¿Qué retorna `load_dataset("PatronusAI/financebench")` y por qué?

<details>
<summary>Ver respuesta</summary>

Retorna un `DatasetDict`, un container dict-like donde las keys son los splits del dataset (acá solo `"train"`) y los values son objetos `Dataset` backed por Apache Arrow. La forma DatasetDict es la manera de la librería de expresar "este dataset tiene múltiples splits"; `Dataset` es la unidad queryable real. Aún cuando hay un solo split (como en datasets eval-only), el wrapper DatasetDict se preserva por consistencia de API.

</details>

### Pregunta 2 — ¿Por qué FinanceBench solo trae el split `"train"` cuando es un dataset de evaluación?

<details>
<summary>Ver respuesta</summary>

Es una convención de naming de la librería `datasets`. Cualquier split único se nombra `"train"` por default, sin importar la intención de uso. FinanceBench es **evaluation-only** — Patronus lo diseñó para que midas qué tan bueno es tu sistema RAG ya construido, no para entrenar nada. La decisión fijada de la lesson es: **nunca dividir FinanceBench**. Las 150 QAs quedan intactas como eval set sagrado; para fine-tuning (Stage 3) usamos otra fuente de 10-K y construimos splits de ahí. Esto garantiza zero contamination — el modelo nunca ve las QAs del eval durante el entrenamiento.

</details>

### Pregunta 3 — ¿Cuál es la diferencia entre Parquet y Apache Arrow, y por qué HuggingFace usa los dos?

<details>
<summary>Ver respuesta</summary>

Parquet es un formato columnar **on-disk**, optimizado para almacenamiento (comprimido, tipado, columnar para que los reads sean parciales). Arrow es un formato columnar **in-memory**, optimizado para procesamiento (memory-mapped, zero-copy, estándar cross-language). HuggingFace los combina: descarga Parquet del Hub → materializa como Arrow en RAM → expone vía la API `Dataset`. Parquet mantiene el cold storage compacto; Arrow hace que las operaciones hot sean rápidas y permite que datasets más grandes que la RAM sigan siendo queryable vía memory-mapped I/O.

</details>

### Pregunta 4 — ¿Qué línea rompe silenciosamente la abstracción de lazy-loading y por qué importa?

<details>
<summary>Ver respuesta</summary>

`data = list(train)` — la llamada a `list()` materializa cada row como un `dict` Python en RAM, descartando los beneficios de memory-mapping y zero-copy de Arrow. Desde ese punto volvimos a uso de memoria tipo pandas, y la siguiente operación (ej: un filter o map) es 10-100× más lenta que llamar `train.filter()` o `train.map()` directo. Lo mismo pasa con `.to_pandas()` en datasets grandes y con loops Python ingenuos que acumulan resultados en una lista. El fix siempre es: quedarte adentro de la API de `Dataset`.

</details>

### Pregunta 5 — ¿Por qué el campo `evidence` es el campo más importante del dataset?

<details>
<summary>Ver respuesta</summary>

Porque `evidence` es el **ground truth** para evaluación de retrieval: cada item es un passage del PDF que justifica la respuesta, curado por un annotator humano de Patronus. Sin él, evaluar retrieval es a ciegas — no tendrías forma de decir "el chunk que recuperaste contiene la respuesta". Con él, computas Recall@k chequeando si los chunks recuperados se solapan con los passages de evidence. Este único campo es lo que convierte a FinanceBench de curiosidad a benchmark riguroso. (Ver `docs/CONTEXT.md` § 8 — Passages and chunks.)

</details>

### Pregunta 6 — `mean = 1,712` y `median = 1,450` para la longitud de evidence. ¿Qué te dice ese gap, y por qué importa?

<details>
<summary>Ver respuesta</summary>

`mean > median` significa que la distribución tiene **cola derecha** — algunos valores extremos altos están tirando el promedio para arriba. En nuestro caso el max evidence es 12,123 chars (cerca de 8× el median), confirmando la cola. Esto importa porque: (1) reportar "longitud promedio del evidence es 1,712 chars" engañaría — la mitad del dataset está debajo de 1,450, y el promedio está inflado por los outliers; (2) operacionalmente, una chunking strategy de 512 tokens cabe el median sobrado pero falla con la cola larga. Por eso el proyecto reporta min/p25/median/p75/p95/max/mean en vez de solo el mean (ver `docs/CONTEXT.md` § 9), y por eso reportamos Recall@k en múltiples valores de `k` en vez de solo k=1.

</details>

### Pregunta 7 — ¿Por qué reportamos 4 retrieval metrics en vez de solo Recall@k?

<details>
<summary>Ver respuesta</summary>

Porque cada métrica tiene su blind spot:
- **Recall@k** mide cobertura pero ignora la posición en el ranking — Recall@5=80% se ve bien aunque cada hit esté en posición 5.
- **MRR** premia tener el primer chunk relevante temprano, pero solo mira el primer hit.
- **NDCG@k** mide la calidad completa del ranking logarítmicamente — captura casos que Recall pierde.
- **MAP** maneja múltiples relevants per query — y **23% de FinanceBench tiene multi-evidence**, lo que MRR underweight.

Reportar las cuatro es el estándar académico; el cherry-picking ("¡mira, mi Recall@10 es 95%!") es como los vendors engañan. (Ver `docs/CONTEXT.md` § 11 para la intuición completa de cada métrica.)

</details>

### Pregunta 8 — ¿Por qué el proyecto compara 4 chunking strategies en vez de comprometerse con una?

<details>
<summary>Ver respuesta</summary>

Porque la auditoría encontró una **distribución bimodal** del ratio `evidence_text` / `evidence_text_full_page`: 46% de los evidences son esencialmente la página entera (tablas financieras), 29% son fragmentos chicos dentro de páginas densas, y 25% queda en el medio. Ninguna chunking strategy gana en ambos regímenes:
- Chunks full-page capturan el 46% perfecto pero diluyen el 29%.
- Chunks de 512 tokens capturan fragmentos chicos bien pero parten tablas grandes.
- Semantic chunking respeta boundaries detectados por sentence transformers.
- Late chunking embedea el documento completo primero, después chunkea el embedding.

El proyecto mide las cuatro y deja que los datos decidan cuál gana en cada régimen. Las decisiones formales aterrizan en `docs/chunking_decisions.md` cuando cierre el sub-bloque 6.

</details>

### Pregunta 9 — ¿Por qué el 91% de locality del multi-evidence justifica saltarse MMR en la baseline?

<details>
<summary>Ver respuesta</summary>

MMR (Maximal Marginal Relevance) penaliza redundancia entre los top-k para empujar passages diversos — está diseñado para casos donde los items relevantes están dispersos. Con 91% de los casos multi-evidence clusterizados en la misma sección del 10-K, el top-k vanilla con cosine similarity ya captura ambos passages naturalmente (son semánticamente parecidos AND físicamente adyacentes). Agregar MMR optimizaría para el 9% outlier al costo de complicar la baseline y degradar levemente el caso dominante. La postura correcta de ingeniería es: shipear la baseline más simple que los datos soporten, instrumentarla bien, y revisitar si el failure analysis muestra que el 9% está arrastrando la métrica desproporcionadamente.

</details>

### Pregunta 11 — ¿Por qué el proyecto shipea una **receta** de descarga de PDFs en vez de commitear los PDFs al repo?

<details>
<summary>Ver respuesta</summary>

Tres razones se acumulan:
1. **Higiene del repo** — git guarda cada revisión de cada archivo trackeado para siempre. Agregar 165 MB de binarios hoy infla los clones para cada developer futuro permanentemente. Después de algunos años de updates, el repo sería multi-gigabyte y `git log` contra los binarios sería inutilizable.
2. **Licensing** — FinanceBench es `CC-BY-NC-4.0`. Re-hostear los mismos PDFs desde un repo público distinto es una redistribución del dataset licenciado de otro bajo otra sombrilla — legalmente ambiguo. Apuntar a los users al source upstream `patronus-ai/financebench` es inequívocamente OK.
3. **Single source of truth** — si Patronus actualiza un PDF, el upstream recibe el fix. Una copia en nuestro repo no, lo cual driftearía silenciosamente el eval contra documentos viejos.

El patrón de receta esquiva los tres problemas: cada clone fresco baja lo que upstream tenga en ese momento, el script es idempotente (safe de re-correr), paralelo (8 workers, ~10 segundos) y selectivo (84 PDFs únicos de 368 upstream). El patrón es estándar para proyectos ML que dependen de corpora third-party, y llamarlo explícitamente en el README + `docs/reproducibility.md` lo convierte de "no nos molestamos en commitear" en una decisión deliberada y defendible de ingeniería.

</details>

### Pregunta 12 — ¿Cuál es el siguiente sub-bloque pendiente del bloque FinanceBench Loader, y por qué ese orden?

<details>
<summary>Ver respuesta</summary>

El siguiente sub-bloque pendiente es la **construcción del corpus indexable** (sub-bloque 5). Ahora que tenemos los 84 PDFs parseados como JSONL página-por-página (sub-bloque 4), el siguiente paso es aplicar las **4 chunking strategies** (header-repetition, semantic, contextual, late) sobre ese corpus parseado para generar los chunks que el embedder va a consumir. La separación parser/chunker (`docs/CONTEXT.md` § 13.6) es lo que permite que los 4 experiments corran en **segundos sobre el JSONL**, no minutos re-parseando los PDFs cada vez (~26 min × 4). Después viene el sub-bloque 6: `docs/chunking_decisions.md`, el write-up formal del rationale de qué strategy gana en qué régimen de la distribución bimodal de evidencia/página (§ 2.4). El orden está forzado por la dependencia: no puedes chunkear antes de parsear, no puedes parsear antes de descargar, y no deberías descargar antes de entender qué vale la pena indexar — que es exactamente lo que la auditoría de las 5 dimensiones (sub-bloque 2) produjo.

</details>
